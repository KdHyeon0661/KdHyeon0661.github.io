# 가상화·클라우드 워크로드 보안(하이퍼바이저, 이미지/스냅샷, IAM 기초)

## 0) 큰 그림(One-Page)

- **하이퍼바이저 보안**: 최소 표면(호스트 하드닝·패치·보안 부트/TPM), 관리 평면 격리, VM 경계(sVirt/SELinux·AppArmor), 디바이스 직할(Passthrough/SR-IOV) 최소화, 네트워크/스토리지 격리.
- **이미지/스냅샷 보안**: 골든 이미지(Immutable) 파이프라인, 비밀(Secrets) 제로, 서명/무결성, 취약점 스캔, **스냅샷 암호화·공유 통제·DR 복제**, 만료 및 표준화(이미지 패밀리).
- **IAM 기초**: 최소 권한(정적 키 금지), **역할(Role) + 임시 자격(STS/OIDC/SAML)**, 리소스 기반 정책/퍼미션 바운더리/SCP, **워크로드 아이덴티티(EC2 Instance Profile, GCP SA, Azure Managed Identity)**.

# 1) 하이퍼바이저 보안(Hypervisor Security)

## 1.1 공통 원칙

- **호스트 OS 하드닝**: 불필요 패키지 제거, 커널/펌웨어/드라이버 최신화, 보안 부트·TPM·정상 부팅(Measure) 활용.
- **관리 평면 격리**: 관리 NIC(Out-of-Band) 분리, 방화벽으로 관리 포트(예: ESXi Host Client/SSH, Libvirt, Hyper-V WMI) 제한.
- **멀티테넌시·경계**: VM Escape 방지(가상화 기능 최소, 최신 qemu/virtio), sVirt/SELinux 또는 AppArmor로 VM 파일/프로세스 라벨링.
- **직접 장치 할당 주의**: PCI-Passthrough/SR-IOV는 성능 장점 있지만 **격리 약화** 가능. 꼭 필요한 워크로드에만 제한 적용.
- **네트워크/스토리지**: 포트그룹/VLAN/VXLAN 등으로 테넌트 분리, 데이터스토어 권한 최소화, 스토리지 암호화.

## 1.2 KVM(libvirt/qemu) 하드닝

### (A) 호스트 기본

```bash
# 1) 커널/펌웨어 최신화
sudo dnf update -y  # RHEL/Alma/Rocky
# 또는
sudo apt update && sudo apt upgrade -y

# 2) 불필요 서비스 제거·비활성
sudo systemctl disable --now avahi-daemon || true
sudo systemctl disable --now cups || true

# 3) 보안 프로파일(예: SELinux Enforcing)
getenforce || true
sudo setenforce 1
```

### (B) libvirt 접근 제어

```bash
# libvirt UNIX 소켓 권한/그룹 확인 (기본: libvirt, libvirt-qemu)
sudo usermod -aG libvirt adminuser
# TLS 원격 관리가 필요하면 x509 준비(테스트/운영은 별도 PKI 권장)
```

### (C) sVirt(SELinux)로 VM 격리

- libvirt는 VM마다 **고유 라벨**(qemu_t 파생)을 부여하여 디스크/프로세스 격리를 강화합니다.
- 커스텀 경로에 디스크를 둘 경우 라벨 복구 필요:

```bash
sudo semanage fcontext -a -t virt_image_t "/vmstore(/.*)?"
sudo restorecon -Rv /vmstore
```

### (D) qemu 프로세스 권한 축소(예)

```bash
# /etc/libvirt/qemu.conf
# user/group를 qemu 전용 계정으로, dynamic_ownership 유지(파일 라벨·권한 자동 조정)
user = "qemu"
group = "qemu"
dynamic_ownership = 1
```

### (E) 네트워크 격리

```bash
# 브리지 vs NAT: 외부 노출이 필요 없으면 NAT 네트워크 유지
virsh net-list --all
# 관리/테넌트/스토리지 네트워크는 물리/가상 스위치로 분리
```

**검증**  
- `virsh dominfo <vm>` · `ps -Z -p <qemu-pid>` 로 SELinux 라벨 확인.  
- 관리 호스트에서만 libvirt 접근 가능한지 방화벽 확인.

**롤백**  
- `qemu.conf` 백업 복원, `semanage fcontext -d` → `restorecon -Rv`.

## 1.3 VMware ESXi 하드닝(핵심)

- **Lockdown Mode** 활성(DCUI/SSH 최소), **Secure Boot/TPM** 사용.

- **로그 원격 전송**(Syslog) 및 **알림**(vCenter Alarm) 구성.

```bash

esxcli system settings advanced set -o /UserVars/SuppressShellWarning -i 1
# 패치/이미지 프로파일 관리(VUM/Lifecycle Manager)로 정기 업데이트
```

# 2) 이미지/스냅샷 보안(Image/Snapshot Security)

> 목표: **골든 이미지 빌드 → 서명/스캔 → 배포/공유 통제 → 스냅샷 암호화·복구·만료**  
> 이미지는 **OS·러ntime·에이전트·설정**을 표준화하여 **드리프트(상태 표류)**를 줄입니다.

## 2.1 골든 이미지 원칙

- **비밀(Secrets) 제로**: 이미지에 API 키/토큰/SSH 키/비밀번호를 **절대 포함 금지**.  
  → 런타임에 **Secret Manager/Parameter Store/KMS/Key Vault**에서 주입.
- **최소화**: 패키지·서비스 최소, 불필요 계정/포트 제거, SSH 비밀번호 로그인 금지.
- **취약점 스캔**: 빌드 직후 OS 패키지/런타임 스캔(예: Trivy/Grype/OVAL).
- **서명/무결성**: 산출물(OVF/OVA, AMI 메타/매니페스트)을 **서명/해시 기록**.
- **버저닝**: “이미지 패밀리/채널(stable/canary) + 날짜/빌드번호” 표기.
- **만료**: EOL 이미지/스냅샷 **자동 만료/삭제 정책**.

---

## 2.2 Packer로 골든 AMI/이미지 빌드(예제)

### (A) Packer HCL: AWS AMI

```hcl
# packer/aws-ami.pkr.hcl
packer {
  required_plugins {
    amazon = { source = "hashicorp/amazon", version = ">= 1.2.0" }
  }
}

variable "base_ami"    { type = string }
variable "ami_name"    { type = string }
variable "kms_key_id"  { type = string }
variable "vpc_id"      { type = string }
variable "subnet_id"   { type = string }
variable "ssh_user"    { type = string  default = "ec2-user" }

source "amazon-ebs" "hardened" {
  ami_name             = var.ami_name
  instance_type        = "t3.micro"
  region               = "ap-northeast-2"
  source_ami           = var.base_ami
  ssh_username         = var.ssh_user
  subnet_id            = var.subnet_id
  vpc_id               = var.vpc_id
  launch_block_device_mappings {
    device_name = "/dev/xvda"
    volume_size = 20
    volume_type = "gp3"
    delete_on_termination = true
    encrypted = true
    kms_key_id = var.kms_key_id
  }
  tags = {
    "ImageFamily" = "web-base"
    "Owner"       = "secops"
  }
}

build {
  name    = "hardened-ami"
  sources = ["source.amazon-ebs.hardened"]

  provisioner "shell" {
    inline = [
      "sudo dnf update -y || sudo apt-get update -y",
      "sudo dnf upgrade -y || sudo apt-get dist-upgrade -y",
      # 최소화: 불필요 서비스 제거 예시
      "sudo systemctl disable --now avahi-daemon || true",
      "sudo systemctl disable --now cups || true",
      # SSH 하드닝
      "sudo sed -i 's/^#\\?PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config",
      "sudo sed -i 's/^#\\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config",
      "sudo systemctl reload sshd || sudo systemctl restart ssh",
      # 로그/에이전트
      "sudo systemctl enable --now amazon-ssm-agent || true"
    ]
  }
}
```

**검증**  
- 빌드 완료 후 AMI 태그/암호화 여부 확인, SSH 비밀번호 로그인 불가 여부 점검.  
- 취약점 스캔(예: CI에서 `trivy rootfs --severity HIGH,CRITICAL`).

**롤백**  
- 이전 버전 AMI ID로 Auto Scaling Launch Template 버전 되돌리기.

## 2.3 스냅샷 보안(암호화·공유·DR)

### (A) AWS — EBS 스냅샷 암호화/공유 통제

```bash
# 1) 계정 기본 EBS 암호화 활성(권장)
aws ec2 enable-ebs-encryption-by-default

# 2) 스냅샷 생성(암호화된 볼륨에서)
SNAP_ID=$(aws ec2 create-snapshot --volume-id vol-xxxx --description "web-data" --query SnapshotId --output text)

# 3) 스냅샷 복사 + 다른 KMS로 재암호화(교차 계정/리전 DR용)
aws ec2 copy-snapshot \
  --source-region ap-northeast-2 \
  --source-snapshot-id $SNAP_ID \
  --encrypted \
  --kms-key-id arn:aws:kms:ap-northeast-1:111122223333:key/abcd-... \
  --destination-region ap-northeast-1

# 4) 공유 설정(신중): 특정 계정에만
aws ec2 modify-snapshot-attribute \
  --snapshot-id $SNAP_ID \
  --attribute createVolumePermission \
  --operation-type add \
  --user-ids 111122223333
```

**검증**  
- `describe-snapshots`로 `Encrypted=true`, `KmsKeyId` 확인.  
- CloudTrail에서 CreateSnapshot/ModifySnapshotAttribute 이벤트 모니터링.

**롤백**  
- 공유 허용 취소(`operation-type remove`), DR 리전에 복사본 삭제.

# 3) IAM 기초(Least Privilege & Workload Identity)

> **핵심**: “사람은 SSO로, 워크로드는 **역할(Role)** 로”. **정적 액세스 키 금지**, **임시 자격(STS/OIDC/SAML)**.

## 3.1 개념 맵

- **식별자(Identities)**: 사용자(사람), 그룹, **역할(Role)**(워크로드/서비스용).
- **정책(Policies)**: JSON/문장으로 권한 정의(허용 중심 + 명시적 거부).
- **리소스 기반 정책**: S3 버킷/키/토픽 등 리소스 측 정책.
- **권한 경계(Permission Boundary)**: 최대치 상한선.
- **조직 정책(SCP)**: 계정 전체 상한선(거버넌스).
- **임시 자격**: `AssumeRole`(AWS STS), Workload Identity Federation(GCP), Managed Identity(Azure).

---

## 3.2 AWS — 역할/정책 예제

### (A) S3 읽기 전용 정책(JSON)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow", "Action": ["s3:GetObject","s3:ListBucket"], "Resource": [
      "arn:aws:s3:::my-logs", "arn:aws:s3:::my-logs/*"
    ]}
  ]
}
```

### (B) EC2 인스턴스 프로파일(워크로드 역할)

```bash
# 역할 신뢰 정책(EC2가 이 역할을 받을 수 있도록)
cat > trust-ec2.json <<'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow", "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole" }
  ]
}
EOF

aws iam create-role --role-name role-web --assume-role-policy-document file://trust-ec2.json
aws iam put-role-policy --role-name role-web --policy-name s3-read-logs --policy-document file://s3-readonly.json
aws iam create-instance-profile --instance-profile-name ip-web
aws iam add-role-to-instance-profile --instance-profile-name ip-web --role-name role-web
```

### (C) GitHub Actions → AWS OIDC로 AssumeRole(정적 키 제거)

```json
// trust-oidc.json (역할 신뢰 정책; OIDC 발급자/리포지토리 제약)
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Federated": "arn:aws:iam::<acct-id>:oidc-provider/token.actions.githubusercontent.com" },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": { "token.actions.githubusercontent.com:aud": "sts.amazonaws.com" },
        "StringLike": { "token.actions.githubusercontent.com:sub": "repo:myorg/myrepo:*" }
      }
    }
  ]
}
```

**CI 파이프라인**은 이 역할을 `sts:AssumeRoleWithWebIdentity`로 받아 임시 자격만 사용.

### (D) Permission Boundary(최대 상한)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Deny", "Action": "iam:*", "Resource": "*" },
    { "Effect": "Allow", "Action": ["s3:*"], "Resource": "arn:aws:s3:::team-bucket/*" }
  ]
}
```

※ 사용자/역할에 바운더리를 부착하면 **그 이상 권한 상승 불가**.

## 3.4 Azure — Managed Identity & RBAC

```bash
# VM에 시스템 할당 Managed Identity 활성화
az vm identity assign -g rg-app -n vm-web

# 스토리지 읽기 역할 부여(리소스 스코프에서 최소 권한)
PRINCIPAL_ID=$(az vm show -g rg-app -n vm-web --query identity.principalId -o tsv)
az role assignment create \
  --assignee $PRINCIPAL_ID \
  --role "Storage Blob Data Reader" \
  --scope "/subscriptions/<sub>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/mystorage"
```

- 비밀은 **Key Vault**에서 참조, MSI로 토큰을 받아 접근.

## 3.5 공통 베스트 프랙티스

- **정적 키 금지**(사용 불가하도록 정책화), 임시 자격만 사용.  
- 사람은 **SSO/MFA**, 워크로드는 **역할/서비스 계정/Managed Identity**.  
- **권한 그래프 줄이기**: 와일드카드(`*`)·`Administrator`류 회피, **리소스 수준**으로 한정.  
- **거버넌스**: AWS Organizations의 **SCP**, Azure **Blueprint/Policy**, GCP **Organization Policy**로 상위 제한.  
- **키/권한 회전** 자동화, **태그/라벨** 기준으로 접근 제어(ABAC) 병행.

# 4) 모니터링·탐지(CloudTrail/Logs/KQL/Queries)

## 4.1 AWS — Athena로 스냅샷 공개 탐지(예)

```sql
-- CloudTrail Lake/Athena에 파티션된 trail_logs가 있다고 가정
SELECT eventTime, userIdentity.principalId, eventName, requestParameters.snapshotId, requestParameters.createVolumePermission
FROM trail_logs
WHERE eventName = 'ModifySnapshotAttribute'
  AND json_extract_scalar(requestParameters, '$.createVolumePermission.add.items[0].group') = 'all'
  AND eventTime > current_timestamp - interval '1' day;
```

**대응**: 공유 제거 API 호출 + EventBridge 규칙으로 실시간 알림(SNS/Slack).

# 5) 시나리오(End-to-End)

## 시나리오 A — “골든 이미지 파이프라인 + 배포 링(Ring)”

1. **Packer**로 하드닝 AMI/이미지 빌드(암호화·SSM 에이전트·SSH 하드닝).  
2. CI에서 **취약점 스캔 + 비밀 검출** → 실패 시 빌드 중단.  
3. **서명/해시 기록** → 아티팩트 저장소.  
4. **Dev → Staging → Prod** 순으로 **점진 배포**(Auto Scaling `Launch Template` 버전).  
5. **드리프트 탐지**: SSM/Config로 런타임 표준 준수 확인.  
6. **만료 정책**: n일 지난 이미지/스냅샷 자동 삭제.

**Terraform 스니펫(Launch Template 버저닝)**

```hcl
resource "aws_launch_template" "web" {
  name_prefix   = "web-lt-"
  image_id      = var.ami_id
  instance_type = "t3.micro"
  iam_instance_profile { name = "ip-web" }
  disable_api_termination = true
  update_default_version  = true
  tag_specifications {
    resource_type = "instance"
    tags = { "PatchLevel" = "baseline", "Owner" = "secops" }
  }
}
```

## 시나리오 B — “DR 스냅샷 복제 + 교차 계정 공유”

1. **EBS 스냅샷 암호화**(KMS CMK).  
2. **타 리전 복사 + 재암호화**(DR KMS 키).  
3. **교차 계정** DR 계정에만 **정밀 공유**.  
4. **EventBridge**로 변경 감지 및 알림.

(위 AWS CLI 예제 참조)

## 시나리오 C — “CI/CD OIDC → 역할 가정(Zero Secret)”

1. GitHub Actions에서 OIDC 토큰 발급.  
2. AWS 역할의 신뢰 정책에 `sub=repo:org/repo:*` 제한.  
3. 파이프라인에서 `aws sts assume-role-with-web-identity`로 임시 자격 획득.  
4. `terraform apply`에 필요한 **최소 권한 정책**만 부여.

# 7) 흔한 함정 & 대응

- **이미지에 비밀 포함** → 빌드 단계에서 **정적 스캔** + 런타임에서만 주입.  
- **스냅샷 공개** → 조직 정책으로 **공개 금지** + 이벤트 탐지/차단.  
- **역할 남용/와일드카드** → 리소스 단위로 축소, Permission Boundary/SCP로 상한선.  
- **직접 장치 할당 남발** → **성능 대비 보안 리스크** 판단, 테넌트 격리 호스트.  
- **드리프트** → 불변 인프라(Immutable) + 주기적 **재배포/교체**.
