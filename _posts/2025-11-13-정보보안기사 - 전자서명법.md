# 완전 정리 — 개념, 제도, 기술·운영 실무, 점검 체크리스트

## 개요: 왜 ‘전자서명법’인가

대한민국 **전자서명법**은 전자서명과 전자서명인증업무의 신뢰성·투명성을 보장하여 전자거래를 촉진하려는 기본법이다. 2020년 전부개정으로 과거의 **‘공인인증서’ 단일우위 구조를 폐지**하고, 민간 다양한 전자서명 간 **경쟁·상호운용**을 전제로 한 신뢰 프레임을 마련했다. 법의 **목적·정의·이용자 권리·인증사업자 의무**는 법률 본문에, **평가·인정·유효기간 등 절차 사항**은 시행령에 규정된다.

> 핵심 타임라인
> • 2020-12: 전자서명법 전부개정(공인인증서 우대 폐지, 평가·인정 제도) → 2021~22년 제도 안착
> • 2022-10-20: 현행 전자서명법 시행(최신 본문 기준)
> • 2023-09-15: 시행령 최신화(‘운영기준 준수사실 **인정의 유효기간 1년**’ 등 명문화)

---

## 법 체계 한눈에 보기

- **법률(전자서명법)**: 목적, 전자서명 정의, 전자서명인증업무 범위, 사업자 의무·책임, 평가·인정의 법적 근거, 이용자 보호 등.
- **시행령**: 평가기관·인정기관 기준, **인정 유효기간(1년)**, 절차·고시 위임, 자료 제출·검사 등 집행 세부.
- **고시/가이드**(과기정통부·KISA): 평가항목 상세, 기술·운영통제, 감사·사고대응, CPS(인증업무준칙) 서식 등(기관 홈페이지 고시와 사업자 CPS에 반영).

---

## 핵심 개념 정리
### 전자서명(Electronic Signature)

전자적 형태의 서명으로서 **서명자 신원과 서명내용 변조 여부를 확인**할 수 있도록 하는 수단(일반·고도 전자서명 포함).(법 제2)

### 전자서명인증업무(Trust Service)

인증서 발급·관리, 신원확인, 상태정보(OCSP/CRL) 제공 등. 사업자는 **운영기준을 준수**하고 **평가-인정을 통해 신뢰표시**를 사용할 수 있다(법·시행령).

### ‘평가·인정’ 제도(핵심 포인트)

- **평가기관**: 인증사업자의 운영기준 준수 여부를 심사·평가.
- **인정기관**: 평가결과에 따라 **‘운영기준 준수사실의 인정’을 부여**(표시 사용 허용).
- **인정 유효기간: 1년**(시행령 제4조). 갱신 시 재평가 필요.

---

## 시장 변화: ‘공인’ 폐지 이후 무엇이 달라졌나

- 단일 공인인증 우위 폐지 → **다양한 민간 인증수단(패스·카카오·금융권 등)**이 경쟁.
- 국가가 특정 기술을 강제하지 않음 → **기술중립성** + **신뢰 프레임(평가·인정)** 조합.
- 이용자 관점: 용이성(앱·바이오·FIDO), **법적 효력은 서명 효력 요건 충족 시 동일**.

---

## 실무자가 꼭 알아야 할 **의무·책임** 체크

1) **CPS(인증업무준칙)** 수립·공개 및 준수(업무범위, 신원확인, 키관리, 사고대응, 보험 등) — 사업자 CPS 사례는 ‘운영기준’ 조항·평가요건을 그대로 반영.
2) **연 1회 이상** 외부평가(또는 국제통용평가 인정) 및 **인정 유효기간 1년** 내 갱신.
3) 자료 제출·검사·감사 협조, 위반 시 시정명령·인정 취소 가능(시행령).

## 기술 스택과 법 요건 매핑(요지)

| 법·제도 요구 | 기술/운영 통제 예시 |
|---|---|
| 서명 진정성·무결성 | **비대칭키 서명**(RSA-PSS, ECDSA), 메시지 다이제스트(SHA-256+), 서명값/증명서 체인 검증 |
| 서명자 신원확인 | eKYC/대면·비대면 신원확인 프로세스, 위험기반 심화확인(RBA) |
| 키·인증서 수명주기 | HSM 내 키 생성/보관, 키백업·교체, CRL/OCSP 상태제공 |
| 감사·사고대응 | 모든 ‘발급·정지·폐지·상태조회’에 대한 **불변 로그** + 타임스탬프, 침해대응 Runbook |
| 평가·인정 | CPS 준칙, 내부감사, 외부평가 준비 증적(정책·절차·로그·리포트) |

## 전자서명 **기술 예제** 모음

### OpenSSL로 **PKCS#7/CMS** 서명·검증

```bash
# 개인키·자체서명 인증서(데모)

openssl req -x509 -newkey rsa:3072 -keyout signer.key -out signer.crt -days 365 -nodes -subj "/CN=Demo Signer"

# 파일 해시 후 CMS Detached 서명(PKCS#7)

openssl cms -sign -binary -in contract.pdf -signer signer.crt -inkey signer.key \
  -out contract.sig -outform DER -nodetach

# 서명 검증(인증서 체인·신뢰루트 구성 필요)

openssl cms -verify -binary -in contract.sig -inform DER -content contract.pdf \
  -CAfile trust-bundle.pem -purpose any -out /dev/null
```

### **JWS(서명된 JSON)** — 서버에서 메시지 무결성 보장

```python
# python -m pip install pyjwt cryptography

import jwt, datetime
from cryptography.hazmat.primitives.asymmetric import ec
from cryptography.hazmat.primitives import serialization

# ECDSA P-256 키 생성(데모)

private_key = ec.generate_private_key(ec.SECP256R1())
pem = private_key.private_bytes(
    serialization.Encoding.PEM, serialization.PrivateFormat.PKCS8, serialization.NoEncryption()
)
public_pem = private_key.public_key().public_bytes(
    serialization.Encoding.PEM, serialization.PublicFormat.SubjectPublicKeyInfo
)

payload = {"doc_id": "INV-2025-001", "signed_at": datetime.datetime.utcnow().isoformat()}
token = jwt.encode(payload, pem, algorithm="ES256")

# 검증

decoded = jwt.decode(token, public_pem, algorithms=["ES256"])
print(decoded)
```

### **XAdES/PAdES/CAdES** 개념 메모

- **PAdES**: PDF 내 전자서명(장기검증용 타임스탬프·증빙 포함).

(국내법은 특정 포맷을 강제하지 않지만, 상호운용·장기검증 관점에서 표준 포맷 사용 권장)

## 인증서·상태 정보 운영(OCSP/CRL)

### OCSP Responder 개념 구성

```
[Client] -- OCSP Req/Res --> [OCSP Responder] --(CRL/DB)--> [CA DB/HSM]
```
- **실시간 상태조회**(유효/폐지/보류) 제공. 응답에 **서명** 필수(서버키 보호).
- 고가용성: 이중화(Active-Active), 캐시 TTL, **스테이플링**(서버가 응답을 클라이언트에 전달).

### CRL 운영 팁

- 주기 발행(예: 4h~24h) + 긴급 폐지는 **CRLDP+OCSP 병행**.
- CRL 사이즈 최적화(Delta CRL, 분할).

## **CPS(인증업무준칙)** 구성 템플릿(요약)

1) 서문/정의/약어
2) 조직·책임(임원책임, 보안책임, 독립감사)
3) 신원확인(가입자·대리인, RBA, 증빙 보존)
4) 인증서 생명주기(신규/갱신/정지/폐지, 재발급)
5) 키관리(HSM, 백업, 분리보관, 키파괴)
6) 보안통제(물리/인력/접근통제, 변경관리, 취약점·패치)
7) 로그·모니터링(무결성, 보존기간, 타임스탬프)
8) 사고관리·보험(배상책임, 통지, 보고)
9) 준거법·분쟁해결(관할, 고지)
→ 국내 주요 사업자 CPS에는 **‘운영기준 준수사실의 인정’ 요건**과 **유효기간 1년** 조항이 반복 반영되어 있다.

## **개발자 가이드**: 서명·검증·타임스탬프 통합

### REST 예시(서명 요청→서명값 반환)

```http
POST /v1/signatures
Content-Type: application/json
{
  "algo": "ECDSA_P256_SHA256",
  "payload_b64": "BASE64(DATA)",
  "cert_id": "sub-ca-2025-01"
}
```

### 검증 파이프라인(의사코드)

```python
def verify(signature, payload, cert_chain):
    # 1) chain 검증(루트 신뢰묶음)
    validate_chain(cert_chain, trust_store)
    # 2) 유효기간/KeyUsage 확인
    check_cert_profile(cert_chain[0])
    # 3) 서명 알고리즘·파라미터 확인(ECDSA, RSASSA-PSS)
    # 4) OCSP/CRL 상태조회(Soft/Hard fail 정책)
    status = ocsp_check(cert_chain[0].serial)
    # 5) 서명 검증
    return crypto_verify(signature, payload, cert_chain[0].pubkey)
```

### 팁

- 타임스탬프(TSA)로 ‘당시 유효성’을 봉인.
- 체인·CRL/OCSP 응답을 서명문서에 **포함**(PAdES-LTV/XAdES-A/CAdES-A).

## 빈출 Q&A

**Q1. 전자서명 간 ‘법적 효력’ 차이가 있나?**
A. 특정 제품이 우월하지 않는다. **효력은 서명 진정성·무결성·신원확인 등 요건 충족으로 판단**하며, 평가·인정은 신뢰성 판단에 도움을 준다.

**Q2. ‘인정’ 표시는 영구한가?**
A. 아님. **유효기간 1년**, 갱신 시 재평가·시정조치 추적 필요(시행령 제4조).

**Q3. KISA 역할은?**
A. 과기정통부 고시에 따라 **인정 관련 실무 수행 기관**으로 지정·운영(고시·CPS 반영 사례 다수).

## 부록 A. 수학 한 컷(개념 상기)

전자서명은 보통 아래의 **비대칭 암호**에 기댄다. (개념 강조용)
$$
\text{Sign}_{sk}(m)\rightarrow \sigma,\quad \text{Verify}_{pk}(m,\sigma)\in\{\mathtt{true},\mathtt{false}\}
$$
- 보안의 직관: **개인키(sk)** 없이는 \(\sigma\) 생성이 어려워야 하며, 임의변조 시 검증 실패.

## 부록 B. 운영 절차(예시 BPMN 텍스트)

```
User -> RA: 등록신청(증빙)
RA -> KYC: 신원확인(RBA)
RA -> CA: 발급요청(CSR)
CA -> HSM: 서명키 보호, 인증서 발급
CA -> Repo: CRL/OCSP 게시
CA -> User: 인증서 교부
SIEM <- CA/RA: 불변로그 전송
```

## 결론

국내 전자서명 생태계는 **기술중립** 하에 **평가·인정**으로 신뢰를 담보하는 구조로 정착했다. 실무자는 **CPS 중심의 정책-기술-운영 통합**과 **연 1회 인정 갱신**을 축으로 **키/인증서·상태정보·감사로그**를 빈틈없이 설계·운영해야 한다. 이렇게 하면 법적 요구와 장기검증(LTV)·분쟁대응까지 일관되게 커버할 수 있다.
