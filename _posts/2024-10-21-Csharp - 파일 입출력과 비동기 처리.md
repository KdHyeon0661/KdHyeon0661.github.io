# C# 파일 입출력(I/O)과 비동기 처리

소프트웨어 개발에서 **입출력(I/O)**은 CPU 연산에 비해 압도적으로 느린 작업입니다. 디스크에서 파일을 읽고 쓰는 시간은 메모리 접근 시간의 수만 배에 달하기 때문에, I/O를 어떻게 다루느냐가 전체 애플리케이션의 성능과 확장성을 결정짓는 핵심 요인이 됩니다.

C#은 단순한 텍스트 파일 조작부터, 대용량 바이너리 데이터의 스트리밍, 그리고 스레드 블로킹을 없앤 완벽한 비동기(`async/await`) 처리까지 파일 시스템을 제어할 수 있는 강력하고 유연한 도구들을 제공합니다. 이 글에서는 파일 I/O의 기본기부터 시작해, 실무에서 데이터를 안전하게 보호하기 위한 원자적 쓰기(Atomic Write) 패턴과 엔터프라이즈급 로그 회전(Rolling Log) 시스템 구현까지 깊이 있게 파헤쳐 보겠습니다.

---

## 1. 텍스트 파일 작업: 단순성과 인코딩

가장 가벼운 작업이나 설정 파일을 읽고 쓸 때는 `System.IO.File` 클래스가 제공하는 정적 메서드들을 사용하는 것이 가장 직관적입니다.

```csharp
using System.IO;
using System.Text;

// 1. 단순한 텍스트 파일 읽기/쓰기
File.WriteAllText("document.txt", "안녕하세요, C#입니다.");
string content = File.ReadAllText("document.txt");

// 2. 여러 줄 처리
var lines = new[] { "첫 번째 줄", "두 번째 줄", "세 번째 줄" };
File.WriteAllLines("multiline.txt", lines);

// 3. 인코딩의 명시적 제어 (매우 중요)
// 현대 시스템의 표준인 UTF-8 (BOM 없음) 사용을 권장합니다.
File.WriteAllText("utf8.txt", "유니코드 텍스트", Encoding.UTF8);

// 레거시 시스템 연동 시 특정 인코딩(예: EUC-KR) 사용
Encoding.RegisterProvider(CodePagesEncodingProvider.Instance); // .NET Core 이상 필수
string eucKrText = File.ReadAllText("euckr.txt", Encoding.GetEncoding(51949));
```

**주의사항**: `ReadAllText`나 `ReadAllLines`는 파일의 모든 내용을 한 번에 메모리(RAM)로 끌어올립니다. 파일 크기가 수백 MB를 넘어간다면 즉시 `OutOfMemoryException`이 발생할 수 있으므로, 대용량 파일에는 절대 사용해서는 안 됩니다.

## 2. 스트림(Stream) 기반 접근: 유연성과 제어

용량이 큰 파일이나 실시간으로 쏟아지는 데이터를 다룰 때는 **스트림(Stream)**을 사용해야 합니다. 스트림은 데이터를 한 번에 메모리에 올리지 않고, 빨대로 물을 빨아들이듯 조금씩(Chunk) 흐르게 만들어 처리하는 추상화된 파이프라인입니다.

### StreamReader와 StreamWriter

텍스트 기반의 대용량 파일을 효율적으로 읽고 쓰기 위한 도구입니다.

```csharp
// 효율적인 파일 쓰기 (버퍼 사이즈 최적화)
using var writer = new StreamWriter("output.txt", append: true, Encoding.UTF8, bufferSize: 8192);
for (int i = 0; i < 1000; i++)
{
    await writer.WriteLineAsync($"데이터 {i}: {DateTime.Now}");
}

// 대용량 파일 한 줄씩 스트리밍 읽기 (메모리 폭발 방지)
using var reader = new StreamReader("largefile.txt");
string? line;
while ((line = await reader.ReadLineAsync()) != null)
{
    ProcessLine(line); // 한 줄씩 처리 후 메모리에서 해제됨
}
```

### FileStream과 최적화 옵션

이미지, 동영상 등 바이너리 데이터를 다루거나, OS 레벨의 파일 제어가 필요할 때는 가장 근본적인 `FileStream`을 사용합니다. 특히 생성자에서 제공하는 옵션들을 통해 성능을 극한으로 끌어올릴 수 있습니다.

```csharp
using var optimizedStream = new FileStream(
    path: "performance.dat",
    mode: FileMode.OpenOrCreate,
    access: FileAccess.ReadWrite,
    share: FileShare.Read, // 다른 프로세스에서 읽을 수 있도록 허용
    bufferSize: 65536,     // 64KB 단위의 넉넉한 버퍼
    options: FileOptions.Asynchronous | FileOptions.SequentialScan
);
```

* **Asynchronous**: 비동기 I/O를 활성화합니다. (뒤에서 자세히 다룹니다)
* **SequentialScan**: 데이터를 처음부터 끝까지 순차적으로 읽을 것임을 OS에 알려주어, 디스크 헤더의 움직임을 최적화하고 데이터를 미리 캐싱(Prefetching)하게 만듭니다.

## 3. 비동기 I/O (Async I/O): 응답성의 핵심

동기(Synchronous) 방식으로 I/O를 수행하면, 디스크가 물리적으로 회전하여 데이터를 가져오는 수 밀리초~수 초 동안 CPU 스레드는 아무 일도 하지 못한 채 블로킹(Blocking)됩니다. 트래픽이 많은 서버 환경에서 이는 심각한 자원 낭비입니다.

비동기 I/O(`async/await` + `FileOptions.Asynchronous`)를 사용하면, I/O 요청을 OS 하드웨어 컨트롤러에 위임하고 현재 스레드는 즉시 스레드 풀로 반환되어 다른 작업을 처리합니다.

### 실전: 진행률과 취소가 지원되는 비동기 파일 복사

단순한 복사를 넘어, 사용자에게 진행 상황을 보고(`IProgress<T>`)하고 언제든 취소(`CancellationToken`)할 수 있는 현대적인 파일 복사 메서드입니다.

```csharp
public async Task CopyFileWithProgressAsync(
    string sourcePath, 
    string destinationPath,
    IProgress<double> progress = null,
    CancellationToken cancellationToken = default)
{
    long fileSize = new FileInfo(sourcePath).Length;
    long totalBytesRead = 0;
    const int bufferSize = 65536; // 64KB
    
    // FileOptions.Asynchronous 옵션이 핵심입니다.
    await using var sourceStream = new FileStream(sourcePath, FileMode.Open, FileAccess.Read, 
                                                  FileShare.Read, bufferSize, FileOptions.Asynchronous);
    await using var destStream = new FileStream(destinationPath, FileMode.Create, FileAccess.Write, 
                                                FileShare.None, bufferSize, FileOptions.Asynchronous);
    
    var buffer = new byte[bufferSize];
    int bytesRead;
    
    // 버퍼 단위로 스트리밍하며 비동기 복사
    while ((bytesRead = await sourceStream.ReadAsync(buffer, 0, buffer.Length, cancellationToken)
                                       .ConfigureAwait(false)) > 0)
    {
        cancellationToken.ThrowIfCancellationRequested(); // 취소 요청 확인
        
        await destStream.WriteAsync(buffer, 0, bytesRead, cancellationToken)
                        .ConfigureAwait(false);
        totalBytesRead += bytesRead;
        
        // 진행률 업데이트 (UI 스레드로 안전하게 전달됨)
        progress?.Report((double)totalBytesRead / fileSize * 100);
    }
}
```

> **💡 팁**: 라이브러리 코드에서는 `ConfigureAwait(false)`를 사용하여 컨텍스트 캡처를 방지하는 것이 성능에 도움이 됩니다. UI 애플리케이션에서는 상황에 따라 다르지만, 백엔드 서비스나 클래스 라이브러리에서는 습관적으로 붙이는 것이 좋습니다.

## 4. 실전 엔터프라이즈 패턴

실무에서 단순히 파일을 읽고 쓰는 것을 넘어, 데이터의 무결성을 보장하고 시스템을 안정적으로 운영하기 위한 고급 패턴들입니다.

### 패턴 1: 원자적 파일 업데이트 (Atomic File Update)

설정 파일이나 중요한 데이터를 덮어쓸 때, 쓰는 도중 정전이 되거나 프로그램이 죽으면 파일이 반쪽짜리가 되어 영구적으로 손상됩니다. 이를 막기 위해 **임시 파일에 완전히 다 쓴 후 원본과 순식간에 바꿔치기**하는 패턴을 사용합니다.

```csharp
public async Task AtomicWriteAsync(string filePath, string content, bool keepBackup = false)
{
    string tempPath = filePath + ".tmp";
    string backupPath = filePath + ".backup";
    
    try
    {
        // 1. 안전한 임시 파일에 데이터를 완전히 씁니다.
        await File.WriteAllTextAsync(tempPath, content, Encoding.UTF8);
        
        // 2. OS 레벨의 원자적(Atomic) 교체 작업 수행
        //    ※ 같은 파일 시스템 내에서만 원자성이 보장됩니다. 
        //      네트워크 드라이브나 다른 볼륨 간 이동은 원자적이지 않을 수 있습니다.
        if (OperatingSystem.IsWindows())
        {
            // Windows: 원본을 백업으로 옮기고, 임시파일을 원본으로 교체 (원자성 보장)
            File.Replace(tempPath, filePath, backupPath, ignoreMetadataErrors: true);
        }
        else
        {
            // Unix/Linux: Move 작업이 원자적으로 동작함
            File.Move(tempPath, filePath, overwrite: true);
        }

        // 3. 백업 파일 처리 (선택 사항)
        if (!keepBackup && File.Exists(backupPath))
        {
            File.Delete(backupPath);
        }
    }
    finally
    {
        // 4. 실패하더라도 쓰다 만 임시 파일의 잔해를 확실히 청소합니다.
        if (File.Exists(tempPath)) File.Delete(tempPath);
    }
}
```

### 패턴 2: 로그 회전(Rolling) 시스템

서버가 1년 365일 돌아가면 로그 파일의 크기가 수십 GB에 달해 열어볼 수 없게 됩니다. 용량이 일정 기준을 넘으면 기존 파일을 백업/압축하고 새 파일에 기록하는 롤링 로그 시스템의 구현입니다.

```csharp
using System.IO.Compression;
using System.Text.RegularExpressions;

public class RollingFileLogger : IAsyncDisposable
{
    private readonly string _logDir;
    private readonly string _baseName;
    private readonly long _maxFileSize;
    private readonly int _maxBackups;
    private StreamWriter _currentWriter;
    private long _currentSize;

    public RollingFileLogger(string logDir, string baseName, long maxFileSize = 10 * 1024 * 1024, int maxBackups = 5)
    {
        _logDir = logDir;
        _baseName = baseName;
        _maxFileSize = maxFileSize;
        _maxBackups = maxBackups;
        
        Directory.CreateDirectory(logDir);
        InitializeCurrentWriter();
    }

    private void InitializeCurrentWriter()
    {
        string currentPath = Path.Combine(_logDir, $"{_baseName}.current.log");
        _currentWriter = new StreamWriter(currentPath, append: true, Encoding.UTF8);
        _currentSize = new FileInfo(currentPath).Length;
    }

    public async Task LogAsync(string message)
    {
        string logLine = $"[{DateTime.Now:yyyy-MM-dd HH:mm:ss.fff}] {message}";
        long lineSize = Encoding.UTF8.GetByteCount(logLine + Environment.NewLine);
        
        // 용량 초과 시 회전(Rotation) 수행
        if (_currentSize + lineSize > _maxFileSize)
        {
            await RotateLogFileAsync();
        }
        
        await _currentWriter.WriteLineAsync(logLine);
        await _currentWriter.FlushAsync();
        _currentSize += lineSize;
    }

    private async Task RotateLogFileAsync()
    {
        await _currentWriter.FlushAsync();
        _currentWriter.Dispose(); // 현재 파일 잠금 해제
        
        string currentPath = Path.Combine(_logDir, $"{_baseName}.current.log");
        string timestamp = DateTime.Now.ToString("yyyyMMdd_HHmmss");
        string archivePath = Path.Combine(_logDir, $"{_baseName}_{timestamp}.log");
        
        File.Move(currentPath, archivePath); // 백업 파일로 이동
        
        // 비동기로 압축 진행 (GZip)
        await CompressAndCleanupAsync(archivePath);
        
        InitializeCurrentWriter(); // 새 로그 파일 생성
    }

    private async Task CompressAndCleanupAsync(string filePath)
    {
        // 1. 파일 압축
        await using (var source = File.OpenRead(filePath))
        await using (var dest = File.Create(filePath + ".gz"))
        await using (var gzip = new GZipStream(dest, CompressionMode.Compress))
        {
            await source.CopyToAsync(gzip);
        }
        File.Delete(filePath); // 원본 텍스트 파일 삭제
        
        // 2. 오래된 백업 파일 정리 (최대 개수 유지)
        //    파일명에 포함된 타임스탬프를 파싱하여 정렬 (CreationTime 사용 시 파일 복사/이동으로 변경될 수 있음)
        var backups = Directory.GetFiles(_logDir, $"{_baseName}_*.log.gz")
            .Select(f => new
            {
                Path = f,
                Timestamp = ExtractTimestampFromFileName(f)
            })
            .Where(x => x.Timestamp.HasValue)
            .OrderByDescending(x => x.Timestamp.Value)
            .ToList();
            
        for (int i = _maxBackups; i < backups.Count; i++)
        {
            File.Delete(backups[i].Path);
        }
    }

    private DateTime? ExtractTimestampFromFileName(string filePath)
    {
        string fileName = Path.GetFileNameWithoutExtension(Path.GetFileNameWithoutExtension(filePath)); // .log.gz -> .log
        var match = Regex.Match(fileName, $@"{_baseName}_(\d{{8}}_\d{{6}})");
        if (match.Success && DateTime.TryParseExact(match.Groups[1].Value, "yyyyMMdd_HHmmss", null, 
                                                    System.Globalization.DateTimeStyles.None, out var ts))
        {
            return ts;
        }
        return null;
    }

    public async ValueTask DisposeAsync()
    {
        if (_currentWriter != null)
        {
            await _currentWriter.FlushAsync();
            _currentWriter.Dispose();
        }
    }
}
```

### 패턴 3: 대용량 JSON 스트리밍 파싱 (C# 8+, .NET 6+ 권장)

수 GB에 달하는 JSON 배열 데이터를 메모리에 한 번에 로드하지 않고, 한 건씩 파싱하여 스트리밍하는 모던 기법입니다.

⚠️ **주의**: `JsonDocument.ParseAsync`는 전체 JSON을 메모리에 로드하여 DOM을 구성하므로, 진정한 스트리밍이라고 할 수 없습니다. 아래 예제에서는 .NET 6 이상에서 제공하는 `JsonSerializer.DeserializeAsyncEnumerable`을 사용하여 메모리 사용량을 최소화합니다.

```csharp
using System.Text.Json;

public async IAsyncEnumerable<Product> ReadJsonArrayStreamAsync(string filePath, CancellationToken cancellationToken = default)
{
    await using var fileStream = File.OpenRead(filePath);
    
    var options = new JsonSerializerOptions
    {
        PropertyNameCaseInsensitive = true
    };

    // DeserializeAsyncEnumerable은 JSON 배열의 각 요소를 스트리밍 방식으로 역직렬화합니다.
    await foreach (var product in JsonSerializer.DeserializeAsyncEnumerable<Product>(fileStream, options, cancellationToken))
    {
        if (product is not null)
            yield return product;
    }
}

// 호출부
await foreach (var product in ReadJsonArrayStreamAsync("massive_products.json"))
{
    Console.WriteLine($"상품 처리 중: {product.Name}"); // OOM(메모리 부족) 걱정 없이 처리 가능
}
```

> **참고**: `DeserializeAsyncEnumerable`은 .NET 6 이상에서 사용할 수 있습니다. 만약 하위 버전을 사용해야 한다면 `Utf8JsonReader`를 직접 활용하는 저수준 구현이 필요하지만, 복잡도가 크게 증가합니다. 대부분의 최신 프로젝트는 .NET 6+를 대상으로 하므로 이 예제로 충분합니다.

## 5. 방어적 파일 I/O: 예외 처리와 재시도 로직

파일 입출력은 애플리케이션 외부의 환경(디스크, 네트워크, 안티바이러스 프로그램 등)에 의존하므로 예외 발생 빈도가 매우 높습니다. 특히 다른 프로세스(백신 등)가 파일을 일시적으로 물고 있어 발생하는 **공유 위반(Sharing Violation)**은 반드시 재시도(Retry) 로직으로 방어해야 합니다.

```csharp
public async Task<string> SafeReadFileAsync(string filePath)
{
    const int maxRetries = 5;
    int retryCount = 0;
    
    while (true)
    {
        try
        {
            return await File.ReadAllTextAsync(filePath, Encoding.UTF8).ConfigureAwait(false);
        }
        catch (IOException ex) when (ex.HResult == unchecked((int)0x80070020)) // 32: 공유 위반 에러 코드
        {
            if (++retryCount >= maxRetries) throw;
            
            Console.WriteLine($"파일 잠김 감지. 재시도 중... ({retryCount}/{maxRetries})");
            
            // 지수 백오프(Exponential Backoff): 대기 시간을 점점 늘려 부하를 줄임
            int delayMs = 100 * (int)Math.Pow(2, retryCount); // 100, 200, 400, 800, 1600ms ...
            await Task.Delay(delayMs).ConfigureAwait(false);
        }
        catch (UnauthorizedAccessException)
        {
            Console.WriteLine("권한 부족. 관리자 권한을 확인하세요.");
            throw;
        }
        catch (DirectoryNotFoundException)
        {
            Console.WriteLine("잘못된 디렉터리 경로입니다.");
            throw;
        }
    }
}
```

## 마무리

C#의 파일 입출력 시스템은 단순한 스크립트 작성부터 하이엔드 서버의 스토리지 제어까지 모두 소화할 수 있는 견고한 아키텍처를 갖추고 있습니다.

실무에서 파일 I/O를 다룰 때는 다음 핵심 원칙을 반드시 기억하십시오.

1. **메모리를 존중하라**: 용량을 알 수 없는 파일은 무조건 `Stream`을 사용하여 청크(Chunk) 단위로 읽으세요.
2. **비동기를 기본으로 하라**: 디스크 I/O 구간에서 스레드가 멈춰있지 않도록 `async/await`와 `FileOptions.Asynchronous`를 습관화하세요.
3. **자원을 확실히 놓아주어라**: `using` 문(또는 `await using`)을 사용하여 사용이 끝난 스트림과 파일 핸들을 즉각적으로 OS에 반환하세요.
4. **최악을 대비하라**: 도중에 전원이 나가도 데이터가 깨지지 않게 원자적(Atomic) 업데이트를 사용하고, 파일 잠김에 대비한 재시도 로직을 구축하세요. 또한 파일 시스템 경계(네트워크 드라이브 등)에서의 원자성 한계를 인지하세요.
