## 3) 2024-08-14 시행 개정 상세(실무 번역)

### 3.1 침해사고 보고·제재 실효성 강화
- **신고 시한·항목**이 **구체화**되었고, 유형에 따라 **신속 보고(예: T+24h)** 및 **사후보완**이 요구된다.
- **사전예방·개선명령** 근거가 정비되어 **사고 이전의 위험요인**에도 명령 가능.
- **과징금·과태료 상향**, **반복/고의 위반 가중** → **문서로만 존재하는 프로그램**이 아닌, 실제 작동하는 **컴플라이언스 체계** 요구.

### 3.2 본인확인기관(CI) 안전성 확보조치
- **CI(연계정보)** 생성·관리·제공 전 과정의 **기술·관리 통제**(권한 최소화, 암호화·전송보안, 로그·점검·위탁관리, 보관기간·파기)가 명문화. **연령확인** 체계와도 결합.

### 3.3 불법스팸 규제 최신화
- **전송자 처벌 수위**와 **통신사 전송방지 의무** 강화, **표시의무** 정비. 발송 정책·대행사 계약·사전차단 프로세스를 업데이트할 것. (운영·감사 포인트는 §6.3 참고)

### 3.4 정보보호 거버넌스(CISO)
- **CISO 지정·신고·겸직 제한** 요건을 **시행령**이 구체화(자산·매출·업종 등). **직보 라인**과 **연 1회 이상 이사회 보고**를 내규화해 독립성과 실효성을 확보.

### 6.3 불법스팸 준수 — 수신동의·야간전송 제한 로직

def can_send_ad(user, now_kst, purpose, channel):
    # 1) 유효한 수신동의 증빙
    consent = get_latest_consent(user.id, channel)
    if not (consent and consent.active and consent.purpose == purpose):
        return False, "no valid opt-in"

    # 2) 야간전송 제한(예: 21:00~08:00 내부정책)
    if now_kst.hour >= 21 or now_kst.hour < 8:
        return False, "night-time restriction"

    # 3) 필수 표기(광고, 발신번호, 수신거부) 준비 상태 확인
    return True, "ok"

## 9) 모의 점검 Q&A(샘플)

1) **Q: 인지 후 36시간째 1차 보고**  
   **A:** **SLA 미충족 위험** — 내부 타임라인을 “**T+12h 초안 / T+24h 본보고**”로 재설계. 증빙 수집은 **즉시 개시**(로그 롤오버 방지).  

2) **Q: CI DB에 운영자 전원 SELECT 가능**  
   **A:** **역할 최소화**·접근사유 기록·승인, **필드 암호화(AEAD/FPE)**·Column-level 마스킹을 적용.  

3) **Q: 마케팅 대행사에 수신동의 이력 위탁**  
   **A:** 위탁계약에 **보존·열람·감사권·사고통지**를 명시하고 **월간 리포트** 의무화. (불법스팸 가이드 최신화 반영)

4) **Q: CISO가 개발본부장 겸직(대규모)**  
   **A:** **겸직 제한**·독립성 기준 점검. 보고 라인·대리체계 보완.

## 11) 부록 — 즉시 활용 가능한 산출물

### A) 침해사고 보고서(요약판)

```markdown
# 침해사고 보고서(요약)
- 사건번호 / 버전 / 작성일
- 1. 개요: 인지시각, 영향시스템, 서비스 영향
- 2. 경과: T+0~T+24h 타임라인(담당자, 조치목록)
- 3. 피해범위: 유형·건수·데이터범주
- 4. 원인: 공격기법·오구성·취약점(CVE 등)
- 5. 조치: 격리·복구·패치·키교체
- 6. 재발방지: 아키텍처·정책·모니터링 개선
- 7. 증빙: 로그·이미지·패킷·체인오브커스터디(해시/서명)
- 8. 연락책임자: CISO/대리, 컴플라이언스, 법무
```

### D) 리스크·SLA 산정 간단식

$$
\text{Incident\_Penalty\_Risk} \approx \text{Impact} \times \text{Non\_Compliance\_Probability}
$$

- **Impact**: 유저 규모·데이터 민감도·가용성 영향  
- **Non_Compliance_Probability**: 보고 지연/제출 누락·명령 미이행 가능성

## 12) 결론 — “규정 ↔ 코드 ↔ 증적”의 닫힌 고리

1) **규정**: 망법/시행령 최신본을 주기적으로 검토해 **SOP·SLA**로 내재화.  
2) **코드/설정**: **암호화·전송보안·권한분리·사전차단**을 실제 서비스 코드·플랫폼 설정으로 구현.  
3) **증적**: **보고 타임라인·로그 무결성·이행기록**을 자동 수집·보관·감사.

> 규제는 **최소선**이다. 위협 환경(예: DBIR 2024, ENISA TL 2024)을 반영한 **추가 통제**(EDR/XDR, 데이터분류·DLP, 취약성 상시관리)가 필수.
