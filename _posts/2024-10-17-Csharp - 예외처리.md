# C# 예외 처리: 견고한 시스템을 위한 방어적 프로그래밍

소프트웨어 개발에서 오류는 피할 수 없는 현실입니다. 네트워크 연결 끊김, 파일 시스템 권한 문제, 잘못된 사용자 입력 등 다양한 이유로 프로그램은 예기치 않은 상황에 직면합니다. C#의 예외 처리(Exception Handling) 시스템은 이러한 비정상적인 상황을 체계적으로 관리하고, 애플리케이션의 견고성과 사용자 경험을 향상시키는 가장 강력한 도구입니다.

예외 처리의 핵심 철학은 **"예외는 오류 신호일 뿐, 일반적인 제어 흐름(Control Flow)으로 사용해서는 안 된다"**는 것입니다. 이 글에서는 예외 처리의 기본 구조부터 스택 추적(Stack Trace)의 보존, 비동기 환경에서의 예외 처리, 그리고 엔터프라이즈급 전역 예외 처리 아키텍처까지 실무에 필요한 모든 것을 깊이 있게 다루어 보겠습니다.

## 기본 구조: try-catch-finally

C# 예외 처리의 가장 뼈대가 되는 구성 요소입니다. 예외가 발생할 수 있는 위험한 코드를 격리하고, 오류를 안전하게 잡아내며, 성공 여부와 관계없이 사용한 자원을 정리합니다.

### catch 블록의 순서 규칙

예외를 잡을 때는 반드시 **가장 구체적인 예외부터 일반적인 예외 순서로** 작성해야 합니다. 컴파일러는 위에서부터 아래로 `catch` 블록을 평가하며, 첫 번째로 조건이 일치하는 블록 안으로 진입합니다.

```csharp
try
{
    // 파일 I/O 등 예외 발생 가능성이 있는 코드
    string content = File.ReadAllText("data.txt");
}
catch (FileNotFoundException ex)  // 1. 가장 구체적인 예외
{
    Console.WriteLine("파일을 찾을 수 없습니다.");
}
catch (IOException ex)            // 2. 더 넓은 범위의 입출력 예외
{
    Console.WriteLine("입출력 오류가 발생했습니다.");
}
catch (Exception ex)              // 3. 모든 예외의 조상 (가장 마지막에 위치해야 함)
{
    Console.WriteLine($"예상치 못한 치명적 오류: {ex.Message}");
}

```

### finally 블록: 무조건 실행되는 보증 수표

`finally` 블록은 `try` 블록 안에서 예외가 발생하든 안 하든, 심지어 `catch` 블록 안에서 `return`을 만나 함수를 빠져나가려 할 때조차도 반드시 실행됩니다. 열어둔 파일 스트림이나 데이터베이스 연결 등 외부 자원을 안전하게 닫아주는(해제하는) 용도로 사용됩니다.

## 예외 발생과 다시 던지기 (Rethrowing)의 함정

시스템에 문제가 생겼을 때 의도적으로 예외를 발생시키려면 `throw` 키워드를 사용합니다. 이때 기존에 잡았던 예외를 상위 호출자에게 다시 넘겨야 할 때, 수많은 주니어 개발자가 치명적인 실수를 범하곤 합니다.

```csharp
public void ProcessData(string data)
{
    try
    {
        ValidateAndProcess(data);
    }
    catch (FormatException ex)
    {
        Console.WriteLine("데이터 형식 오류 발생, 상위로 에러를 전파합니다.");
        
        // [안 좋은 예] throw ex; 
        // 기존의 스택 추적(Stack Trace) 정보를 모조리 날려버리고, 
        // 마치 여기서 예외가 처음 발생한 것처럼 덮어씌웁니다. 디버깅이 불가능해집니다.
        
        // [좋은 예] throw;
        // 원래 예외가 발생했던 깊은 호출 스택의 경로를 그대로 보존한 채 상위로 던집니다.
        throw; 
    }
}

```

### 예외 래핑 (Exception Wrapping)

하위 계층(예: 데이터베이스 통신 레이어)에서 발생한 기술적인 예외를 그대로 UI 레이어까지 던지는 것은 캡슐화 원칙에 어긋납니다. 이를 도메인에 맞는 의미 있는 예외로 감싸서(래핑하여) 던지는 패턴이 자주 사용됩니다.

```csharp
public User GetUser(int id)
{
    try
    {
        // DB 통신
        return Database.Query<User>($"SELECT * FROM Users WHERE Id = {id}");
    }
    catch (SqlException sqlEx)
    {
        // 내부 기술 예외(SqlException)를 InnerException으로 품은 채, 
        // 비즈니스 의미를 담은 DataAccessException으로 포장하여 던집니다.
        throw new DataAccessException($"사용자 ID {id} 조회 중 DB 오류 발생", sqlEx);
    }
}

```

## using 문: try-finally의 우아한 대체재

`finally` 블록에서 매번 `.Dispose()`를 호출하여 자원을 해제하는 것은 코드를 길고 지루하게 만듭니다. C#은 `IDisposable` 인터페이스를 구현하는 객체에 대해 `using` 문을 제공하여 이 과정을 자동화합니다.

```csharp
// C# 8.0 이전의 전통적인 using 블록 (내부적으로 try-finally로 변환됨)
using (var fileStream = File.Open("data.txt", FileMode.Open))
{
    byte[] buffer = new byte[1024];
    fileStream.Read(buffer, 0, buffer.Length);
} // 중괄호를 벗어나는 순간 자동으로 Dispose()가 호출됩니다.

// C# 8.0 이상: using 선언 (가장 권장되는 모던 문법)
using var readerStream = File.Open("data2.txt", FileMode.Open);
using var reader = new StreamReader(readerStream);
string content = await reader.ReadToEndAsync();
// 현재 속한 블록(메서드 등)이 끝날 때 역순으로 자동으로 Dispose()가 호출됩니다.

```

## 예외 필터: when 키워드의 마법

C# 6.0부터 도입된 예외 필터(`when` 키워드)는 지정된 조건이 참일 때만 `catch` 블록을 실행하도록 합니다.

이 기능이 강력한 이유는, 조건이 맞지 않아 예외를 잡지 않을 경우 **스택 풀기(Stack Unwinding)가 발생하지 않아 메모리 덤프의 상태가 온전히 보존된다**는 점입니다. 이를 응용하면 예외의 흐름을 방해하지 않고 몰래 로그만 남기는 고급 기법을 작성할 수 있습니다.

```csharp
try
{
    ProcessOrder(order);
}
catch (Exception ex) when (LogException(ex))
{
    // LogException이 무조건 false를 반환하므로 이 블록 안으로는 절대 들어오지 않습니다.
}

// 부작용 없이 예외를 로깅하기 위한 헬퍼 메서드
private static bool LogException(Exception ex)
{
    Console.WriteLine($"[CRITICAL] {ex.GetType().Name}: {ex.Message}");
    return false; // false를 반환하여 원래 예외가 호출 스택을 따라 계속 날아가도록 둡니다.
}

```

또한, 네트워크 호출 시 특정 HTTP 상태 코드(예: 500 내부 서버 오류)에 대해서만 재시도 로직을 태우는 분기 처리에도 매우 유용하게 쓰입니다.

## 비동기 환경(async/await)의 예외 처리

비동기 메서드에서의 예외 처리는 겉보기엔 동기 코드와 똑같이 `try-catch`를 쓰면 되지만, 내부 동작에 주의해야 할 점들이 있습니다.

### Task.WhenAll과 AggregateException

여러 개의 비동기 작업을 동시에 쏘아 보내고 기다릴 때, 그중 여러 개에서 동시에 예외가 발생할 수 있습니다. 이때 시스템은 여러 예외를 하나로 묶어 `AggregateException`으로 던집니다.

```csharp
public async Task ProcessMultipleUrlsAsync(string[] urls)
{
    var tasks = urls.Select(url => DownloadContentAsync(url)).ToList();
    
    try
    {
        // 모든 다운로드 작업이 끝날 때까지 병렬로 대기합니다.
        await Task.WhenAll(tasks);
    }
    catch (Exception)
    {
        // 주의: await Task.WhenAll()에 단순 try-catch를 걸면, 
        // 여러 예외 중 가장 '첫 번째' 예외만 잡히게 됩니다.
        
        // 모든 예외의 목록을 확인하려면 Task 객체의 Exception 속성을 열어보아야 합니다.
        var allExceptions = tasks.Where(t => t.IsFaulted)
                                 .SelectMany(t => t.Exception!.InnerExceptions);
                                 
        foreach (var ex in allExceptions)
        {
            Console.WriteLine($"- 실패 원인: {ex.Message}");
        }
    }
}

```

## 예외 처리 vs 반환 코드 (성능 관점의 설계)

예외 처리는 스택 추적을 구성하고 컨텍스트를 수집하는 과정에서 상당한 성능 비용(Overhead)을 요구합니다. 따라서 "사용자가 문자를 입력해야 하는데 숫자를 입력한 상황"처럼 **충분히 예상 가능하고 빈번하게 발생하는 논리적 오류**에는 예외를 던지는 대신 `Try-` 패턴을 사용하는 것이 정석입니다.

```csharp
// [성능 저하 패턴] 예외에 의존하는 검증
public int ConvertToNumber(string input)
{
    // 실패할 때마다 무거운 FormatException이 생성되고 스택 트레이스가 캡처됩니다.
    return int.Parse(input); 
}

// [성능 최적화 패턴] Try-Parse 패턴 사용
public bool TryConvert(string input, out int result)
{
    // 예외를 전혀 발생시키지 않고, 논리적인 반환값(bool)으로 성공 여부만 가볍게 전달합니다.
    return int.TryParse(input, out result);
}

```

## 애플리케이션 수준의 전역 예외 처리 (Global Exception Handling)

아무리 `try-catch`를 꼼꼼히 작성해도, 개발자가 미처 예측하지 못한 예외는 발생하기 마련입니다. 이 예외가 프로그램의 메인 스레드를 뚫고 나가면 애플리케이션은 사용자에게 흉측한 크래시 화면을 띄우며 강제 종료됩니다.

이를 막기 위해 시스템의 가장 바깥쪽 외곽(최상위 레이어)에 전역 예외 처리기(Global Exception Handler)를 설치하여, 애플리케이션이 우아하게 죽거나 사용자에게 친절한 에러 메시지를 반환하도록 아키텍처를 구성해야 합니다.

### ASP.NET Core 미들웨어 활용

웹 서버 환경에서는 컨트롤러마다 예외 처리를 중복해서 작성하지 않고, 파이프라인의 최상단 미들웨어에서 모든 예외를 낚아채 규격화된 JSON 응답으로 변환합니다.

```csharp
public class ExceptionHandlingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ExceptionHandlingMiddleware> _logger;
    
    public ExceptionHandlingMiddleware(RequestDelegate next, ILogger<ExceptionHandlingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }
    
    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            // 다음 파이프라인(컨트롤러 등)으로 요청을 넘깁니다.
            await _next(context);
        }
        catch (Exception ex)
        {
            // 컨트롤러에서 처리되지 않고 튕겨 나온 모든 예외가 이곳으로 모입니다.
            _logger.LogError(ex, "서버에서 처리되지 않은 예외가 발생했습니다.");
            await HandleExceptionAsync(context, ex);
        }
    }
    
    private static async Task HandleExceptionAsync(HttpContext context, Exception ex)
    {
        context.Response.ContentType = "application/json";
        
        // 예외 타입에 따라 HTTP 상태 코드와 메시지를 분기 처리합니다.
        context.Response.StatusCode = ex switch
        {
            UnauthorizedAccessException => StatusCodes.Status401Unauthorized,
            ArgumentException => StatusCodes.Status400BadRequest,
            _ => StatusCodes.Status500InternalServerError
        };
        
        var errorResponse = new
        {
            error = ex.GetType().Name,
            message = "요청을 처리하는 도중 문제가 발생했습니다.",
            // 보안을 위해 실제 Stack Trace는 운영 환경에서 절대 노출하지 않습니다.
        };
        
        await context.Response.WriteAsJsonAsync(errorResponse);
    }
}

```

## 마무리: 견고한 예외 처리 전략

예외 처리는 땜질식 처방이 아니라, 시스템의 신뢰성을 결정짓는 핵심 설계(Architecture) 요소입니다.

1. **도메인 예외를 설계하십시오**: 기본 `Exception`을 던지기보다 시스템의 비즈니스 의미를 담은 사용자 정의 예외(`InsufficientFundsException` 등)를 명확히 정의하세요.
2. **층(Layer)에 맞는 책임을 부여하십시오**: 데이터베이스 연결 오류는 리포지토리에서 로깅하고, 비즈니스 규칙 위반은 서비스 레이어에서 처리하며, UI 레이어에서는 사용자 친화적인 메시지로 변환하세요.
3. **자원 누수를 막으십시오**: 외부 자원(파일, 네트워크, DB)을 사용할 때는 무조건 `using` 선언을 통해 안전망을 확보하세요.
4. **성능을 저울질하십시오**: 일상적인 실패 흐름은 예외가 아니라 `TryXXX` 패턴의 반환 코드로 처리하여 불필요한 시스템 리소스 낭비를 막으십시오.

이러한 원칙들을 염두에 두고 코드를 작성한다면, 어떠한 돌발 상황이 발생하더라도 당황하지 않고 매끄럽게 대응하는 최고 수준의 C# 애플리케이션을 완성할 수 있을 것입니다.
