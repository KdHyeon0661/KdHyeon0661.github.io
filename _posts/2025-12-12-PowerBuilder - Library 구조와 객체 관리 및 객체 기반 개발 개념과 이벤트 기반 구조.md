# Library(PBL) 구조와 객체 관리 및 객체 기반 개발 개념과 이벤트 기반 구조

PowerBuilder 애플리케이션은 여러 객체들이 모여 구성되며, 이러한 객체들은 라이브러리 파일에 저장되고 관리됩니다. 또한 PowerBuilder는 객체 지향 개념과 이벤트 기반 프로그래밍 모델을 채택하여, 유지보수와 확장이 용이한 구조를 제공합니다. 이 글에서는 PBL 파일의 구조와 객체 관리 방법, 그리고 PowerBuilder에서의 객체 지향 개발 개념과 이벤트 기반 구조의 핵심을 자세히 살펴보겠습니다.

---

## Library(PBL) 구조와 객체 관리

### PBL 파일이란?

PBL(PowerBuilder Library)은 PowerBuilder 객체(윈도우, DataWindow, 메뉴, 함수, 구조체 등)를 저장하는 물리적인 파일입니다. 확장자는 `.pbl`이며, 여러 객체를 하나의 PBL에 저장할 수 있습니다.

- PBL은 **바이너리 형식**으로 저장되며, 일반 텍스트 편집기로 열 수 없습니다.
- 하나의 애플리케이션은 여러 개의 PBL로 구성될 수 있으며, 필요에 따라 라이브러리 검색 경로를 통해 참조합니다.
- PBL은 소스 코드 저장소 역할과 동시에 컴파일된 중간 코드를 포함하여 실행 시 직접 사용됩니다.

### PBL의 내부 구조

PBL 파일은 내부적으로 객체별로 저장되며, 각 객체는 다음과 같은 정보를 포함합니다.

- **객체 헤더**: 객체 이름, 유형, 생성 일자, 수정 일자, 크기 등의 메타데이터
- **소스 코드**: 객체에 연결된 PowerScript 코드 및 속성 정의
- **컴파일된 코드**: 실행 시 빠르게 로드할 수 있는 중간 코드(p-code) 또는 기계어 코드
- **종속성 정보**: 다른 객체에 대한 참조 정보

PBL 파일은 라이브러리 관리자(Library Painter)를 통해 체계적으로 관리할 수 있습니다.

```
[PBL 파일 구조 (개념적)]
┌─────────────────────────────────┐
│        PBL 파일 헤더              │
├─────────────────────────────────┤
│ 객체 1: 윈도우 w_main             │
│   ├─ 헤더 (이름, 유형, 날짜)       │
│   ├─ 속성/스크립트                 │
│   └─ 컴파일된 코드                 │
├─────────────────────────────────┤
│ 객체 2: DataWindow d_customer     │
│   ├─ 헤더                         │
│   ├─ 속성/스크립트                 │
│   └─ 컴파일된 코드                 │
├─────────────────────────────────┤
│ 객체 3: 함수 f_calc_tax           │
│   ├─ 헤더                         │
│   ├─ 스크립트                      │
│   └─ 컴파일된 코드                 │
└─────────────────────────────────┘
```

### 라이브러리 검색 경로(Library Search Path)

애플리케이션 실행 시, PowerBuilder는 특정 순서대로 PBL을 검색하여 필요한 객체를 찾습니다. 이 순서는 Application Object의 속성에서 **Library Search Path**로 정의됩니다.

- 세미콜론(;)으로 구분된 PBL 파일 목록을 지정합니다.
- 앞에 나열된 PBL일수록 우선순위가 높습니다.
- 동일한 이름의 객체가 여러 PBL에 존재하면 먼저 발견된 객체가 사용됩니다.

예시:
```
C:\MyApp\main.pbl;C:\MyApp\common.pbl;C:\Shared\utils.pbl
```

이러한 구조는 다음과 같은 이점을 제공합니다.
- 공통 라이브러리를 분리하여 여러 애플리케이션에서 재사용 가능
- 버그 수정이나 업데이트 시 해당 PBL만 교체하면 됨
- 개발 단계와 운영 단계의 라이브러리 분리

### Library Painter를 통한 객체 관리

PowerBuilder IDE의 **Library Painter**는 PBL 파일과 그 안의 객체를 관리하는 전용 도구입니다. Library Painter에서는 다음과 같은 작업을 수행할 수 있습니다.

- **객체 조회**: 특정 PBL 내 객체 목록 확인
- **객체 복사/이동/삭제**: 다른 PBL로 객체 복사 또는 이동
- **객체 내보내기(Export)**: 객체를 텍스트 형식(.srw, .srd 등)으로 저장하여 버전 관리 시스템과 연동
- **객체 가져오기(Import)**: 텍스트 파일에서 객체를 PBL로 복원
- **라이브러리 검색**: 이름이나 유형으로 객체 검색
- **체크인/체크아웃**: 소스 제어 시스템과 연동하여 객체 버전 관리 (선택 사항)

```
[Library Painter 화면 예시]
C:\MyApp\main.pBL
   ├─ w_main (Window)
   ├─ w_login (Window)
   ├─ d_customer (DataWindow)
   └─ u_std_button (User Object)

C:\MyApp\common.pBL
   ├─ f_round (Function)
   ├─ n_calc (Non-Visual Object)
   └─ s_address (Structure)
```

> **💡 실무 팁: PBL 최적화(Optimize)**  
> 객체를 자주 수정하고 삭제하다 보면 PBL 파일 내부에 불필요한 찌꺼기가 쌓여 용량이 커지고 파일이 손상될 수 있습니다. Library Painter에서 정기적으로 PBL 파일을 마우스 오른쪽 버튼으로 클릭한 후 **Optimize** 메뉴를 실행해 주는 것이 좋습니다. 최적화를 통해 내부 단편화를 제거하고 파일 크기를 줄일 수 있습니다.

### 객체 종속성 관리

PowerBuilder 객체는 다른 객체를 참조할 수 있습니다. 예를 들어, 윈도우에 DataWindow 컨트롤이 배치되면 해당 DataWindow 객체에 대한 종속성이 생깁니다. 객체를 삭제하거나 이동할 때는 종속성을 고려해야 합니다.

- **종속성 확인 방법**: Library Painter에서 객체의 **속성**을 보거나, **Browse** 기능을 통해 참조하는 객체 목록을 확인할 수 있습니다.
- **순환 참조 방지**: 객체 설계 시 순환 참조는 피하는 것이 좋습니다.

## 객체 기반 개발 개념

PowerBuilder는 객체 지향 프로그래밍(Object-Oriented Programming, OOP) 개념을 지원합니다. 이는 코드 재사용성, 유지보수성, 확장성을 높이는 데 기여합니다.

### PowerBuilder의 객체 지향 요소

#### 클래스(Class)와 인스턴스(Instance)
- **클래스**: 객체의 설계도입니다. PowerBuilder의 모든 객체(윈도우, 메뉴, 사용자 객체 등)는 클래스에 해당합니다.
- **인스턴스**: 클래스를 기반으로 메모리에 생성된 실체입니다. `OPEN` 함수로 윈도우 인스턴스를 생성하거나, `CREATE` 문으로 비시각적 객체 인스턴스를 생성합니다.

#### 상속(Inheritance)
PowerBuilder는 시각적 객체(윈도우, 메뉴, 사용자 객체)와 비시각적 객체(클래스 사용자 객체)에서 상속을 지원합니다.
- 상속을 통해 부모 객체의 속성, 이벤트, 함수, 구조를 그대로 물려받고, 필요한 부분만 재정의(Override)할 수 있습니다.
- 상속 계층은 **객체 브라우저(Object Browser)**에서 확인할 수 있습니다.

```
[상속 계층 예시]
   ┌─────────────────┐
   │  w_parent (부모) │ (공통 기능: 닫기 버튼, 배경색)
   └────────┬────────┘
            │ 상속
   ┌────────┴────────┐
   │ w_child (자식)   │ (추가 기능: 저장 버튼)
   └─────────────────┘
```

#### 다형성(Polymorphism)
다형성은 같은 이름의 메서드나 이벤트가 객체에 따라 다르게 동작하는 것을 의미합니다.
- 부모 클래스에서 정의된 함수를 자식 클래스에서 **오버라이드(Override)**하여 다른 기능을 구현할 수 있습니다.
- 사용자 정의 이벤트도 상속 관계에서 재정의 가능합니다.

#### 캡슐화(Encapsulation)
PowerBuilder에서는 인스턴스 변수(Instance Variable)와 함수, 이벤트에 대해 접근 지정자를 설정할 수 있습니다.
- **PUBLIC**: 모든 스크립트에서 접근 가능
- **PROTECTED**: 해당 객체와 상속받은 객체에서만 접근 가능
- **PRIVATE**: 해당 객체 내에서만 접근 가능

이를 통해 객체 내부 구현을 숨기고 인터페이스만 노출할 수 있습니다.

### 시각적 객체와 비시각적 객체

PowerBuilder의 객체는 크게 두 가지 유형으로 나눌 수 있습니다.

#### 시각적 객체(Visual Objects)
- 사용자에게 보이는 인터페이스 요소입니다.
- 예: Window, Menu, DataWindow 컨트롤, CommandButton, SingleLineEdit 등
- 이벤트(Clicked, Modified, etc.)가 풍부하게 정의되어 있습니다.

#### 비시각적 객체(Non-Visual Objects)
- 화면에 보이지 않고 비즈니스 로직, 데이터 처리, 공통 기능을 담당합니다.
- 예: Transaction 객체, DataStore, Custom Class 사용자 객체, 함수 등
- 시각적 객체에 종속되지 않으므로 재사용성이 높습니다.
- `CREATE` 문으로 인스턴스를 생성하고 `DESTROY`로 제거합니다.

```
// Custom Class 사용자 객체(n_calculator) 생성 및 소멸 예시
n_calculator lnv_calc          // 1. 변수 선언 (클래스 타입 지정)
lnv_calc = CREATE n_calculator // 2. 인스턴스 생성 (메모리 할당)

// 3. 객체의 함수나 속성 사용
Long ll_result
ll_result = lnv_calc.of_calculate(100, 200)

// 4. 사용 완료 후 반드시 소멸 (메모리 누수 방지)
DESTROY lnv_calc
```

### 객체 재사용과 상속의 실제 활용

PowerBuilder 개발에서 상속은 매우 중요한 재사용 기법입니다.
- **표준 윈도우 상속**: 공통 기능(예: 프로그램 종료, 도움말 호출)을 가진 기본 윈도우를 만들고, 모든 자식 윈도우가 이를 상속받아 개발 시간 단축.
- **표준 컨트롤 사용자 객체**: 자주 사용되는 버튼(조회, 저장, 삭제)을 표준 사용자 객체로 정의하여 일관된 동작과 디자인 적용.
- **비시각적 비즈니스 객체**: 데이터베이스 접근, 계산 로직 등을 별도의 클래스 사용자 객체로 작성하여 여러 윈도우에서 공유.

### 객체 브라우저(Object Browser)

PowerBuilder의 **Object Browser**는 현재 애플리케이션에 정의된 모든 객체, 시스템 객체, 전역 변수, 함수 등의 구조를 탐색할 수 있는 도구입니다.
- **Browse** 탭에서 객체 유형별로 목록 확인
- **상속 계층도** 표시
- 객체의 속성, 이벤트, 함수 시그니처 확인
- 코드 내에서 객체 이름 위에 커서를 놓고 F1을 누르면 브라우저로 이동

## 이벤트 기반(Event Driven) 구조

PowerBuilder는 전형적인 **이벤트 기반 프로그래밍** 모델을 채택하고 있습니다. 애플리케이션의 흐름은 사용자 동작(클릭, 키 입력 등)이나 시스템 상태 변화(타이머, 데이터베이스 완료 등)에 의해 발생하는 이벤트에 의해 결정됩니다.

### 이벤트(Event)의 개념

이벤트는 객체에서 발생하는 특정 사건을 나타냅니다. 각 이벤트에는 연결된 스크립트(핸들러)를 작성할 수 있으며, 해당 이벤트가 발생하면 자동으로 실행됩니다.

- **시스템 정의 이벤트**: PowerBuilder가 미리 정의해 놓은 이벤트로, 대부분의 객체가 공통으로 가집니다.
  - `Open`, `Close`, `Clicked`, `Modified`, `Timer`, `Key` 등
- **사용자 정의 이벤트**: 개발자가 필요에 따라 직접 정의할 수 있는 이벤트입니다.
  - 객체의 **Event List**에서 추가 가능
  - 고유한 이벤트 ID를 지정하거나, `pbm_custom01` ~ `pbm_custom15` 등의 매핑 가능
  - 이벤트를 발생시키려면 `TRIGGER EVENT` 또는 `POST EVENT` 문 사용

### 이벤트 핸들러 작성

각 객체(윈도우, 컨트롤, 사용자 객체 등)는 이벤트 목록을 가지고 있으며, 개발자는 원하는 이벤트에 스크립트를 작성합니다.

- 이벤트 스크립트는 PowerScript로 작성됩니다.
- 하나의 이벤트에는 여러 줄의 코드를 작성할 수 있습니다.
- 이벤트 내에서 다른 이벤트를 트리거하거나 포스트할 수 있습니다.

```
// 버튼 cb_ok의 Clicked 이벤트 예시
string ls_name
ls_name = sle_name.Text
IF ls_name = "" THEN
   MessageBox("경고", "이름을 입력하세요.")
   RETURN
END IF
cb_ok.Enabled = FALSE   // 버튼 비활성화
Open(w_main)            // 메인 윈도우 열기
```

### 이벤트 전달 방식: Trigger vs Post

이벤트를 프로그래밍 방식으로 발생시킬 때는 두 가지 방법이 있습니다.

#### TRIGGER EVENT
- **동기적(Synchronous)**으로 실행됩니다.
- 이벤트 핸들러가 완료될 때까지 현재 스크립트의 실행이 중단됩니다.
- 즉시 처리가 필요한 상황에 사용합니다.

```
// TRIGGER 예시: 즉시 실행
Parent.Trigger Event pbm_custom01()
```

#### POST EVENT
- **비동기적(Asynchronous)**으로 실행됩니다.
- 현재 스크립트는 계속 실행되고, 이벤트는 나중에(현재 이벤트 핸들러 종료 후) 메시지 큐에 쌓여 처리됩니다.
- 시간이 오래 걸리는 작업이나, 현재 작업에 영향을 주지 않으면서 추가 처리가 필요할 때 유용합니다.

```
// POST 예시: 메시지 큐에 넣고 바로 다음 줄 실행
Parent.Post Event pbm_custom01()
```

### 이벤트 실행 순서와 상속

상속 관계에서 이벤트는 부모 객체의 이벤트 스크립트와 자식 객체의 이벤트 스크립트가 연결되어 실행될 수 있습니다. PowerBuilder의 기본 동작은 **Extend ancestor script**입니다.

- **Extend (확장, 기본값)**: 부모 객체의 이벤트 스크립트가 먼저 실행된 후, 이어서 자식 객체의 스크립트가 실행됩니다.
- **Override (재정의)**: 부모의 스크립트를 무시하고 자식 객체에 작성된 스크립트만 실행됩니다. (이벤트 속성 창에서 "Override ancestor script"를 선택)
- **CALL 문 사용**: Override 상태에서 특정 시점에 부모 스크립트를 강제로 호출하고 싶을 때 `CALL` 문을 사용합니다.

```
// 자식 윈도우의 Open 이벤트에서 부모 Open 호출
CALL w_parent::Open   // 부모의 Open 이벤트 스크립트 실행
// 이후 자식의 추가 코드 실행
```

> **참고**: 이벤트 속성 창(Event Detail)에서 "Extend Ancestor Script" 체크박스를 통해 기본 동작을 제어할 수 있습니다.

### 이벤트와 함수의 비교

| 구분 | 이벤트(Event) | 함수(Function) |
|------|--------------|----------------|
| 목적 | 특정 사건에 대응 | 특정 기능 수행 |
| 호출 방식 | 시스템에 의해 자동 호출, 또는 Trigger/Post로 호출 | 스크립트에서 직접 호출 |
| 반환값 | 있을 수 있음 (Long, String 등) | 반환형 정의 필수 |
| 상속 동작 | Extend/Override 선택 가능 | 오버라이드 가능, 부모 호출 방식 다름 |
| 비동기 실행 | POST EVENT 가능 | 비동기 실행 불가 |

일반적으로 단순한 사건 처리에는 이벤트를, 계산이나 공통 로직에는 함수를 사용합니다. 다만, PowerBuilder에서는 사용자 정의 이벤트도 함수처럼 인자를 전달하고 결과를 반환받을 수 있어 유연하게 사용됩니다.

## 실무 팁: 효율적인 객체 관리와 이벤트 설계

### 라이브러리 분할 전략
- **기능별 분할**: 화면 관련 객체(Window, DataWindow)는 `ui.pbl`, 비즈니스 로직 객체는 `business.pbl`, 공통 유틸리티는 `common.pbl`로 분리.
- **계층별 분할**: 프레젠테이션 계층, 비즈니스 계층, 데이터 접근 계층으로 PBL을 나누어 향후 3-Tier 전환에 대비.

### 객체 명명 규칙
일관된 명명 규칙은 유지보수성을 높입니다.
- 윈도우: `w_` 접두사 (예: `w_customer_list`)
- DataWindow: `d_` 접두사 (예: `d_customer_grid`)
- 사용자 객체: `u_` (시각적), `n_` (비시각적) 접두사
- 함수: `f_` 접두사
- 구조체: `s_` 접두사

### 이벤트 설계 가이드
- **이벤트 이름은 직관적으로**: 사용자 정의 이벤트는 `ue_` 접두사(예: `ue_refresh_data`) 사용.
- **과도한 이벤트 연결 피하기**: 이벤트가 너무 많으면 디버깅이 어려워짐.
- **이벤트와 함수의 적절한 분리**: 복잡한 로직은 함수로 분리하고, 이벤트에서는 함수를 호출하는 방식으로 구성.

## 결론

PowerBuilder의 라이브러리(PBL) 구조는 객체를 물리적으로 저장하고 검색 경로를 통해 유연하게 참조할 수 있게 합니다. Library Painter를 활용하면 객체 관리가 용이하며, 정기적인 Optimize를 통해 PBL의 건강 상태를 유지할 수 있습니다.

객체 기반 개발 개념(상속, 캡슐화, 다형성)을 적용하면 재사용성과 유지보수성이 높은 애플리케이션을 구축할 수 있습니다. 시각적 객체와 비시각적 객체를 적절히 조합하여 관심사를 분리하는 것이 중요합니다.

마지막으로 이벤트 기반 구조는 사용자와 시스템의 상호작용을 자연스럽게 처리하며, Extend/Override 기본 동작과 Trigger/Post의 차이를 이해하면 효율적인 이벤트 설계가 가능합니다. 이러한 개념들을 잘 활용하면 PowerBuilder로 견고하고 확장 가능한 비즈니스 애플리케이션을 개발할 수 있을 것입니다.
