# C# 인터페이스: 객체 지향 설계의 우아한 계약서

객체 지향 프로그래밍에서 **인터페이스(Interface)**는 클래스가 "무엇을 할 수 있어야 하는가"에 대한 엄격한 계약(Contract)을 정의합니다. 내부적으로 어떻게 동작하는지(구현 세부 사항)는 전혀 관여하지 않고, 오직 외부와 소통할 수 있는 입출구(메서드, 속성 등)의 형태만을 명시합니다.

이러한 철저한 추상화는 클래스 간의 결합도를 낮추고, 런타임에 부품을 갈아 끼우듯 다형성을 실현하는 현대 C# 소프트웨어 설계의 핵심입니다. 이 글에서는 인터페이스의 기초부터 시작하여, C# 8.0 이상에서 도입된 기본 구현(Default Implementation) 등 강력한 최신 기능과 실무 설계 패턴까지 모두 다루어 보겠습니다.

## 인터페이스의 본질과 기본 문법

인터페이스는 클래스가 구현해야 할 청사진입니다. 필드(상태)를 가질 수 없으며, 오직 메서드, 속성, 인덱서, 이벤트의 시그니처(이름과 매개변수)만을 정의합니다. 인터페이스의 이름은 관례적으로 대문자 `I`로 시작합니다.

```csharp
public interface IShape
{
    // 접근 제한자를 명시하지 않아도 기본적으로 public입니다.
    void Draw();
    double GetArea();
}

// 클래스는 콜론(:)을 사용하여 인터페이스를 '구현'합니다.
public class Circle : IShape
{
    public double Radius { get; set; }
    
    // 인터페이스에 정의된 모든 메서드를 반드시, 빠짐없이 구현해야 합니다.
    public void Draw() => Console.WriteLine("원을 그립니다.");
    public double GetArea() => Math.PI * Radius * Radius;
}

```

클래스가 여러 인터페이스를 동시에 구현(다중 상속)할 수 있다는 점은 추상 클래스(단일 상속만 가능)와 구분되는 가장 강력한 특징입니다. 이를 통해 객체에 여러 가지 역할을 동시에 부여할 수 있습니다.

### 명시적 인터페이스 구현 (Explicit Implementation)

클래스가 두 개 이상의 인터페이스를 구현할 때, 우연히 메서드 이름이 겹치는 경우가 발생할 수 있습니다. 혹은 인터페이스의 메서드를 클래스의 `public` API로 노출하고 싶지 않을 때 **명시적 구현**을 사용합니다.

```csharp
public interface ILogger 
{ 
    void Log(string message); 
}

public class DualLogger : ILogger
{
    // 1. 일반적인 public 메서드 (클래스 고유의 기능)
    public void LogToConsole(string message) => Console.WriteLine("[PUBLIC] " + message);
    
    // 2. 명시적 구현: 메서드 이름 앞에 인터페이스 이름을 붙입니다. 접근 제한자(public)는 쓸 수 없습니다.
    void ILogger.Log(string message) => Console.WriteLine("[INTERFACE] " + message);
}

var logger = new DualLogger();
logger.LogToConsole("Hello");  // 가능
// logger.Log("Hello");        // 불가능: 클래스 인스턴스로는 접근할 수 없게 숨겨집니다.

// 반드시 인터페이스 타입으로 형변환(캐스팅)해야만 숨겨진 메서드를 호출할 수 있습니다.
((ILogger)logger).Log("Hello"); 

```

## 현대적 C#의 강력한 인터페이스 기능들

C#이 버전업을 거듭하면서 인터페이스는 단순한 껍데기를 넘어 놀라운 기능들을 가지게 되었습니다.

### 기본 구현 메서드 (Default Interface Methods, C# 8.0+)

과거에는 수많은 클래스가 이미 사용 중인 인터페이스에 새로운 메서드를 하나 추가하면, 이를 구현한 모든 클래스에 컴파일 에러가 터지는 대참사(파괴적 변경)가 발생했습니다. 이를 막기 위해 인터페이스 내부에 메서드의 **기본 구현(Default Implementation)**을 작성할 수 있게 되었습니다.

```csharp
public interface IDataProcessor
{
    // 기존에 있던 필수 구현 메서드
    string Process(string data);
    
    // 나중에 추가된 메서드: 기본 로직을 제공하므로 기존 클래스들이 에러를 뿜지 않습니다.
    public virtual string Preprocess(string data) => data.Trim().ToUpper();
}

```

### 정적 추상 멤버 (Static Abstract Members, C# 11.0+)

제네릭 타입에서 수학 연산자(`+`, `-` 등)나 정적 팩토리 메서드를 강제할 수 있게 해주는 혁신적인 기능입니다.

```csharp
// "이 인터페이스를 구현하는 타입은 반드시 덧셈 연산자와 Zero 속성을 가져야 한다"는 계약입니다.
public interface IAddable<T> where T : IAddable<T>
{
    static abstract T Zero { get; }
    static abstract T operator +(T left, T right);
}

// 이 제네릭 메서드는 int, double뿐만 아니라 IAddable을 구현한 어떤 커스텀 타입이든 다 합칠 수 있습니다.
public static T Sum<T>(IEnumerable<T> values) where T : IAddable<T>
{
    var result = T.Zero;
    foreach (var value in values)
        result += value;
    return result;
}

```

### 공변성(out)과 반공변성(in)

제네릭 인터페이스를 설계할 때 상속 관계에 있는 타입 간의 유연한 캐스팅을 지원하는 기능입니다.

```csharp
// out 키워드(공변성): 이 제네릭 타입은 출력(Return) 용도로만 쓰입니다.
public interface IProducer<out T> { T Produce(); }

// in 키워드(반공변성): 이 제네릭 타입은 입력(Parameter) 용도로만 쓰입니다.
public interface IConsumer<in T> { void Consume(T item); }

// 활용
IProducer<string> stringProducer = GetStringProducer();
IProducer<object> objectProducer = stringProducer; // string은 object의 자식이므로 자연스럽게 대입 가능 (공변)

IConsumer<object> objectConsumer = GetObjectConsumer();
IConsumer<string> stringConsumer = objectConsumer; // object를 처리할 수 있는 소비자는 당연히 string도 처리 가능 (반공변)

```

## 인터페이스를 활용한 3대 설계 패턴

인터페이스는 클래스 간의 결합도를 끊어내는 가위 역할을 합니다. 이를 활용한 대표적인 실무 설계 패턴 세 가지를 소개합니다.

### 1. 전략 패턴 (Strategy Pattern)

알고리즘의 뼈대는 놔둔 채, 런타임에 내부 부품(전략)만 갈아 끼워 다양한 동작을 수행하게 합니다. 상속의 부작용을 피하는 가장 이상적인 패턴입니다.

```csharp
public interface ISortStrategy<T>
{
    void Sort(List<T> items);
}

public class QuickSort<T> : ISortStrategy<T> { /* 퀵 정렬 구현 */ }
public class MergeSort<T> : ISortStrategy<T> { /* 병합 정렬 구현 */ }

// 정렬을 수행하는 컨텍스트 클래스는 구체적인 정렬 알고리즘을 모릅니다.
// 오직 ISortStrategy라는 인터페이스에만 의존합니다. (의존성 역전)
public class Sorter<T>
{
    private ISortStrategy<T> _strategy;
    
    public void SetStrategy(ISortStrategy<T> strategy) => _strategy = strategy;
    public void ExecuteSort(List<T> items) => _strategy.Sort(items);
}

```

### 2. 어댑터 패턴 (Adapter Pattern)

110V 플러그를 220V 콘센트에 꽂게 해주는 돼지코처럼, 이미 만들어진 레거시 클래스와 현대적인 인터페이스 사이의 호환성을 맞춰줍니다.

```csharp
// 수정할 수 없는 구형(레거시) 클래스
public class LegacyLogger
{
    public void WriteToConsole(string msg) => Console.WriteLine(msg);
}

// 우리가 새로 만든 세련된 인터페이스
public interface IModernLogger { void Log(string message); }

// 어댑터 클래스: 현대적인 인터페이스를 구현하되, 실제 일은 레거시 객체에게 몰래 넘깁니다.
public class LegacyLoggerAdapter : IModernLogger
{
    private readonly LegacyLogger _legacyLogger;
    public LegacyLoggerAdapter(LegacyLogger logger) => _legacyLogger = logger;
    
    public void Log(string message)
    {
        _legacyLogger.WriteToConsole($"[어댑터 통과] {message}");
    }
}

```

### 3. 데코레이터 패턴 (Decorator Pattern)

인터페이스를 활용하여 기존 클래스의 코드를 단 한 줄도 수정하지 않고, 그 객체를 포장지처럼 감싸 새로운 기능을 동적으로 추가합니다.

```csharp
public interface IDataService { string GetData(string id); }

// 핵심 로직이 담긴 원본 클래스
public class DatabaseService : IDataService 
{ 
    public string GetData(string id) => "DB 데이터"; 
}

// 부가 기능(캐싱)을 덧붙이기 위한 데코레이터 클래스
public class CachingDecorator : IDataService
{
    private readonly IDataService _innerService;
    private readonly Dictionary<string, string> _cache = new();
    
    public CachingDecorator(IDataService innerService) => _innerService = innerService;
    
    public string GetData(string id)
    {
        // 1. 캐시에 데이터가 있으면 즉시 반환 (기능 추가)
        if (_cache.TryGetValue(id, out var cached)) return cached;
            
        // 2. 캐시에 없으면 원본 객체에게 일을 시킴
        var data = _innerService.GetData(id);
        
        // 3. 가져온 데이터를 캐시에 저장 후 반환 (기능 추가)
        _cache[id] = data;
        return data;
    }
}

```

## 인터페이스 vs 추상 클래스: 실무 선택 기준

많은 개발자가 이 둘의 차이를 헷갈려합니다. 다음 가이드라인을 참고하여 선택하세요.

| 기준 | 인터페이스 (Interface) | 추상 클래스 (Abstract Class) |
| --- | --- | --- |
| **상속 횟수** | 여러 개를 동시에 구현할 수 있습니다. (다중 구현) | 오직 단 하나의 부모만 상속받을 수 있습니다. |
| **상태 보관** | 필드(변수)를 절대 가질 수 없습니다. 오직 뼈대만 제공합니다. | 필드와 상태를 직접 가질 수 있습니다. |
| **주요 목적** | 서로 다른 객체들이 수행해야 할 **'역할과 행동의 계약'**을 정의할 때 | 자식 클래스들이 공유해야 할 **'공통된 상태와 기본 로직'**을 묶어둘 때 |
| **적용 사례** | `IDisposable` (버릴 수 있음), `IComparable` (비교할 수 있음) | `Animal` (동물), `Stream` (입출력 스트림) |

## 성능 주의사항: 박싱(Boxing)

가장 가벼운 값 타입인 구조체(`struct`)가 인터페이스를 구현하고, 그 구조체를 인터페이스 타입의 변수에 담게 되면 치명적인 성능 저하가 발생합니다.

```csharp
struct Point : IShape { public void Draw() { } }

// ❌ 박싱 발생: 스택에 있던 Point 구조체가 힙 메모리로 복사되는 엄청난 낭비가 발생합니다.
IShape shape = new Point(); 

// ✅ 해결책: 제네릭 제약 조건을 사용하면 박싱 없이 구조체를 인터페이스의 형태로 넘길 수 있습니다.
void DrawShape<T>(T shape) where T : IShape
{
    shape.Draw(); 
}

```

## 마무리

"구현(Implementation)이 아닌 인터페이스(Interface)에 프로그래밍하라." 객체 지향 설계의 가장 유명한 격언입니다.

인터페이스는 구체적인 클래스에 직접 의존하여 코드가 단단하게 굳어지는 것을 막아주고, 유닛 테스트를 위한 모의 객체(Mock) 생성을 가능하게 하는 현대 아키텍처의 산소 같은 존재입니다. 클래스를 설계할 때 "어떻게 구현할까?"보다 "외부와 어떤 계약을 맺어야 할까?"를 먼저 고민하는 습관을 들인다면, 여러분의 C# 코드는 훨씬 더 유연하고 유지보수하기 쉬운 모습으로 진화할 것입니다.
