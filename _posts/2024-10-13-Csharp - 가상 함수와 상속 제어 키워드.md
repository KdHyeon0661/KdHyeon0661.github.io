# C# 가상 함수와 상속 제어 키워드 완벽 가이드

객체 지향 프로그래밍(OOP)에서 상속은 코드의 재사용성을 높여주지만, 동시에 객체 간의 강한 결합을 유발하기도 합니다. 부모 클래스가 물려준 기능을 자식 클래스가 어떻게 받아들이고, 변형하고, 혹은 거부할 것인지를 정밀하게 통제하는 것은 견고한 소프트웨어 아키텍처를 설계하는 데 있어 필수적인 기술입니다.

C#은 이러한 상속의 흐름과 다형성(Polymorphism)을 제어하기 위해 `virtual`, `override`, `new`, `abstract`, `sealed`라는 강력한 키워드들을 제공합니다. 이 글에서는 각 키워드의 문법적 의미를 넘어, 메모리 내부에서 일어나는 동작 원리(vtable)와 실무적인 설계 원칙까지 깊이 있게 살펴보겠습니다.

## 핵심 요약: 언제 무엇을 사용할까

본격적인 설명에 앞서, 상황에 따라 어떤 키워드 조합을 선택해야 하는지 요약한 가이드라인입니다.

| 목표 | 추천 키워드 구성 | 주요 특징 및 효과 |
| --- | --- | --- |
| 런타임 다형성 확장 지점 제공 | `virtual`(부모) + `override`(자식) | 동적 디스패치를 통해 실제 객체 타입의 구현을 호출합니다. |
| 더 이상 재정의 방지 | `sealed override` | 특정 단계에서 구현을 고정하여 버전 안정성과 JIT 최적화 성능을 향상시킵니다. |
| 단순히 이름만 가리기 | `new` | 다형성이 적용되지 않으며 정적 바인딩을 수행합니다. 혼란을 유발할 수 있어 사용을 자제합니다. |
| 강제 계약과 공통 로직의 혼합 | `abstract` (메서드/클래스) | 인스턴스화가 불가능해지며, 파생 클래스에서 반드시 내용을 구현하도록 강제합니다. |
| 상속 자체를 원천 차단 | `sealed class` | API의 안정성과 보안을 확보하고 코드 추론성을 높입니다. |

## virtual과 override: 런타임 다형성의 핵심

`virtual` 키워드는 부모 클래스에서 "이 메서드는 자식 클래스에서 입맛에 맞게 바꿀 수 있다"고 허락하는 표시입니다. 자식 클래스는 `override` 키워드를 사용하여 이 메서드를 재정의합니다.

```csharp
class Animal
{
    // 자식 클래스에서 재정의를 허용합니다.
    public virtual void Speak()
    {
        Console.WriteLine("동물이 소리를 냅니다");
    }
}

class Dog : Animal
{
    // 부모의 가상 메서드를 덮어씁니다.
    public override void Speak()
    {
        Console.WriteLine("멍멍");
    }
}

Animal a = new Dog();
a.Speak(); // 출력: 멍멍

```

위 코드에서 변수 `a`의 컴파일 타임 타입은 `Animal`이지만, 런타임에 메모리에 할당된 실제 객체는 `Dog`입니다. `virtual`과 `override`의 조합은 런타임에 실제 객체의 타입을 추적하여 알맞은 메서드를 찾아내는 동적 디스패치(Dynamic Dispatch)를 활성화합니다.

메서드뿐만 아니라 속성(Property), 인덱서, 이벤트 등 대부분의 클래스 멤버를 가상화할 수 있습니다. 부모의 기존 로직을 완전히 버리지 않고 살을 붙이고 싶다면, 자식 클래스 내부에서 `base` 키워드를 통해 부모의 원래 메서드를 호출할 수도 있습니다.

### 가상 메서드의 내부 동작 원리: vtable (가상 함수 테이블)

컴파일 타임의 변수 타입과 무관하게 런타임에 올바른 메서드를 찾아가는 마법의 이면에는 가상 함수 테이블(vtable)이라는 치밀한 메모리 메커니즘이 존재합니다.

```text
[vtable의 내부 동작 구조]

1. 컴파일 단계
Animal 클래스의 vtable 생성 -> [ 슬롯 0: Animal.Speak() 주소 ]
Dog 클래스의 vtable 생성    -> [ 슬롯 0: Dog.Speak() 주소 (덮어씌워짐) ]

2. 객체 생성 (Animal a = new Dog();)
힙(Heap) 메모리에 Dog 객체가 생성될 때, 객체의 헤더에는 
자신이 속한 클래스의 vtable을 가리키는 숨겨진 포인터(vptr)가 삽입됩니다.

3. 메서드 호출 (a.Speak();)
- 변수 a가 가리키는 실제 메모리(Dog 객체)를 찾아갑니다.
- 객체 내부의 vptr을 타고 Dog 클래스의 vtable로 이동합니다.
- vtable의 0번 슬롯에 기록된 함수 포인터(Dog.Speak)를 읽어 실행합니다.

```

자식 클래스에서 `override`를 선언하면 컴파일러는 부모가 물려준 vtable의 슬롯 주소를 자식의 새로운 메서드 주소로 교체합니다. 이 간접 참조 과정 덕분에 완벽한 다형성이 구현됩니다.

### 생성자에서 가상 메서드 호출의 치명적 위험성

C# 아키텍처를 설계할 때 가장 빈번하게 발생하는 실수 중 하나입니다. 부모 클래스의 생성자 내부에서는 절대 가상 메서드를 호출해서는 안 됩니다.

객체가 생성될 때는 부모 클래스의 생성자가 먼저 실행되고 자식 클래스의 생성자가 나중에 실행됩니다. 만약 부모 생성자에서 가상 메서드를 호출하게 되면, 다형성에 의해 자식 클래스에서 재정의한 메서드가 실행됩니다. 문제는 이 시점에 자식 클래스의 생성자는 아직 단 한 줄도 실행되지 않아 내부 필드들이 초기화되지 않은 상태라는 점입니다.

결국 초기화되지 않은 불안정한 상태의 데이터에 접근하게 되어 디버깅하기 매우 까다로운 `NullReferenceException`이나 논리적 오류를 발생시킵니다. 따라서 초기화 로직이 복잡하다면 생성자가 아닌 별도의 팩토리 메서드(Factory Method) 패턴을 활용하는 것이 안전합니다.

## new 키워드: 이름 숨김의 함정과 가상 체인 단절

상속 관계에서 메서드를 정의할 때 `override` 대신 `new` 키워드를 사용할 수 있습니다. 그러나 이는 객체 지향의 다형성을 제공하는 것이 아니라, 부모 클래스의 메서드 이름을 단순히 숨기는(Hiding) 정적 바인딩을 수행합니다.

```csharp
class Foo
{
    public virtual void Print() => Console.WriteLine("Foo");
}

class Bar : Foo
{
    // 부모의 가상 메서드를 덮어쓰지 않고, 전혀 다른 새로운 메서드를 만듭니다.
    public new void Print() => Console.WriteLine("Bar");
}

Foo f = new Bar();
f.Print(); // 출력: Foo (정적 바인딩 발생)

```

위 예제에서 변수 `f`의 타입이 `Foo`이기 때문에, 컴파일러는 `Bar`의 객체임에도 불구하고 `Foo`의 메서드를 강제로 호출해 버립니다.

`new` 키워드의 가장 치명적인 특징은 가상 메서드 체인(Virtual Chain)을 영구적으로 끊어버린다는 것입니다. 중간에 `new`로 선언된 메서드는 부모의 vtable 슬롯을 교체하지 않고 별도의 독립적인 슬롯을 파버리기 때문에, 이후의 파생 클래스에서 아무리 `override`를 시도해도 최상위 부모 타입으로 참조할 때는 접근할 수 없게 됩니다.

결과적으로 `new` 키워드는 코드를 읽는 개발자에게 엄청난 혼란을 초래하므로, 외부 레거시 라이브러리의 메서드 이름과 우연히 충돌하여 어쩔 수 없이 이름을 숨겨야 하는 특수한 상황을 제외하고는 사용을 철저히 지양해야 합니다.

## abstract: 계약의 강제와 유연성의 균형

클래스나 메서드에 `abstract` 키워드를 붙이면, 해당 클래스는 미완성 상태가 되어 직접 인스턴스화(`new`)할 수 없게 됩니다. 대신 이를 상속받는 자식 클래스에게 특정한 메서드를 반드시 구현하라는 강력한 계약(Contract)을 강제합니다.

```csharp
abstract class Shape
{
    // 구현부가 없습니다. 파생 클래스는 이 메서드를 무조건 만들어야 합니다.
    public abstract double Area(); 
    
    // 추상 클래스도 일반 가상 메서드를 포함하여 공통 로직을 제공할 수 있습니다.
    public virtual string Name => "도형"; 
}

class Circle : Shape
{
    public double Radius { get; }
    public Circle(double radius) => Radius = radius;
    
    // 부모의 추상 메서드를 override로 완성합니다.
    public override double Area() => Math.PI * Radius * Radius;
    public override string Name => "원";
}

```

추상 클래스는 인터페이스(Interface)와 유사하게 행동하지만, 필드 상태를 가질 수 있고 기본 동작을 미리 구현해 둘 수 있다는 점에서 알고리즘의 뼈대를 제공하는 템플릿 메서드(Template Method) 패턴을 설계할 때 가장 이상적입니다.

## sealed: 확장의 경계 설정과 성능 최적화

`sealed` 키워드는 `abstract`와 정확히 반대되는 개념입니다. 상속의 꼬리를 끊고 더 이상의 확장을 원천적으로 봉쇄합니다.

클래스 전체를 봉인(`sealed class`)하여 상속 자체를 막을 수도 있고, 특정 메서드만 봉인(`sealed override`)하여 해당 동작이 더 이상 하위 클래스에서 변질되지 않도록 고정할 수도 있습니다.

```csharp
class Animal
{
    public virtual void Speak() { }
}

class Dog : Animal
{
    // Dog 클래스부터는 Speak 메서드의 동작이 영구히 고정됩니다.
    public sealed override void Speak()
    {
        Console.WriteLine("멍멍");
    }
}

class Bulldog : Dog
{
    // 컴파일 오류: 상위에서 sealed로 봉인했기 때문에 더 이상 덮어쓸 수 없습니다.
    // public override void Speak() { } 
}

```

### 성능을 극대화하는 sealed의 JIT 최적화

`sealed` 키워드는 단순히 설계의 의도를 표현하는 것을 넘어, 프로그램의 실행 속도를 끌어올리는 강력한 힌트가 됩니다.

가상 메서드 호출은 vtable을 한 번 더 거쳐야 하므로 약간의 오버헤드가 발생합니다. 하지만 클래스나 메서드에 `sealed`가 붙어 있다면, JIT(Just-In-Time) 컴파일러는 이 클래스가 더 이상 파생되지 않을 것임을 100% 확신할 수 있습니다.

호출 대상이 완벽하게 명확해지므로, JIT 컴파일러는 vtable을 거치는 동적 디스패치 과정을 생략하고 직접 호출(Direct Call)로 변환하거나 아예 메서드 본문을 호출부에 인라인(Inline)시켜 버리는 극단적인 최적화를 수행합니다. 실제로 핫스팟(반복이 매우 잦은 핵심 로직)에서 `sealed` 클래스를 사용하면 비봉인 클래스 대비 비약적인 성능 향상을 얻을 수 있습니다.

## 리스코프 치환 원칙 (Liskov Substitution Principle)

오버라이딩을 활용할 때 가장 중요하게 지켜야 할 객체 지향 설계 원칙입니다. 자식 클래스는 부모 클래스의 계약을 절대 깨뜨려서는 안 됩니다.

즉, 프로그램 내에서 부모 클래스의 객체를 사용하는 모든 곳에 자식 클래스의 객체를 대신 집어넣더라도, 프로그램은 본래 기대했던 동작을 한 치의 오차 없이 수행해야 합니다. 만약 자식 클래스에서 `override`를 하면서 예기치 않은 예외를 던지거나 부모가 의도하지 않은 방향으로 상태를 변경한다면, 다형성의 신뢰가 무너지고 겉잡을 수 없는 버그가 양산됩니다.

## 실전 설계 패턴: 템플릿 메서드 패턴

지금까지 살펴본 키워드들을 완벽하게 조화시켜 상위 클래스에서 전체적인 흐름을 통제하고, 하위 클래스에서 세부 구현을 채워 넣으면서도, 안정성을 유지하는 실전 패턴입니다.

```csharp
abstract class DataProcessor
{
    // 뼈대가 되는 전체 알고리즘의 흐름입니다.
    public void Process(string input)
    {
        var cleaned = Clean(input);
        var parsed = Parse(cleaned);
        Output(parsed);
    }

    // 가상 메서드: 기본 로직을 제공하되, 자식이 원하면 바꿀 수 있습니다.
    protected virtual string Clean(string s) => s.Trim();
    
    // 추상 메서드: 자식 클래스가 무조건 스스로 구현해야 합니다.
    protected abstract object Parse(string s);
    
    protected virtual void Output(object result)
    {
        Console.WriteLine($"처리 결과: {result}");
    }
}

class CsvProcessor : DataProcessor
{
    // 데이터 정제 로직을 재정의하면서 sealed를 걸어 이후 파생 클래스가 이 규칙을 깨지 못하게 고정합니다.
    protected sealed override string Clean(string s)
    {
        return s.Replace("\r\n", "\n").Trim();
    }

    // 추상 메서드의 규약에 따라 구체적인 CSV 파싱 로직을 완성합니다.
    protected override object Parse(string s)
    {
        return s.Split(',').Select(x => x.Trim()).ToList();
    }
}

```

## 마무리

C#의 상속 제어 키워드들은 클래스 간의 관계를 설계하는 아키텍트의 섬세한 도구들입니다. `virtual`과 `override`는 프로그램이 부드럽게 진화할 수 있는 유연성을 제공하고, `abstract`는 반드시 지켜야 할 굳건한 뼈대를 세우며, `sealed`는 더 이상의 혼란을 막고 성능을 끌어올리는 강력한 쐐기가 됩니다.

실무에서 클래스를 설계할 때는 명시적인 이유 없이 무분별하게 모든 것을 가상화하지 마십시오. 확장이 명확히 예상되는 지점만 열어두고, 안정성과 성능이 필요한 부분은 적극적으로 봉인(`sealed`)하는 것이 유지보수하기 좋고 단단한 소프트웨어를 만드는 지름길입니다.
