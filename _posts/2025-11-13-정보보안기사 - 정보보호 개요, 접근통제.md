# SECTION 17 정보보안 일반/관리 — 01 정보보호 개요 · 02 접근통제(Access Control)

## 학습 목표와 범위

- **시험 관점 핵심**: CIA(기밀성·무결성·가용성) 확장속성, 위험관리 지표(ALE·SLE·ARO), 보안통제 유형(관리/기술/물리, 예방/탐지/교정), **정책→지침→표준→절차** 위계, **접근통제 모델(DAC/MAC/RBAC/ABAC)**, IAM 구성요소, **감사·책임추적성** 설정, **실무형 시나리오**(OS/DB/클라우드/K8s 등)에서의 최소권한 적용과 검증.
- **실무 관점 핵심**: 데이터 분류/표지, BIA(업무영향분석)로 RTO/RPO 설정, 지표(MTTD/MTTR), **Zero Trust** 원칙, **클라우드 IAM**(정적 권한→JIT·JEA), **감사·모니터링**(SIEM·EDR·CloudTrail), **변경관리**와 **권한검토(Access Review)**.

## 정보보호 개요(Foundations)

### 보안 목표와 확장 속성

- **기본 3요소(CIA)**
  - **Confidentiality(기밀성)**: 인가된 주체만 정보 접근
  - **Integrity(무결성)**: 정보의 정확성·완전성·일관성 보장
  - **Availability(가용성)**: 필요한 시점에 접근 가능
- **확장 속성**: **Authenticity(진정성)**, **Accountability(책임추적성)**, **Non-repudiation(부인방지)**, **Privacy(프라이버시)**.

### 위험(Risk)과 정량·정성 평가

- **용어 정렬**:
  - **자산(Asset)**: 보호 대상(정보/시스템/평판)
  - **위협(Threat)**: 자산에 악영향을 줄 수 있는 잠재 사건
  - **취약점(Vulnerability)**: 위협이 악용할 수 있는 약점
  - **영향(Impact)**: 발생 시 손실 규모
- **정성 평가**: High/Medium/Low 등 척도로 우선순위화
- **정량 평가(예)**:
  - **단일손실예상액**: $$SLE = AV \times EF$$
    (AV: 자산가치, EF: 노출계수/손실비율)
  - **연간발생예상빈도**: $$ARO$$
  - **연간손실예상액**: $$ALE = SLE \times ARO$$
- **리스크 산식(간이)**:
  $$Risk = Likelihood \times Impact$$

> **미니 예시**: DB서버(AV=2억 원), 유출 시 손실 20%(EF=0.2), 연 0.5회(ARO).
> \(SLE = 40,000,000\), \(ALE = 20,000,000\) → **연 2천만 원 수준 통제 투자 타당성** 판단 근거.

### 정책·표준·지침·절차(위계)

| 계층 | 목적 | 예시 |
|---|---|---|
| **정책(Policy)** | 경영 의지/원칙 선언 | 접근통제는 최소권한 원칙을 따른다 |
| **표준(Standard)** | 최소요구/측정 가능한 기준 | 관리자 계정은 MFA 필수, 암호화 강도는 TLS 1.2+ |
| **지침(Guideline)** | 모범사례 권고 | 코드리뷰 체크리스트, 비밀관리 모범사례 |
| **절차(Procedure)** | 수행 단계/담당/입출력 정의 | 신규계정 개설 절차(SOD 검증→승인→프로비저닝) |

**정책 YAML 샘플**
```yaml
policy_name: AccessControl
purpose: "조직 정보자산 보호를 위한 접근통제 원칙 수립"
scope: ["모든 정보시스템/클라우드/온프레미스", "임직원/협력사 계정"]
principles:
  - least_privilege
  - need_to_know
  - separation_of_duties
  - default_deny
requirements:
  - mfa_required: true
  - password_standard: "NIST-aligned, min length 12"
  - privileged_access: "JIT/JEA 사용, 세션기록 의무"
reviews:
  - access_review_cycle: "분기"
  - exception_approval: "보안책임자(CISO) 승인"
logging:
  - audit_required_events: ["권한부여/회수", "정책변경", "인증 실패 누적"]
```

| 구분 | 유형 | 예 |
|---|---|---|

|  | **탐지(Detective)** | IDS/IPS, SIEM 규칙, 무결성 모니터링 |
|  | **교정(Corrective)** | 패치, 백업 복구, 비번 초기화 |
| 영역 | **관리(Administrative)** | 정책, 교육, 배경조사, 계약 |
|  | **기술(Technical)** | 암호화, ACL, WAF, EDR |
|  | **물리(Physical)** | 출입통제, CCTV, 금고 |

### 프레임워크 맵핑(요지)

- **ISO/IEC 27001:2022**: ISMS 구축·운영 요구사항, Annex A(27002:2022 통제)
- **NIST CSF 2.0**: **Govern** 추가(거버넌스) + Identify/Protect/Detect/Respond/Recover
- **NIST SP 800-53 Rev.5**: 통제 카탈로그(AC, AU, IA, IR, SC 등)

### 데이터 분류·표지(예)

| 등급 | 예시 | 핵심 통제 |
|---|---|---|
| 공개 | IR 자료, 공시 | 변조방지, 정확성 |
| 내부 | 사내 메일 | 최소권한, 접근기록 |
| 중요 | 고객PII | 암호화(전송/저장), DLP, 보관기간 |
| 매우중요 | 핵심기술 | 하드 격리, 강력MFA, HSM, 엄격한 접근검토 |

**분류 레이블링 규칙(예)**: 파일명 접두어, 문서 머리·바닥글 워터마크, S3 버킷 라벨/CMK 구분.

### BIA·연속성

- **RTO**(복구목표시간), **RPO**(복구시점목표) 설정으로 백업·복구 수준 결정
- DR 시나리오: 단일 AZ → 멀티 AZ → 멀티 리전
- **테이블 예**
| 서비스 | RTO | RPO | 대책 |
|---|---|---|---|
| 주문결제 | 1h | 5min | 이중화, DB 동기복제, 재해복구 훈련/분기 |

### 성과지표(KPI/KRI)

- **KPI**: 패치적용 리드타임, 접근검토 완료율, 취약점 잔여량
- **KRI**: 관리자 비인가 시도, 실패 인증률, 권한상승 탐지건수
- **운영 예**: “Critical 패치 7일 내 95% 이상”.

## 접근통제(Access Control)

### AAA와 IAM 구성요소

- **Authentication(인증)**: ID 확인(MFA·패스키·OIDC)
- **Authorization(인가)**: 권한 판정(정책/역할/속성)
- **Accounting(감사)**: 사용 행위 기록·책임추적성
- **IAM**: 디렉터리(AD/IdP), 프로비저닝/디프로비저닝, 승인 워크플로, 인증(SSO), 권한정책, **주기적 접근검토**.

### 원칙

- **최소권한(Least Privilege)**, **알아야 할 필요(Need-to-know)**
- **업무분리(SOD)**: 상충 권한 분리(요청·승인, 개발·운영, 발주·검수)
- **기본차단(Default Deny)** + **명시적 허용**
- **시간·맥락 기반**: JIT(Just-In-Time), JEA(Just-Enough-Access)

### 접근통제 모델과 비교

| 모델 | 핵심 | 장점 | 주의 |
|---|---|---|---|
| **DAC** | 소유자 중심 권한부여 | 유연 | 오남용·권한확산 |
| **MAC** | 라벨/등급(기밀/비밀) | 강력 | 운영비용 높음 |
| **RBAC** | 역할 기반(직무) | 관리 용이 | 역할 폭증(권한 스프롤) |
| **ABAC** | 속성 기반(주체·자원·행위·환경) | 세밀·동적 | 정책 복잡성 |
| **ReBAC** | 관계 기반(그래프) | 협업 모델 적합 | 성능/모델링 난이도 |

**ABAC 정책 수식(개념)**
$$Permit \iff Subject(a)\wedge Resource(b)\wedge Action(c)\wedge Env(e)\wedge Policy(a,b,c,e)$$

### 운영체제 수준 예제

#### (A) Linux 파일권한/ACL

```bash
# 프로젝트 디렉터리 생성 및 기본권한

sudo mkdir -p /srv/projectX
sudo chown :devteam /srv/projectX
sudo chmod 2770 /srv/projectX    # SGID로 그룹 상속, 외부차단

# ACL로 세분화(qa그룹에 read-only)

sudo setfacl -m g:qa:rx /srv/projectX
sudo getfacl /srv/projectX
```

**capabilities(권능)로 최소권한 실행**
```bash
# ping이 root 없이 동작하도록 cap_net_raw 부여(예시)

sudo setcap cap_net_raw+p /usr/bin/ping
getcap /usr/bin/ping
```

**sudo 최소권한**
```bash
# /etc/sudoers.d/deploy

%deploy ALL=(webuser) NOPASSWD: /usr/local/bin/rollout.sh
```

#### (B) Windows NTFS ACL

```powershell
# 폴더에 역할별 최소권한 부여

$path="D:\ProjectX"
icacls $path /inheritance:r
icacls $path /grant "ProjectX-Dev:(OI)(CI)M" "ProjectX-QA:(OI)(CI)R" "ProjectX-Owners:(OI)(CI)F"
# 관리자 계정 로컬 LAPS 사용 권고(개별 고유 PW), 감사정책에 권한변경 로깅

```

### 데이터베이스 접근통제

**RBAC(역할) 설계**
```sql
-- 스키마 분리
CREATE SCHEMA app AUTHORIZATION dbo;

-- 역할 생성
CREATE ROLE app_reader;
CREATE ROLE app_writer;
CREATE ROLE app_admin;

-- 권한 할당(최소)
GRANT SELECT ON SCHEMA::app TO app_reader;
GRANT INSERT, UPDATE, DELETE ON SCHEMA::app TO app_writer;
GRANT EXECUTE ON SCHEMA::app TO app_admin;

-- 사용자 매핑
EXEC sp_addrolemember 'app_reader', 'alice';
EXEC sp_addrolemember 'app_writer', 'bob';
```

**행수준 보안(Row-Level Security) 예(개념)**
```sql
-- 테넌트 ID 기반 필터 함수(예: SQL Server)
CREATE FUNCTION app.fn_tenantPredicate(@tenant_id AS INT)
RETURNS TABLE WITH SCHEMABINDING AS
RETURN SELECT 1 AS fn_result
WHERE CAST(SESSION_CONTEXT(N'tenant_id') AS INT) = @tenant_id;

CREATE SECURITY POLICY app.tenantPolicy
ADD FILTER PREDICATE app.fn_tenantPredicate(tenant_id) ON app.orders
WITH (STATE = ON);
```

### 네트워크·플랫폼 예제

**L3 ACL(요점)**
```cisco
ip access-list extended DB-ONLY
  permit tcp 10.10.10.0 0.0.0.255 host 10.20.0.10 eq 5432
  deny   ip any host 10.20.0.10
  permit ip any any
interface GigabitEthernet0/1
  ip access-group DB-ONLY in
```

**Kubernetes RBAC + NetworkPolicy**
```yaml
# Role: 특정 네임스페이스에서만 configmaps 읽기

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: {name: cm-reader, namespace: prod}
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get","list","watch"]
---
# RoleBinding: 서비스어카운트에 바인딩

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: {name: bind-cm-reader, namespace: prod}
subjects:
  - kind: ServiceAccount
    name: api-sa
roleRef:
  kind: Role
  name: cm-reader
  apiGroup: rbac.authorization.k8s.io
---
# NetworkPolicy: app->db만 5432 허용

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: {name: db-allow, namespace: prod}
spec:
  podSelector: {matchLabels: {app: db}}
  ingress:
  - from:
    - podSelector: {matchLabels: {app: api}}
    ports:
    - protocol: TCP
      port: 5432
  policyTypes: ["Ingress"]
```

**AWS IAM 최소권한(리소스 제한)**
```json

  "Version": "2012-10-17",
  "Statement": [{
```

**OPA/Rego(ABAC) 예: 업무시간·위치 기반**
```rego
package authz

default allow = false

allow {
  input.action == "read"
  input.resource.type == "report"
  input.subject.department == "finance"
  # 업무시간(09~18) & 사내망에서만
  input.env.hour >= 9
  input.env.hour < 18
  startswith(input.env.source_ip, "10.10.")
}
```

### 세션·토큰·비밀관리 베스트프랙티스

- **세션**: 짧은 유효기간, **Sliding/Refresh Token** 회전, **BFF 패턴**(브라우저 토큰 노출 최소화)
- **쿠키**: `HttpOnly`, `Secure`, `SameSite=Lax/Strict`
- **토큰**: JWT는 **짧게**, 민감정보 저장 금지, 서명키 로테이션(KMS/HSM)
- **비밀관리**: Vault/Secrets Manager 사용, 코드·이미지 내 **비밀 하드코딩 금지**.

### 감사·책임추적성

- **감사 이벤트**: 권한부여/회수, 관리자 로그인, 정책 변경, 실패 누적
- **Windows**: 고급 감사 정책(권한 사용/디렉터리 서비스 변경/계정관리)
- **Linux**: `auditd` 규칙 예시
```bash
# /etc/audit/rules.d/access.rules

-w /etc/sudoers -p wa -k sudo-change
-a always,exit -F arch=b64 -S setxattr,fsetxattr -k acl-change
```
- **클라우드**: CloudTrail/Activity Log + SIEM/UEBA로 권한오남용 탐지

### 접근검토(Access Review)와 제거(De-provisioning)

- **주기**: 분기/반기, 조직변경/퇴사 즉시
- **대상**: 고위험 권한(관리자, 데이터추출, 결제승인), **비활성 계정**
- **증빙**: 리뷰 결과·예외 승인서, JML(Joiner-Mover-Leaver) 흐름과 연결

### 흔한 오구성·공격·대응

| 유형 | 사례 | 대응 |
|---|---|---|
| 권한스프롤 | 누적 역할·그룹 | 역할정리·리퍼런스 역할, 캠페인성 접근검토 |
| IDOR | `/users/123` 직접 접근 | 객체레벨 권한검증(서버측), 리소스 필터 |
| Confused Deputy | 토큰/STS 위임 오용 | audience·resource 제한, 위임 범위 축소 |
| 과도한 와일드카드 | `s3:*`, `*:*` | 작업정의→필요 액션 최소화 |
| 장기자격증명 | Access key 미로테이션 | **단기 세션/JIT**, 키 회전 정책 |
| 기본허용 | 넓은 Network ACL | 기본차단·명시허용, 마이크로세그멘테이션 |

## 실습 시나리오 모음(합법·자가 환경)

### Linux 프로젝트 디렉터리 최소권한

```bash
sudo groupadd projx && sudo usermod -aG projx alice
sudo mkdir -p /srv/projx && sudo chgrp projx /srv/projx
sudo chmod 2770 /srv/projx
sudo setfacl -m u:auditor:rX /srv/projx
sudo getfacl /srv/projx
```
**검증**: alice는 RW, auditor는 읽기만, 타 사용자는 접근불가.

## 기출 포인트 & 실전 문답(샘플)

**Q1. “정책-표준-지침-절차 차이?”**
A. 정책=원칙, 표준=측정가능 기준, 지침=권고, 절차=수행 단계.

**Q2. “RBAC vs ABAC 선택 기준?”**
A. 직무 기반·단순 권한이면 **RBAC**, 시간/위치/태그 등 **맥락** 포함 세밀 정책이면 **ABAC**.

**Q3. “Zero Trust 접근통제 핵심?”**
A. 암묵적 신뢰 제거, **명시적 검증**, **최소권한**, **세션 단위** 평가(지속적).

**Q4. “ALE 계산식?”**
A. \(ALE = SLE \times ARO\). 예시로 연간 투자 타당성 평가에 사용.

## 부록 — 도식(ASCII) 예

**Zero Trust(요지)**
```
[User/Device] --(MFA, posture)--> [Policy Engine (ABAC/RBAC)]
         \                         /            \
          \---> [IdP/SSO] <------/              \--> [Resource: App/DB/Storage]
                  |                                      |
           [Audit & UEBA] <------------------------------/
```

**RBAC 구성**
```
[직무] -> [역할(Role)] -> [권한(Permissions)] -> [리소스(Resource)]
개인 사용자는 '역할'에만 매핑. 권한은 역할에 귀속.
```

## 마무리

- **핵심**: 보안은 **거버넌스(정책·표준)** + **위험기반 우선순위** + **접근통제 구현(RBAC/ABAC)** + **감사/지속개선**의 **사이클**이다.
- **실무 팁**: “권한은 ‘일시·맥락·업무’에 묶고, 로그로 ‘행위’를 남긴다.” 이것이 **책임추적성**과 **재발 방지**의 출발점이다.
