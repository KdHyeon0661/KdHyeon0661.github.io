# Wired LANs: Ethernet

## Ethernet Protocol

### IEEE Project 802

#### 왜 IEEE 802가 중요한가

Ethernet을 이해하려면 먼저 **IEEE 802 아키텍처**를 보는 게 좋다.

- **IEEE 802 LAN/MAN Standards Committee (LMSC)** 가 이더넷, Wi-Fi, MAN 등 **모든 LAN/MAN 계열 표준**을 관리한다.
- 802.x 번호는 각각 **서브 프로젝트 / 워킹 그룹**을 의미:
  - 802.1: 브리징, VLAN 등 공통 상위 제어
  - 802.3: 유선 Ethernet (CSMA/CD 기반 LAN, 오늘날의 모든 유선 이더넷)
  - 802.11: Wi-Fi
  - 그 외 802.15 (PAN), 802.16 (MAN) 등

IEEE 802의 기본 구조는 다음 두 계층으로 나뉜다:

1. **Logical Link Control (LLC)** – 원래 802.2에서 정의
2. **Media Access Control (MAC)** – 각 802.x 기술별로 다름 (802.3, 802.11, …)

즉, 데이터링크 계층을 **LLC + MAC** 으로 세분화해서,

- LLC는 **상위 계층(IP 등)에 공통 인터페이스**를 제공하고,
- MAC은 **매체 접근 방식(CSMA/CD, CSMA/CA, 토큰 등)** 을 담당한다.

Ethernet(802.3)의 경우:

- **LLC**: 사실상 오늘날 IP 위주 환경에서는 크게 드러나지 않지만,
  여전히 802 아키텍처에서는 “프로토콜 식별” 등의 역할로 존재.
- **MAC**:
  - 프레임 포맷(목적지 MAC, 출발지 MAC, Type/Length, FCS 등),
  - CSMA/CD(전통 이더넷) 또는 **스위치 기반 full-duplex** 동작(현대 이더넷),
  - 주소 학습, 필터링, 충돌 관리 등을 정의한다.

#### vs Ethernet II (DIX)

역사적으로 “Ethernet” 프레임에는 두 계열이 있었다:

1. **Ethernet II (DIX 프레임)** – DEC, Intel, Xerox가 만든 초기 사양
2. **IEEE 802.3 프레임** – 길이/Type 필드 정의가 조금 다름

둘의 차이(요약):

- **Ethernet II**:
  - MAC 헤더: Dest(6) + Src(6) + **Type(2)**
  - Type: 상위 프로토콜(IPv4=0x0800, ARP=0x0806 등)을 직접 나타냄
- **802.3**:
  - MAC 헤더: Dest(6) + Src(6) + **Length(2)**
  - 상위 프로토콜 식별은 LLC / SNAP 헤더를 이용

실제 인터넷 환경에서는 **Ethernet II 프레임 포맷이 사실상 표준**이 되었고,
오늘날 대부분의 NIC/스위치/라우터는 Type 기반 Ethernet II 프레임을 사용한다.

그러나 IEEE 802.3 표준 문서는 여전히

- MAC 서브레이어(프레임 포맷, 주소, CSMA/CD/Full-duplex),
- 다양한 PHY (10BASE-T, 100BASE-TX, 1000BASE-T, 10GBASE-R, …)

를 모두 포괄하는 **공식 이더넷 표준**이다.

### Ethernet Evolution

#### 타임라인 개요

Ethernet은 대략 다음과 같은 **세대별 진화**를 거쳤다:

1. **초기 실험 이더넷** (Xerox PARC, 1970s)
   - 2.94 Mbps, 굵은 동축(coax) 버스, CSMA/CD.

2. **10 Mbps “Classic Ethernet” (10BASE5/10BASE2/10BASE-T)**
   - 10BASE5: 굵은 노란 coax, 최대 500 m/segment, 4 리피터, 총 2.5 km.
   - 10BASE2: 더 얇은 coax, 세그먼트 185 m 정도.
   - 10BASE-T: UTP 기반 별형(Star) 구조, 허브/스위치 중심.

3. **Fast Ethernet (100 Mbps, 802.3u)**
   - 100BASE-TX (Cat 5 UTP), 100BASE-FX (멀티모드 파이버),
   - 자동 협상(auto-negotiation)으로 10/100Mbps 간 호환.

4. **Gigabit Ethernet (1 Gbps, 802.3z/ab)**
   - 1000BASE-SX/LX(파이버), 1000BASE-T(UTP Cat 5e 이상).

5. **10G, 40G, 100G, 400G, 800G Ethernet**
   - 광섬유/트윈액시얼/백플레인 등 다양한 매체에서 동작,
   - 데이터센터, 백본, 고성능 컴퓨팅에 사용.

핵심 변화:

- **공유 버스 + CSMA/CD** → **스위치 기반 point-to-point + full-duplex**
- 속도: **10 Mbps → 수백 Gbps**
- 매체: thick coax → thin coax → UTP → 파이버 → 고급 트윈액시얼, 단일 페어(SPE)까지 확장.

#### “버스 이더넷”에서 “스위치 이더넷”으로

초기 10 Mbps 이더넷은 **Coax 버스** 구조였다:

- 모든 노드가 **하나의 전기적 매체**를 공유,
- 동시에 전송하면 **충돌(collision)** 발생 → CSMA/CD로 해결.

이후 **10BASE-T** 와 허브/스위치가 등장하면서 구조가 크게 바뀐다:

- 물리 토폴로지는 **Star** (중앙에 허브 또는 스위치).
- 허브: 내부적으로 여전히 “공유 버스” 처럼 동작 → CSMA/CD 유지.
- 스위치: 포트마다 **독립적인 충돌 도메인**과 **full-duplex 링크** 제공 → CSMA/CD 필요 없음.

오늘날 대부분의 유선 LAN은:

- **스위치 기반 full-duplex 이더넷** (collision domain 자체가 사라짐),
- 각각의 링크는 **점대점(point-to-point)**,
- 링크 속도는 1G/10G/25G/40G/100G 이상.

하지만 **프레임 포맷, 주소 체계, 기본 동작 모델**은
여전히 10 Mbps 시절 802.3 표준의 철학을 이어받고 있다.

## Standard Ethernet (10 Mbps, IEEE 802.3)

여기서 말하는 “Standard Ethernet”은 보통

- **10 Mbps, baseband, CSMA/CD, 802.3 MAC**,
- 10BASE5/10BASE2/10BASE-T 시대의 이더넷

을 뜻한다.

### Characteristics (특징)

#### 기본 파라미터

전통적인 10 Mbps Ethernet의 대표적인 특성:

- **데이터율**: 10 Mbps
- **매체**:
  - 10BASE5: thick coax, 500 m/segment, 4 리피터, 총 2.5 km, 최대 약 100 station/segment, 총 1024 station.
  - 10BASE2: thin coax, 185 m/segment 정도.
  - 10BASE-T: UTP (Cat 3 이상) 100 m/segment, star topology.
- **전송 모드**: half-duplex (CSMA/CD)
  (스위치 + full-duplex 환경에서는 충돌 도메인 제거)
- **인코딩**: 10 Mbps 계열은 대체로 **Manchester 인코딩** 사용.
- **최소 프레임 길이**: 64 bytes (preamble+SFD 제외)
- **최대 프레임 길이**: 1518 bytes (payload 1500 + 헤더/CRC)
  (VLAN 태그 등 추가 시 1522 bytes 등으로 확장되는 경우 있음)

프레임 포맷(802.3 / Ethernet II 기반):

| 필드 | 길이 (bytes) | 설명 |
|------|--------------|------|
| Preamble | 7 | 1010… 패턴, 클록 동기용 |
| SFD (Start Frame Delimiter) | 1 | 마지막 두 비트 `11` 로 프레임 시작 표시 |
| Destination MAC | 6 | 목적지 MAC 주소 |
| Source MAC | 6 | 출발지 MAC 주소 |
| Type/Length | 2 | 상위 프로토콜 종류(Type) 또는 페이로드 길이(Length) |
| Payload | 46–1500 | 상위 계층 데이터(패딩 포함) |
| FCS (CRC-32) | 4 | 오류 검출용 |



#### 최소 프레임 길이의 의미

CSMA/CD에서 **충돌을 감지**하려면,
송신 노드가 프레임을 다 보내기 전에 **충돌 신호가 되돌아와야 한다**.

- 최대 링크 길이 = $$L_{\max}$$ (왕복 전파 지연 = $$2 T_p$$)
- 전송 속도 = $$R$$ (bit/s)
- 최소 프레임 길이 = $$L_{\min}$$ (bits)

충돌 감지 조건:

$$
\frac{L_{\min}}{R} \ge 2 T_p
$$

즉, 최소 프레임은 **왕복 전파 시간보다 길어야** 한다.

10 Mbps Ethernet에서:

- 최대 케이블 길이/리피터 등을 고려해
  $$2T_p \approx 51.2\ \mu s$$ 수준이 되도록 설계하고,
- 이 시간 동안 전송하는 비트 수:

  $$
  L_{\min} = R \cdot 2T_p \approx 10^7 \cdot 51.2 \times 10^{-6} = 512\ \text{bits}
  $$

- 512 bits = 64 bytes → **최소 프레임 크기**로 채택.

그래서 payload가 너무 짧으면 **패딩(padding)** 을 넣어
프레임 전체 길이가 64 bytes 이상이 되도록 한다.

### Addressing (MAC 주소 체계)

#### MAC 주소의 구조

Ethernet은 **48-bit MAC 주소**(EUI-48)를 사용한다.

- 6 bytes = 48 bits
- 일반적인 표기:

  ```text
  00:1A:2B:3C:4D:5E
  02-42-AC-11-00-02
  ```

구조:

- 상위 24 bit (첫 3 bytes): **OUI(Organizationally Unique Identifier)**
  - IEEE Registration Authority가 제조사/조직 단위로 할당.
- 하위 24 bit (마지막 3 bytes): 제조사가 내부적으로 사용하는 **확장 식별자**
  - 해당 조직 내에서 **유일한 NIC/장비**를 구분.

IEEE RA 문서에서는 48-bit 주소를 **EUI-48** 이라고 부르고,
64-bit 주소(EUI-64)도 정의한다.

#### Unicast / Multicast / Broadcast

첫 바이트의 **하위 비트들**로 주소 타입을 구분한다.

- 가장 덜 중요한 비트(LSB, I/G bit):
  - 0 → **Unicast (개별 노드)**
  - 1 → **Multicast (그룹)**
- 그 옆 비트(U/L bit):
  - 0 → **Globally unique (IEEE 할당 OUI 기반)**
  - 1 → **Locally administered (조직 내 임의 할당)**

특수 주소:

- **Broadcast**: `FF:FF:FF:FF:FF:FF`
  - 해당 LAN의 모든 노드가 수신.

예시:

- `01:00:5E:xx:xx:xx` 계열: IPv4 멀티캐스트 MAC 주소 범위.

#### MAC 주소 파싱 예제 코드

간단한 Python 코드로 MAC 주소를 분석해 보자.

```python
def parse_mac(mac_str):
    # "00:1A:2B:3C:4D:5E" 형태를 정수로 변환
    parts = mac_str.replace("-", ":").split(":")
    if len(parts) != 6:
        raise ValueError("잘못된 MAC 형식")
    value = 0
    for p in parts:
        value = (value << 8) | int(p, 16)

    # 첫 바이트
    first_octet = (value >> 40) & 0xFF
    ig_bit = first_octet & 0x01  # Individual/Group
    ul_bit = (first_octet >> 1) & 0x01  # Universal/Local

    addr_type = "unicast" if ig_bit == 0 else "multicast"
    scope = "global" if ul_bit == 0 else "local"

    oui = (value >> 24) & 0xFFFFFF
    nic = value & 0xFFFFFF

    return {
        "value": value,
        "addr_type": addr_type,
        "scope": scope,
        "OUI": f"{oui:06X}",
        "NIC": f"{nic:06X}",
    }

if __name__ == "__main__":
    mac = "01:23:45:67:89:AB"
    info = parse_mac(mac)
    print(info)
```

이 코드를 실행해 보면:

- `addr_type` 이 `multicast` 로 나오는지,
- `scope` 가 `global` 또는 `local` 인지,
- OUI와 NIC 부분이 어떻게 나뉘는지

직접 확인할 수 있다.

### Access Method (CSMA/CD in Ethernet)

Standard Ethernet의 MAC 접근 방식은 **CSMA/CD** 이다:

- Carrier Sense: 매체가 idle 인지 먼저 듣는다.
- Multiple Access: 여러 노드가 같은 매체 공유.
- Collision Detection: 송신 중에도 신호를 감시하다가,
  이상 신호(충돌)를 감지하면 즉시 중단하고 **jam** 을 보낸다.

#### 알고리즘 요약

1. 송신할 프레임이 있으면:
   - 매체를 듣는다.
   - 채널이 idle 이면 곧바로 전송 (1-persistent CSMA).
2. 전송 중:
   - 자신의 송신 신호와 매체의 실제 신호를 비교.
   - 차이가 나면 **collision** 으로 판단.
3. 충돌 감지 시:
   - 최소 jam(충돌 알림) 신호 전송.
   - **Binary Exponential Backoff**:
     - 충돌 횟수를 $$k$$ 라고 할 때,
     - $$0 \le r < 2^k$$ 중 랜덤 $$r$$ 을 선택,
     - $$r$$ 슬롯 시간(51.2 µs) 만큼 기다린 뒤 1번으로 돌아감.
     - $$k$$ 는 최대 10~16 정도까지 증가(표준에 정의).

#### 작은 시나리오

버스 이더넷에 A와 B 두 호스트가 있다.

- 둘 다 거의 동시에 1500-byte 프레임을 전송하고자 함.
- 시간 축:

```text
t0:  A, B 둘 다 carrier sense → idle 인 것으로 판단
t0+: A, B 거의 동시에 전송 시작
t0+Δ: A가 B의 신호를 감지, B도 A의 신호를 감지 → collision
t0+Δ→: A, B 둘 다 jam 신호 전송 후 중단
이후: A, B 각각 랜덤 backoff 후 재전송 시도
```

이 상황에서 frame 길이가 충분히 길지 않으면
A나 B가 **프레임 송신을 끝내고 나서야** 충돌 신호를 보게 될 수 있다.
그래서 앞에서 설명한 것처럼 **최소 프레임 크기 64 bytes** 가 필요하다.

#### CSMA/CD 모의 코드

아주 단순화된 시뮬레이터:

```python
import random

def csma_cd_sim(num_hosts=4, max_rounds=1000, collision_prob=0.2):
    """
    collision_prob: 단순화를 위해 '같은 슬롯에서 전송하려는 호스트가 있을 확률'처럼 사용
    실제 CSMA/CD와 다르지만, backoff 동작 관찰용
    """
    backoff = [0] * num_hosts
    collisions = 0
    success = 0

    for _ in range(max_rounds):
        ready = []
        for i in range(num_hosts):
            if backoff[i] > 0:
                backoff[i] -= 1
            else:
                # 랜덤하게 전송 의지
                if random.random() < 0.3:
                    ready.append(i)

        if len(ready) == 0:
            continue
        elif len(ready) == 1:
            # 성공 전송
            success += 1
        else:
            # 충돌
            collisions += 1
            for i in ready:
                k = 1  # 매우 단순화: 항상 k=1로 가정
                r = random.randint(0, (1 << k) - 1)
                backoff[i] = r

    return {"success": success, "collisions": collisions}

if __name__ == "__main__":
    result = csma_cd_sim()
    print(result)
```

실제 Ethernet처럼 정확한 시간/슬롯/전파 지연 모델은 아니지만,

- 여러 호스트가 동시에 전송하면 충돌이 늘고,
- backoff 로 인해 시간이 지날수록 충돌이 줄어드는 패턴

을 직관적으로 볼 수 있다.

### Efficiency of Standard Ethernet

#### 이론적 효율 식

포화된(많은 노드가 경쟁하는) CSMA/CD Ethernet에서의
이론적 **효율(throughput/용량)** 은 다음 식으로 근사할 수 있다:

$$
\eta \approx \frac{1}{1 + 6.44 a}
$$

여기서

- $$a = \dfrac{T_p}{T_t} = \dfrac{\text{전파 지연}}{\text{프레임 전송 시간}}$$
  - $$T_p$$: 한쪽 끝에서 다른 끝까지의 **전파 지연**
  - $$T_t$$: 하나의 프레임을 전송하는데 필요한 시간

직관:

- **프레임이 길수록**(T_t가 커질수록) → a가 작음 → 효율 ↑
- **링크가 길거나 더 느릴수록**(T_p가 커질수록) → a가 큼 → 효율 ↓

#### 수치 예제

예제 1: **전형적인 10 Mbps LAN**

- 최대 전파 지연을 고려해 $$T_p \approx 25.6\ \mu s$$ 정도라 하자
  (왕복 지연이 51.2 µs이므로).
- 프레임 길이: 평균 512 bits (64 bytes)라고 가정:

  $$
  T_t = \frac{512}{10^7} = 51.2\ \mu s
  $$

- 따라서:

  $$
  a = \frac{25.6}{51.2} = 0.5
  $$

- 효율:

  $$
  \eta \approx \frac{1}{1 + 6.44 \cdot 0.5}
             = \frac{1}{1 + 3.22}
             \approx 0.237
  $$

즉, **최악에 가깝게 포화된 상황**에서 이론상 20~30% 정도 효율이 나올 수 있다는 의미다
(실제 평균 프레임이 더 길고, 네트워크가 완전 포화가 아닌 경우 효율은 더 높다).

예제 2: **프레임이 길어진 경우(1500 bytes)**

- 프레임 1500 bytes = 12000 bits
- $$T_t = 12000 / 10^7 = 1.2\ \text{ms}$$
- $$a = 25.6 \mu s / 1.2 ms \approx 0.0213$$
- 효율:

  $$
  \eta \approx \frac{1}{1 + 6.44 \cdot 0.0213}
             \approx \frac{1}{1 + 0.137}
             \approx 0.88
  $$

프레임이 충분히 길다면 CSMA/CD Ethernet도
**80~90% 수준의 효율**을 얻을 수 있음을 보여준다.

#### 효율 계산 코드

```python
def ethernet_efficiency(Tp, frame_size_bytes, R=10e6):
    """
    Tp: 전파 지연 (초)
    frame_size_bytes: 프레임 길이 (Bytes, preamble 제외)
    R: 링크 속도 (bps)
    """
    Tt = (frame_size_bytes * 8) / R
    a = Tp / Tt
    eta = 1 / (1 + 6.44 * a)
    return eta, a

if __name__ == "__main__":
    # 예제 1: 최소 프레임 64 bytes, Tp=25.6 us
    Tp = 25.6e-6
    eta1, a1 = ethernet_efficiency(Tp, 64)
    print("64B frame: a =", a1, "eff =", eta1)

    # 예제 2: 1500 bytes
    eta2, a2 = ethernet_efficiency(Tp, 1500)
    print("1500B frame: a =", a2, "eff =", eta2)
```

이 코드를 실행해 보면, 프레임 크기에 따라 효율이 어떻게 바뀌는지 수치로 볼 수 있다.

### Implementation (구현 관점)

#### 물리 토폴로지와 매체

**Classic 10 Mbps Ethernet** 의 대표 구현:

1. **10BASE5 (thick coax bus)**
   - 물리 토폴로지: 버스
   - 매체: 굵은 동축 케이블 (옛날 노란 케이블)
   - 최대 길이: 500 m/segment, 4 repeaters → 총 2500 m
   - transceiver를 케이블에 “tap” 해서 연결.

2. **10BASE2 (thin coax bus)**
   - 더 얇은 coax, BNC 커넥터, T자 어댑터 사용.
   - 세그먼트 길이 약 185 m.

3. **10BASE-T (UTP star)**
   - 물리 토폴로지: 별형 (central hub)
   - 매체: UTP Cat 3 이상, 길이 100 m
   - 논리적으로는 여전히 공유 버스 (허브가 신호를 브로드캐스트)
   - 이후 허브 → 스위치로 발전하면서 충돌 도메인이 분리.

현대의 10/100/1G/10G Ethernet은:

- 거의 항상 **Star/Leaf–Spine 구조**에서 스위치 중심,
- 각 링크는 **point-to-point full-duplex**,
- 매체: Cat 5e/6/6A/7 UTP, 광섬유, 혹은 단일 페어 Ethernet(SPE) 등.

#### 프레임 전/후 처리

OS/드라이버/NIC 관점에서 Standard Ethernet 프레임은 대략 다음 과정을 거친다:

1. **상위 계층(IP/TCP/UDP)** 이 데이터를 보내려 함.
2. OS의 네트워크 스택이 **Ethernet 헤더 추가**:
   - 목적지 MAC: ARP/NDP 등을 통해 해석.
   - 출발지 MAC: NIC의 MAC 주소.
   - Type: IPv4(0x0800), IPv6(0x86DD), ARP(0x0806) 등.
3. NIC가 **CRC-32(FCS)** 계산 후 프레임 끝에 추가.
4. PHY 계층에서 **preamble + SFD**를 붙이고 Manchester 인코딩으로 매체에 송신.
5. 수신 측 NIC는 preamble/SFD를 보고 프레임 경계 동기화 후:
   - CRC-32 검증,
   - 목적지 MAC이 자신/브로드캐스트/등록된 멀티캐스트인지 확인,
   - 상위 계층에 payload 전달.

이 전체 동작을 **802.3 MAC/PHY**가 구체적으로 정의한다.

### Changes in the Standard (표준의 진화)

Standard Ethernet(10 Mbps, CSMA/CD 기반)에서 시작해서
802.3 표준은 지금도 **지속적으로 확장·개정**되고 있다.

#### 속도와 PHY의 확장

IEEE 802.3의 중요한 확장들(대표만):

- 802.3u: **Fast Ethernet (100 Mbps)** – 100BASE-TX/FX 등
- 802.3z/ab: **Gigabit Ethernet (1000 Mbps)** – fiber + 1000BASE-T
- 802.3ae: **10GBASE-R** (10 Gbps Ethernet)
- 그 이후: 40G, 100G, 200G, 400G, 800G Ethernet 등 지속 확장

이 과정에서:

- 더 높은 대역폭을 위해 **4쌍 UTP, 초고속 직렬 링크, PAM-4 변조** 등 다양한 PHY 기술 도입.
- 표준 문서는 **Ethernet PHY 패밀리**를 계속 추가하는 방식으로 성장했다.

#### Half-duplex CSMA/CD → Full-duplex Switch

초기에는 모든 Ethernet이 **shared medium + CSMA/CD + half-duplex** 였지만,
스위치가 보급되면서:

- 각 포트가 **독립 충돌 도메인**이 되고,
- 링크는 **full-duplex** 로 동작,
- CSMA/CD 절차 자체가 **불필요**해졌다.

802.3 표준은

- 여전히 CSMA/CD MAC을 포함하지만 (역호환성),
- 실제 구현에서는 full-duplex가 기본이며
  CSMA/CD 관련 거리 제한/최소 프레임 조건 등은 **사실상 의미가 크게 줄어들었다.**

#### Auto-Negotiation, Energy Efficient Ethernet 등

속도/이더넷 모드의 다양화에 따라:

- **Auto-Negotiation (802.3u)**:
  - 두 장비가 서로 지원 가능한 속도/half-duplex/full-duplex를 협상.
  - 10/100/1G 혼재 환경에서도 플러그만 끼우면 동작하게 함.

- **Energy Efficient Ethernet (EEE, 802.3az)**:
  - 링크가 한동안 idle 상태일 때 PHY를 **저전력 모드**로 전환했다가,
    다시 트래픽이 생기면 빠르게 깨우는 메커니즘 정의.

- **Single Pair Ethernet (SPE)**:
  - 산업/자동차/IoT용으로, UTP 한 쌍만으로 이더넷 제공(예: 10BASE-T1L 등).
  - 긴 거리, 전력 공급(PoDL, Power over Data Line)과 조합.

#### 프레임 포맷 확장: VLAN, QinQ 등

802.3 MAC 프레임 포맷은 기본적으로 **Type/Length + payload** 이지만,
802.1Q 등의 표준과 결합되어:

- **VLAN Tag (802.1Q)**:
  - EtherType 0x8100을 사용해 4-byte 태그 삽입 (PCP, CFI, VLAN ID).
  - 결과적으로 **“확장된 Ethernet 프레임”** (최대 1522 bytes)이 일반화.

- **QinQ(802.1ad)**: VLAN 태그를 두 개 겹쳐 사용하는 서비스 제공자 네트워크용 확장.

이 모든 확장에도 불구하고, **기본 MAC 헤더 구조와 주소 체계, FCS 방식**은
여전히 Standard Ethernet과 본질적으로 동일하다.

## 마무리 정리

이번 절(13.1–13.2)에서 정리한 핵심 포인트:

1. **IEEE 802 프로젝트**
   - 데이터링크를 LLC/MAC으로 나누고,
   - 802.3 (Ethernet), 802.11 (Wi-Fi) 등 다양한 MAC을 공통 구조로 묶는다.

2. **Ethernet Evolution**
   - Coax 버스 + 10 Mbps → UTP + Fast Ethernet → Gigabit/10G/100G+ →
     오늘날의 스위치 기반 full-duplex 이더넷으로 진화.

3. **Standard Ethernet (10 Mbps)**
   - 10BASE5/2/T, CSMA/CD, half-duplex, 최소 프레임 64 bytes,
   - 48-bit MAC(EUI-48) 주소, 프레임 포맷, FCS-32 등.

4. **CSMA/CD와 효율**
   - 충돌 검출을 위해 최소 프레임 길이와 거리 제한이 필요,
   - 포화 상태에서 효율은 $$\eta \approx \dfrac{1}{1 + 6.44 a}$$ 로 근사,
     프레임이 길수록 효율 ↑.

5. **표준의 변화**
   - Fast/Gigabit/10G/40G/100G/400G… PHY 추가,
   - 스위치 기반 full-duplex로 CSMA/CD는 사실상 역사적인 역할,
   - Auto-negotiation, EEE, SPE, VLAN/QinQ 등 수많은 기능 확장.

이 내용을 기반으로 다음 절(예: Fast Ethernet, Gigabit Ethernet, Switched LAN 설계)을 공부할 때,

- “무엇이 그대로이고, 무엇이 변했는가”
- “10 Mbps 시절 제약(거리, CSMA/CD 등)이 오늘날 어떤 부분에서 사라졌는가”

를 축으로 보면 전체 Ethernet 계열 기술을 **한 줄로 연결해서** 이해할 수 있다.
