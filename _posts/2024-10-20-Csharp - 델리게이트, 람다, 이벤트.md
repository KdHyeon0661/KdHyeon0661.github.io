# C# 대리자, 람다, 이벤트: 메서드를 데이터로 다루는 기술

C#은 객체 지향 언어이지만, 함수형 프로그래밍의 강력한 장점들을 적극적으로 수용해 왔습니다. **대리자(Delegate)**는 메서드를 변수처럼 참조하여 안전하게 전달할 수 있게 해주고, **람다(Lambda)**는 이러한 메서드를 이름조차 없이 극도로 간결하게 표현하며, **이벤트(Event)**는 객체 간의 느슨한 결합(Loose Coupling)을 위한 발행-구독(Publish-Subscribe) 패턴을 완성합니다.

이 세 가지 개념은 톱니바퀴처럼 맞물려 작동하며, C#에서 반응형(Reactive)이고 유연한 애플리케이션을 구축하는 가장 중요한 뼈대가 됩니다. 지금부터 이들의 동작 원리와 실무 활용법을 깊이 있게 파헤쳐 보겠습니다.

---

## 1. 델리게이트(Delegate): 안전한 메서드 보관함

델리게이트는 메서드 자체를 담을 수 있는 특수한 타입입니다. C++의 함수 포인터나 자바스크립트의 콜백(Callback) 함수와 역할은 같지만, 반환 타입과 매개변수가 정확히 일치해야만 담을 수 있는 **타입 안전성(Type Safety)**을 보장합니다.

### 기본 델리게이트와 콜백 메커니즘

델리게이트의 진가는 다른 메서드에게 "작업을 수행할 때 내가 넘겨주는 이 로직(메서드)을 실행해 줘"라고 요청하는 콜백 패턴에서 발휘됩니다.

```csharp
// 1. 델리게이트 타입 선언 (int를 받고 반환값이 없는 형태)
public delegate void ProcessCallback(int number);
// 2. int를 받고 bool을 반환하는 형태
public delegate bool FilterCriteria(int number);

public class DataProcessor
{
    // 콜백 델리게이트를 매개변수로 받아 데이터 처리를 위임함
    public void ProcessData(int[] numbers, ProcessCallback callback)
    {
        foreach (int number in numbers)
        {
            callback(number); // 넘겨받은 로직 실행
        }
    }
    
    // 필터링 델리게이트를 매개변수로 받아 동적 필터링 수행
    public List<int> FilterData(int[] numbers, FilterCriteria criteria)
    {
        var result = new List<int>();
        foreach (int number in numbers)
        {
            if (criteria(number)) result.Add(number);
        }
        return result;
    }
}

class Program
{
    static void Main()
    {
        var processor = new DataProcessor();
        int[] numbers = { 1, 2, 3, 4, 5 };
        
        // 콜백 메서드 전달 (람다식 활용)
        Console.WriteLine("각 숫자 출력:");
        processor.ProcessData(numbers, n => Console.Write($"{n} "));
        
        // 짝수만 걸러내는 필터 로직 전달
        Console.WriteLine("\n\n짝수만 필터링:");
        var evenNumbers = processor.FilterData(numbers, n => n % 2 == 0);
        Console.WriteLine(string.Join(", ", evenNumbers));
    }
}

```

### 멀티캐스트 델리게이트(Multicast Delegate)

델리게이트 변수 하나에 `+=` 연산자를 사용하면 여러 개의 메서드를 줄줄이 이어 붙일 수 있습니다. 호출 한 번에 등록된 모든 메서드가 순차적으로 실행됩니다.

```csharp
public delegate void LogMessage(string message);

class Program
{
    static void LogToConsole(string message) => Console.WriteLine($"[CONSOLE] {message}");
    static void LogToFile(string message) => Console.WriteLine($"[FILE] {message}");
    static void LogToDatabase(string message) => Console.WriteLine($"[DB] {message}");
    
    static void Main()
    {
        LogMessage logger = LogToConsole;
        
        // 델리게이트에 메서드 추가 (멀티캐스트)
        logger += LogToFile;
        logger += LogToDatabase;
        
        Console.WriteLine("모든 로거 호출:");
        logger("애플리케이션 시작됨"); // 3개의 로거가 모두 실행됨
        
        Console.WriteLine("\n파일 로거 제거 후:");
        logger -= LogToFile;  // 특정 메서드 체인 끊기
        logger("작업 진행 중"); // 콘솔과 DB 로거만 실행됨
        
        // 안전한 호출 (Null 조건 연산자)
        logger?.Invoke("안전한 실행");
    }
}

```

## 2. 모던 C#의 표준: Action, Func, Predicate

과거에는 매번 `delegate` 키워드로 새로운 타입을 선언해야 했지만, C# 프레임워크는 실무에서 쓰이는 거의 모든 형태를 커버하는 세 가지 제네릭 델리게이트를 기본 제공합니다.

### Action (반환값이 없는 경우)

```csharp
// 매개변수가 없는 Action
Action simpleAction = () => Console.WriteLine("Hello, World!");
simpleAction();

// 2개의 매개변수를 받는 Action
Action<string, int> printInfo = (name, age) => Console.WriteLine($"{name} is {age} years old");
printInfo("Bob", 30);

```

### Func (반환값이 있는 경우)

```csharp
// 매개변수 없이 int 반환 (마지막 제네릭 타입이 반환 타입)
Func<int> getRandom = () => new Random().Next(1, 100);

// 2개의 int를 받아 int를 반환
Func<int, int, int> add = (a, b) => a + b;
Console.WriteLine($"3 + 5 = {add(3, 5)}");

```

### Predicate (조건을 판별하여 bool 반환)

`Func<T, bool>`과 완벽히 동일한 역할을 하며, 리스트 검색 등에 주로 쓰입니다.

```csharp
Predicate<int> isEven = n => n % 2 == 0;

List<int> numbers = new() { 1, 2, 3, 4, 5, 6 };
List<int> evenNumbers = numbers.FindAll(isEven);

```

## 3. 람다 표현식(Lambda Expression)과 클로저

람다 표현식은 `=>` 연산자를 사용하여 델리게이트에 들어갈 익명 함수를 극도로 간결하게 표현합니다.

### 람다의 활용과 LINQ

```csharp
public record Person(string Name, int Age, string Job);

class Program
{
    static void Main()
    {
        var people = new List<Person>
        {
            new Person("Alice", 25, "Engineer"),
            new Person("Bob", 30, "Designer"),
            new Person("Charlie", 22, "Student")
        };
        
        // 문장 람다 (중괄호를 사용하여 여러 줄 작성)
        Action<Person> printDetails = p =>
        {
            Console.WriteLine($"이름: {p.Name}, 나이: {p.Age}");
        };
        
        // LINQ에서의 람다 활용
        var engineers = people.Where(p => p.Job == "Engineer");
        var sortedByAge = people.OrderBy(p => p.Age);
        var namesAndJobs = people.Select(p => new { p.Name, p.Job });
        
        bool allAdults = people.All(p => p.Age >= 18);
    }
}

```

### ⚠️ 치명적인 함정: 클로저(Closure)와 루프 변수 캡처

람다식은 자신을 감싸고 있는 외부의 지역 변수를 포획(Capture)하여 내부에서 사용할 수 있습니다. 이를 클로저라고 합니다. 하지만 `for` 루프 안에서 변수를 캡처할 때 엄청난 버그가 발생할 수 있습니다.

```csharp
class Program
{
    static void Main()
    {
        var actions = new List<Action>();
        
        for (int i = 0; i < 3; i++)
        {
            int captured = i; // ✅ 지역 변수로 복사 (매우 중요!)
            
            // 만약 람다 내부에서 'i'를 직접 사용했다면?
            // 모든 람다가 같은 i의 주소를 참조하게 되어 실행 시점에 모두 3을 출력하게 됩니다.
            actions.Add(() => Console.WriteLine($"Captured value: {captured}"));
        }
        
        foreach (var action in actions)
        {
            action(); // 출력: 0, 1, 2
        }
    }
}

```

## 4. 이벤트(Event): 발행-구독 패턴의 완벽한 캡슐화

델리게이트는 강력하지만 외부에서 `=` 연산자로 기존 메서드를 덮어씌우거나, 멋대로 `Invoke()`를 호출해 버릴 수 있는 보안 취약점이 있습니다. `event` 키워드는 대리자를 캡슐화하여, 외부에서는 오직 구독(`+=`)과 해지(`-=`)만 가능하도록 접근을 제한합니다.

### 표준 이벤트 패턴 (EventHandler)

.NET 환경에서는 `EventHandler<TEventArgs>`를 사용하는 표준 패턴이 존재합니다.

```csharp
// 1. 전달할 데이터 객체
public class ButtonClickedEventArgs : EventArgs
{
    public DateTime ClickTime { get; }
    public ButtonClickedEventArgs(DateTime clickTime) => ClickTime = clickTime;
}

public class Button
{
    public string Label { get; }
    public Button(string label) => Label = label;

    // 2. event 키워드를 통한 선언
    public event EventHandler<ButtonClickedEventArgs> Clicked;
    
    public void Click()
    {
        Console.WriteLine($"버튼 '{Label}' 클릭됨.");
        // 3. 내부 호출
        OnClicked(new ButtonClickedEventArgs(DateTime.Now));
    }
    
    // 4. 보호된 가상 메서드를 통한 이벤트 발생
    protected virtual void OnClicked(ButtonClickedEventArgs e)
    {
        Clicked?.Invoke(this, e); // 널 조건 연산자로 안전하게 실행
    }
}

```

### 이벤트 접근자 커스터마이징 (add / remove)

프로퍼티에 `get`과 `set`이 있듯, 이벤트에는 `add`와 `remove` 접근자가 존재합니다. 이를 재정의하여 구독자가 추가되거나 제거될 때 특별한 로직(로그 기록 등)을 삽입할 수 있습니다.

```csharp
public class EventLogger
{
    // 실제 이벤트를 담을 내부 델리게이트
    private EventHandler<EventArgs> _logEvent;
    
    public event EventHandler<EventArgs> Logged
    {
        add
        {
            Console.WriteLine($"새 구독자 추가됨: {value.Method.Name}");
            _logEvent += value;
        }
        remove
        {
            Console.WriteLine($"구독자 제거됨: {value.Method.Name}");
            _logEvent -= value;
        }
    }
}

```

## 5. 종합 실전 예제: 온도 모니터링 시스템

대리자(`Func` 주입), 람다식, 그리고 이벤트(멀티캐스트)를 모두 활용하여 클래스 간의 결합도를 낮춘(Loose Coupling) 완벽한 실무 아키텍처입니다.

```csharp
using System;
using System.Collections.Generic;
using System.Linq;
using System.Threading;
using System.Threading.Tasks;

// 1. 델리게이트와 이벤트 인자 정의
public delegate void AlertHandler(string alertMessage);

public class TemperatureReadingEventArgs : EventArgs
{
    public double Temperature { get; }
    public DateTime ReadingTime { get; }
    
    public TemperatureReadingEventArgs(double temperature)
    {
        Temperature = temperature;
        ReadingTime = DateTime.Now;
    }
}

// 2. 센서 (발행자: Publisher)
public class TemperatureSensor
{
    private readonly Random _random = new();
    private double _currentTemperature = 20.0;
    private bool _isMonitoring = false;
    
    // 표준 이벤트 선언
    public event EventHandler<TemperatureReadingEventArgs> TemperatureRead;
    public event EventHandler<TemperatureReadingEventArgs> HighTemperatureAlert;
    public event EventHandler<TemperatureReadingEventArgs> LowTemperatureAlert;
    
    // 커스텀 델리게이트 이벤트
    public event AlertHandler CustomAlert;
    
    public void ReadTemperature()
    {
        double change = (_random.NextDouble() * 10) - 5;
        _currentTemperature += change;
        
        var args = new TemperatureReadingEventArgs(_currentTemperature);
        TemperatureRead?.Invoke(this, args);
        
        CheckAlerts(args);
    }
    
    private void CheckAlerts(TemperatureReadingEventArgs e)
    {
        if (e.Temperature > 30.0)
        {
            HighTemperatureAlert?.Invoke(this, e);
            CustomAlert?.Invoke($"고온 경고: {e.Temperature:F1}°C");
        }
        else if (e.Temperature < 10.0)
        {
            LowTemperatureAlert?.Invoke(this, e);
            CustomAlert?.Invoke($"저온 경고: {e.Temperature:F1}°C");
        }
    }
    
    public async Task StartMonitoringAsync(CancellationToken cancellationToken = default)
    {
        _isMonitoring = true;
        while (_isMonitoring && !cancellationToken.IsCancellationRequested)
        {
            ReadTemperature();
            await Task.Delay(1000, cancellationToken);
        }
    }
    
    public void StopMonitoring() => _isMonitoring = false;
}

// 3. 디스플레이 (구독자: Subscriber)
public class TemperatureDisplay
{
    public void SubscribeToSensor(TemperatureSensor sensor)
    {
        // 람다를 사용한 직관적인 이벤트 구독
        sensor.TemperatureRead += (sender, e) =>
            Console.WriteLine($"[{e.ReadingTime:HH:mm:ss}] 현재 온도: {e.Temperature:F1}°C");
            
        sensor.HighTemperatureAlert += (sender, e) =>
        {
            Console.ForegroundColor = ConsoleColor.Red;
            Console.WriteLine($"[주의] 고온: {e.Temperature:F1}°C");
            Console.ResetColor();
        };
        
        sensor.LowTemperatureAlert += (sender, e) =>
        {
            Console.ForegroundColor = ConsoleColor.Blue;
            Console.WriteLine($"[주의] 저온: {e.Temperature:F1}°C");
            Console.ResetColor();
        };
    }
}

// 4. 로거 서비스
public class TemperatureLogger
{
    private readonly List<string> _logEntries = new();
    
    public void SubscribeToSensor(TemperatureSensor sensor)
    {
        sensor.TemperatureRead += OnTemperatureRead;
        sensor.CustomAlert += OnCustomAlert;
    }
    
    private void OnTemperatureRead(object sender, TemperatureReadingEventArgs e)
    {
        _logEntries.Add($"{e.ReadingTime:HH:mm:ss} | 온도: {e.Temperature:F1}°C");
    }
    
    private void OnCustomAlert(string alertMessage)
    {
        _logEntries.Add($"{DateTime.Now:HH:mm:ss} | 알림: {alertMessage}");
    }
    
    public void PrintLog()
    {
        Console.WriteLine("\n=== 온도 로그 기록 ===");
        foreach (var entry in _logEntries) Console.WriteLine(entry);
    }
}

// 5. 분석기 (Func 델리게이트를 통한 전략 주입)
public class TemperatureAnalyzer
{
    private readonly List<double> _temperatureReadings = new();
    
    // Func를 사용하여 외부에서 "분석 알고리즘"을 주입받습니다.
    public double Analyze(Func<List<double>, double> analysisFunction)
    {
        if (_temperatureReadings.Count == 0) return 0;
        return analysisFunction(_temperatureReadings);
    }
    
    public void AddReading(double temperature) => _temperatureReadings.Add(temperature);
    
    public void PrintAnalysis()
    {
        if (_temperatureReadings.Count == 0) return;
        
        Console.WriteLine("\n=== 온도 분석 통계 ===");
        // 람다를 통해 동적으로 로직을 전달
        Console.WriteLine($"평균 온도: {Analyze(readings => readings.Average()):F1}°C");
        Console.WriteLine($"최고 온도: {Analyze(readings => readings.Max()):F1}°C");
        Console.WriteLine($"최저 온도: {Analyze(readings => readings.Min()):F1}°C");
        Console.WriteLine($"온도 편차: {Analyze(readings => readings.Max() - readings.Min()):F1}°C");
    }
}

// 6. 메인 실행
class Program
{
    static async Task Main()
    {
        var sensor = new TemperatureSensor();
        var display = new TemperatureDisplay();
        var logger = new TemperatureLogger();
        var analyzer = new TemperatureAnalyzer();
        
        // 각 모듈이 센서를 구독 (서로의 존재를 모르지만 완벽히 연동됨)
        display.SubscribeToSensor(sensor);
        logger.SubscribeToSensor(sensor);
        sensor.TemperatureRead += (sender, e) => analyzer.AddReading(e.Temperature);
        
        Console.WriteLine("=== 모니터링 시스템 가동 (5초간) ===\n");
        var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));
        
        try { await sensor.StartMonitoringAsync(cts.Token); }
        catch (OperationCanceledException) { }
        
        Console.WriteLine("\n=== 모니터링 종료 ===");
        analyzer.PrintAnalysis();
        logger.PrintLog();
    }
}

```

## 마무리

C#에서 대리자(Delegate)는 메서드를 담는 근본적인 그릇이며, 람다(Lambda)는 그 그릇에 메서드를 가장 직관적으로 담아내는 문법입니다. 그리고 이벤트(Event)는 이 그릇을 안전하게 보호하여 견고한 옵저버(Observer) 패턴을 구축하는 건축 자재입니다.

실무에서 클래스 간의 강한 결합(Coupling)을 피하고 싶거나, 런타임에 동적으로 알고리즘을 주입받아 유연하게 대처하는 파이프라인을 설계하고 싶다면 반드시 이 세 가지 개념을 통달해야 합니다.

특히 람다 클로저의 루프 변수 캡처 문제나, 이벤트 구독 후 발생하는 메모리 누수(Memory Leak)를 막기 위해 해지(`-=`)를 잊지 않는다면 훨씬 완성도 높은 애플리케이션을 설계하실 수 있을 것입니다.
