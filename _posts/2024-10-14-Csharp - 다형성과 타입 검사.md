# C# 다형성과 타입 검사: 유연한 객체 지향 설계의 완성

객체 지향 프로그래밍(OOP)을 지탱하는 거대한 기둥 중 하나는 바로 **다형성(Polymorphism)**입니다. 다형성이란 '여러 가지 형태를 가질 수 있는 능력'을 의미하며, 하나의 코드가 객체의 실제 타입에 따라 각기 다른 동작을 수행하도록 만들어줍니다. 이를 통해 개발자는 기존 코드를 수정하지 않고도 새로운 기능을 쉽게 추가할 수 있는 유연하고 확장 가능한 아키텍처를 구축할 수 있습니다.

C#은 이러한 다형성을 상속과 가상 메서드를 통해 우아하게 구현합니다. 또한, 프로그램이 실행되는 도중(런타임)에 객체의 진짜 정체를 확인하고 안전하게 다루기 위해 `is`, `as`, `typeof`와 같은 정교한 타입 검사 도구들을 제공합니다. 이 글에서는 다형성의 본질적인 개념부터 시작해, 타입 검사 연산자들의 정확한 차이점, 그리고 실무에서 다형성을 활용하는 대표적인 설계 패턴까지 깊이 있게 살펴보겠습니다.

## 상속을 통한 다형성의 실현

다형성의 가장 기본적인 형태는 부모 클래스가 정의한 틀을 자식 클래스들이 각자의 방식대로 채워 넣는 것입니다. C#에서는 부모 클래스의 메서드에 `virtual` 키워드를 붙여 재정의를 허용하고, 자식 클래스에서 `override` 키워드를 사용해 다형성을 활성화합니다.

### 다형성의 기본 개념과 활용

동물들이 각기 다른 소리를 내는 상황을 코드로 모델링해 보겠습니다.

```csharp
public class Animal
{
    public string Name { get; set; } = "";
    
    // virtual: 파생 클래스에서 이 동작을 덮어쓸 수 있도록 허락합니다.
    public virtual void MakeSound()
    {
        Console.WriteLine("동물이 소리를 냅니다.");
    }
}

public class Dog : Animal
{
    public override void MakeSound() => Console.WriteLine("멍멍!");
}

public class Cat : Animal
{
    public override void MakeSound() => Console.WriteLine("야옹~");
}

```

다형성의 진정한 위력은 여러 자식 객체들을 부모 타입이라는 하나의 바구니에 담아서 일괄적으로 처리할 때 나타납니다.

```csharp
class Program
{
    static void Main()
    {
        // 부모 타입인 Animal 배열 하나에 다양한 자식 객체들을 섞어서 보관할 수 있습니다.
        Animal[] animals = new Animal[]
        {
            new Dog { Name = "바둑이" },
            new Cat { Name = "나비" }
        };
        
        // 컴파일러는 Animal의 MakeSound()를 호출하는 것처럼 보이지만,
        // 실제 런타임에는 메모리에 있는 객체의 진짜 타입(Dog, Cat)에 맞는 메서드가 실행됩니다.
        foreach (Animal animal in animals)
        {
            Console.Write($"{animal.Name}: ");
            animal.MakeSound(); 
        }
    }
}

```

이렇게 코드를 작성하면, 훗날 `Cow`나 `Bird` 같은 새로운 동물 클래스가 추가되더라도 `foreach` 반복문이나 출력 로직은 단 한 줄도 수정할 필요가 없습니다. 이것이 바로 다형성이 가져다주는 유지보수성의 핵심입니다.

### 추상 클래스를 활용한 강력한 제어

부모 클래스 자체가 불완전하여 스스로 객체가 될 수 없고, 오직 자식들에게 뼈대만 제공해야 할 때는 `abstract` (추상) 클래스를 사용합니다. 추상 메서드는 구현부가 없으며, 자식 클래스에서 무조건 구현하도록 강제하는 강력한 계약(Contract) 역할을 합니다.

```csharp
public abstract class Shape
{
    // 추상 메서드: 자식 클래스는 반드시 넓이와 둘레를 구하는 공식을 직접 구현해야 합니다.
    public abstract double GetArea();
    public abstract double GetPerimeter();
    
    // 추상 클래스 내부에서도 공통으로 사용할 일반 메서드를 정의할 수 있습니다.
    public void DisplayInfo()
    {
        Console.WriteLine($"넓이: {GetArea():F2}, 둘레: {GetPerimeter():F2}");
    }
}

public class Circle : Shape
{
    public double Radius { get; set; }
    
    // 원의 공식에 맞게 메서드를 완성합니다.
    public override double GetArea() => Math.PI * Radius * Radius;
    public override double GetPerimeter() => 2 * Math.PI * Radius;
}

```

## 런타임 타입 검사와 안전한 형변환

다형성을 활용해 부모 타입으로 객체들을 묶어두다 보면, 때로는 이 객체의 진짜 정체가 무엇인지 확인하고 자식 클래스만의 특수한 기능을 호출해야 할 때가 생깁니다. 이때 C#이 제공하는 런타임 타입 식별(RTTI) 연산자들을 사용합니다.

### is 연산자와 패턴 매칭

`is` 연산자는 특정 객체가 지정한 타입과 일치하거나, 해당 타입을 상속받았는지 논리적으로 확인합니다. 특히 C# 7.0부터는 타입 확인과 동시에 변수 선언까지 한 번에 처리하는 '패턴 매칭' 기능이 결합되어 코드가 극도로 간결해졌습니다.

```csharp
public class Vehicle { }
public class Car : Vehicle { public void Honk() => Console.WriteLine("빵빵!"); }
public class Bicycle : Vehicle { public void RingBell() => Console.WriteLine("따르릉~"); }

object[] things = { new Car(), new Bicycle(), "문자열", 123, null };

foreach (var thing in things)
{
    // thing이 Car 타입이 맞다면, 즉시 car라는 변수에 캐스팅하여 할당합니다.
    if (thing is Car car)
    {
        car.Honk(); // 별도의 캐스팅 작업 없이 즉시 Car의 메서드를 사용할 수 있습니다.
    }
    else if (thing is string text)
    {
        Console.WriteLine($"문자열의 길이: {text.Length}");
    }
}

```

최신 C#의 `switch` 식(Expression)을 활용하면 복잡한 타입 분기 처리를 아주 우아하게 작성할 수 있습니다.

```csharp
public static string DescribeObject(object obj)
{
    return obj switch
    {
        null => "값이 없습니다",
        int i when i < 0 => $"음수 {i}",
        string s when s.Length == 0 => "빈 문자열",
        Car c => $"자동차 (브랜드: {c.Brand ?? "알 수 없음"})",
        _ => $"기타 알 수 없는 객체: {obj.GetType().Name}"
    };
}

```

### as 연산자: 예외 없는 부드러운 형변환

전통적인 강제 캐스팅 `(Car)obj` 방식을 사용하면, 형변환에 실패했을 때 프로그램이 곧바로 `InvalidCastException`을 던지며 뻗어버립니다. 반면 `as` 연산자는 형변환을 시도해 보고, 실패하면 조용히 `null`을 반환합니다.

```csharp
object myObj = new Bicycle();

// as 연산자를 통한 안전한 캐스팅 (참조 타입에만 사용 가능)
Car? myCar = myObj as Car; 

// 캐스팅 실패 시 null이 반환되므로, 널 조건 연산자(?.) 등과 결합하여 안전하게 처리할 수 있습니다.
if (myCar != null)
{
    myCar.Honk();
}
else
{
    Console.WriteLine("이 객체는 자동차가 아닙니다.");
}

```

### typeof vs GetType() 의 명확한 차이

이 두 가지는 모두 타입 정보(`Type` 객체)를 얻어오지만, 작동하는 시점과 주체가 완전히 다릅니다. 이 차이를 명확히 아는 것은 리플렉션(Reflection)을 다룰 때 매우 중요합니다.

* **`typeof`**: **컴파일 타임**에 동작합니다. 클래스의 '이름' 자체를 매개변수로 받아서 타입 정보를 반환합니다.
* **`GetType()`**: **런타임**에 동작합니다. 메모리에 살아있는 '인스턴스(객체)'를 상대로 호출하여, 그 객체의 진짜 런타임 타입을 반환합니다.

```csharp
Animal animal = new Dog(); // 컴파일 타임 변수 타입은 Animal, 런타임 실제 객체는 Dog

// 1. typeof는 클래스명 자체를 넘깁니다.
Type compileTimeType = typeof(Animal);
Console.WriteLine(compileTimeType.Name); // 출력: Animal

// 2. GetType()은 생성된 인스턴스를 통해 호출합니다.
Type runtimeType = animal.GetType();
Console.WriteLine(runtimeType.Name);     // 출력: Dog

// 3. 비교
Console.WriteLine(animal is Animal);                           // True (Dog는 Animal을 상속받았으므로)
Console.WriteLine(animal.GetType() == typeof(Animal));         // False (실제 타입 Dog와 Animal은 다르므로)

```

## 다형성을 활용한 객체 지향 설계 패턴

다형성과 타입 검사 원리를 융합하면, 실무에서 마주하는 복잡한 비즈니스 로직을 깔끔하게 분리하는 디자인 패턴들을 구현할 수 있습니다.

### 전략 패턴 (Strategy Pattern)

어떤 목적을 달성하는 방법(전략)이 여러 가지일 때, 이 전략들을 객체화하고 런타임에 갈아 끼울 수 있도록 설계하는 패턴입니다. 결제 시스템에 다형성을 적용해 보겠습니다.

```csharp
// 추상화된 전략의 뼈대
public abstract class PaymentStrategy
{
    public abstract void ProcessPayment(decimal amount);
}

// 구체적인 전략 1: 신용카드
public class CreditCardPayment : PaymentStrategy
{
    private string _cardNumber;
    public CreditCardPayment(string cardNumber) => _cardNumber = cardNumber;
    
    public override void ProcessPayment(decimal amount)
    {
        Console.WriteLine($"{amount:C}를 신용카드({_cardNumber})로 결제합니다.");
    }
}

// 구체적인 전략 2: 페이팔
public class PayPalPayment : PaymentStrategy
{
    private string _email;
    public PayPalPayment(string email) => _email = email;
    
    public override void ProcessPayment(decimal amount)
    {
        Console.WriteLine($"{amount:C}를 PayPal({_email})로 결제합니다.");
    }
}

// 문맥(Context): 전략을 가져다 쓰는 클래스
public class PaymentProcessor
{
    private PaymentStrategy _strategy;
    
    public void SetStrategy(PaymentStrategy strategy) => _strategy = strategy;
    
    public void Pay(decimal amount)
    {
        // 내부에 어떤 결제 수단이 들어있든 상관없이, 다형성에 의해 알맞은 결제 로직이 실행됩니다.
        _strategy.ProcessPayment(amount);
    }
}

```

이제 새로운 결제 수단(카카오페이, 비트코인 등)이 추가되어도 `PaymentProcessor` 클래스의 코드는 전혀 수정할 필요가 없습니다. 이것이 객체 지향의 '개방-폐쇄 원칙(OCP)'이 실현되는 순간입니다.

### 템플릿 메서드 패턴 (Template Method Pattern)

추상 클래스를 사용하여 전체적인 알고리즘의 순서(템플릿)는 부모가 고정하고, 그 내부의 세부 구현 단계만 자식 클래스에게 위임하는 패턴입니다.

{% raw %}

```csharp
public abstract class DataExporter
{
    // 템플릿 메서드: 전체 작업의 순서를 결정합니다. 
    // 외부에서 흐름을 훼손하지 못하도록 설계됩니다.
    public void ExportData(string data)
    {
        var processedData = ProcessData(data); // 1. 공통 처리
        var formattedData = FormatData(processedData); // 2. 자식에게 위임한 포맷팅
        SaveData(formattedData); // 3. 자식에게 위임한 저장
    }
    
    // 부모 클래스가 담당하는 공통 로직
    private string ProcessData(string data) => data.Trim().ToUpperInvariant();
    
    // 하위 클래스가 무조건 구현해야 하는 개별 로직
    protected abstract string FormatData(string data);
    protected abstract void SaveData(string data);
}

public class JsonExporter : DataExporter
{
    protected override string FormatData(string data)
    {
        return $"{{\"data\": \"{data}\"}}";
    }
    
    protected override void SaveData(string data)
    {
        Console.WriteLine($"JSON 형식으로 파일 저장 완료: {data}");
    }
}

```

## 주의사항 및 성능 고려사항

다형성과 타입 검사 도구를 사용할 때 주의해야 할 성능 및 아키텍처 관점의 이슈들이 있습니다.

### 값 형식(Struct)과 다형성의 한계

이 글에서 설명한 상속 기반의 다형성은 모두 힙(Heap) 영역에 생성되는 참조 타입(Class)에만 해당합니다. C#의 구조체(`struct`)와 같은 값 타입은 상속을 지원하지 않으므로 직접적인 다형성을 띌 수 없습니다. 만약 값 타입을 인터페이스 타입 변수에 억지로 담아 다형성을 구현하려 하면, 힙에 메모리를 복사하는 무거운 박싱(Boxing) 연산이 발생하여 심각한 성능 저하를 초래할 수 있습니다.

### 과도한 타입 검사(is/as)는 설계의 실패를 암시한다

`is`나 `as`를 통한 타입 식별은 런타임에 메타데이터를 뒤져보는 비용(대략 

$$O(1)$$

의 상수 시간이지만 무시할 수 없는 비용)이 발생합니다. 하지만 성능보다 더 큰 문제는 **객체 지향 원칙의 붕괴**입니다.

만약 여러분의 코드에 `if (animal is Dog) ... else if (animal is Cat)` 형태로 타입을 검사하여 분기하는 코드가 가득하다면, 이는 다형성을 제대로 활용하지 못하고 있다는 '코드 냄새(Code Smell)'입니다. 이런 경우, 밖에서 타입을 확인하여 로직을 처리하는 대신 `virtual` 메서드를 열어주어 **객체 스스로가 자신의 행동을 결정하도록** 설계를 리팩토링해야 합니다.

## 결론: 적절한 추상화와 타입 안전성의 균형

다형성은 객체 지향의 축복이지만, 남용하면 코드를 추적하기 어렵게 만듭니다. 반대로 타입 검사 연산자는 편리하지만, 과용하면 절차 지향적인 스파게티 코드를 낳습니다.

* **다형성을 우선하십시오**: 코드의 결합도를 낮추기 위해 객체의 구체적인 타입보다는 부모 클래스나 인터페이스의 추상화된 행동에 의존하세요.
* **타입 검사는 방어용으로만 사용하십시오**: 외부 시스템에서 넘어온 객체를 변환하거나 불가피한 예외 로직을 처리할 때만 `is` 패턴 매칭과 `as` 연산자를 활용하세요.

설계를 하실 때 "어떻게 하면 외부에서 이 객체의 타입을 모른 채로 지시만 내릴 수 있을까?"를 먼저 고민해 보시기 바랍니다. 이 간단한 사고의 전환이 여러분의 C# 아키텍처를 한 차원 더 높은 수준으로 끌어올려 줄 것입니다.
