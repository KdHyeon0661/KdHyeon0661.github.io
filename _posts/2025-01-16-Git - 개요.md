# GitHub란 무엇인가?

## 개요: Git과 GitHub, 그리고 개발 생태계

- **Git**: 분산 버전 관리 시스템. 코드 변경 이력을 로컬에서 관리하는 핵심 도구입니다.
- **GitHub**: Git 저장소를 호스팅하고, 코드 리뷰(Pull Request), 이슈 트래킹, CI/CD(Actions), 보안 분석 등 **협업과 자동화를 위한 플랫폼**을 제공합니다.

**실무에서의 역할**
개발자는 로컬에서 Git으로 작업한 후, GitHub에 코드를 푸시하여 협업의 중심으로 삼습니다. 여기서 코드 리뷰, 자동화된 테스트와 빌드, 보안 점검을 거쳐 최종적으로 배포까지 이어지는 **통합 개발 라이프사이클**을 관리할 수 있습니다.

---

## 역사와 진화

- **2005**: 리누스 토르발스가 Linux 커널 개발을 위해 Git을 만듦.
- **2008**: GitHub가 설립되며 Git 기반 협업이 대중화.
- **2018**: Microsoft의 인수 이후 엔터프라이즈 기능과 생태계가 급성장.
- **현재**: 단순한 코드 저장소를 넘어 Actions(CI/CD), Codespaces(클라우드 IDE), Advanced Security, Copilot(AI 보조) 등을 포함한 **종합 개발자 플랫폼**으로 진화했습니다.

---

## Git과 GitHub의 핵심 차이

| 항목 | Git | GitHub |
| :--- | :--- | :--- |
| **성격** | 버전 관리 **도구** | 협업 및 자동화 **플랫폼** |
| **주요 기능** | 커밋, 브랜치, 머지 등 버전 관리 | 저장소 호스팅, PR/이슈, Actions, 보안, 프로젝트 관리 |
| **운영 위치** | 개발자 로컬 머신 | 클라우드 서비스(github.com) 또는 엔터프라이즈 서버 |
| **핵심 가치** | 빠르고 강력한 분산형 이력 관리 | 표준화된 협업 프로세스, 자동화, 생태계 통합 |

**실무 포인트**
Git은 강력하지만, 팀 단위로 사용할 때는 일관된 규칙이 필요합니다. GitHub는 **보호된 브랜치, 필수 리뷰, 상태 검사** 등의 정책을 통해 프로젝트 표준을 유지하고 품질을 관리하는 데 핵심 역할을 합니다.

---

## 주요 대안 플랫폼

- **GitLab**: 풍부한 내장 CI/CD와 강력한 자체 호스팅 옵션.
- **Bitbucket**: Jira, Confluence 등 Atlassian 제품군과의 깊은 통합.
- **Azure DevOps**: 마이크로소프트 생태계의 일체형 프로젝트 관리 도구.

**선택 가이드**
- **오픈소스 & 커뮤니티**: GitHub의 방대한 생태계가 장점입니다.
- **기업 내부 배포**: GitHub Enterprise Server 또는 GitLab Self-Managed를 고려하세요.
- **기존 Atlassian 사용팀**: Bitbucket이 매끄러운 통합을 제공합니다.

## 시작하기: 설치와 기본 설정

### 1. Git 설치
[git-scm.com](https://git-scm.com)에서 운영체제에 맞는 설치 프로그램을 다운로드받아 설치합니다. 설치 후 터미널에서 버전을 확인해보세요.
```bash
git --version
```

### 2. GitHub 계정 생성
[github.com](https://github.com)에 접속하여 회원가입을 합니다. **2단계 인증(2FA) 설정을 꼭 활성화**하여 보안을 강화하세요.

### 3. 로컬 Git 기본 설정
작성한 커밋에 정확한 작성자 정보를 포함시키기 위해 사용자 이름과 이메일을 설정합니다.
```bash
git config --global user.name "당신의 이름"
git config --global user.email "github-가입-이메일"
```

## 첫 번째 저장소와 기본 작업 흐름

### 저장소 생성하기
1. **GitHub 웹사이트에서 생성**: 'New repository' 버튼을 클릭하고 이름, 설명, 공개 여부를 설정합니다.
2. **로컬에서 생성 후 연결**:
```bash
mkdir my-project
cd my-project
git init
echo "# My Project" > README.md
git add README.md
git commit -m "첫 커밋: 프로젝트 시작"
```

### 원격 저장소에 연결하고 푸시하기
GitHub에서 생성한 저장소의 URL을 이용해 로컬 저장소와 연결한 후 코드를 푸시합니다.
```bash
git remote add origin https://github.com/당신의계정/my-project.git
git branch -M main
git push -u origin main
```

### 기본 작업 흐름: 브랜치 활용
새 기능이나 버그 수정은 `main` 브랜치가 아닌 별도의 브랜치에서 작업하는 것이 좋습니다.
```bash
# 1. 새로운 기능 브랜치 생성 및 이동
git checkout -b feature/new-login

# 2. 파일을 수정하고 커밋
git add .
git commit -m "feat: 새로운 로그인 UI 추가"

# 3. 원격 저장소에 푸시
git push -u origin feature/new-login
```
이후 GitHub 웹사이트에서 `Pull Request(PR)`를 생성하여 팀원들의 코드 리뷰를 요청합니다.

## 협업의 핵심: Pull Request와 코드 리뷰

Pull Request는 GitHub 협업의 중심입니다. 자신의 코드 변경 사항을 제안하고, 토론하고, 개선한 후 `main` 브랜치에 병합하는 과정을 관리합니다.

**효과적인 PR 작성법**
- **제목**: 변경 사항을 명확히 요약 (예: `fix: 로그인 버튼 클릭 오류 수정`)
- **설명**: 무엇을, 왜 변경했는지 설명하고 관련 이슈 번호를 링크합니다.
- **리뷰어 지정**: 해당 코드를 잘 아는 팀원을 리뷰어로 지정합니다.

리뷰를 통해 버그를 미리 발견하고, 코드 품질을 높이며, 지식을 공유할 수 있습니다.

## 자동화의 힘: GitHub Actions

GitHub Actions를 사용하면 코드 푸시, PR 생성 등 특정 이벤트가 발생할 때 자동으로 스크립트를 실행할 수 있습니다. **CI/CD(지속적 통합/지속적 배포)** 파이프라인을 구축하는 데 필수적입니다.

**간단한 CI 워크플로우 예시 (`.github/workflows/ci.yml`)**
```yaml
name: CI
on: [push, pull_request] # push나 PR 이벤트 시 실행

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4 # 코드 체크아웃
      - name: Install Dependencies
        run: npm ci
      - name: Run Tests
        run: npm test
```
이렇게 설정하면 코드 변경 시마다 자동으로 테스트가 실행되어 `main` 브랜치의 안정성을 유지하는 데 도움이 됩니다.

---

## 필수 보안 관행

1. **2단계 인증(2FA)**: 계정 보안의 첫 번째 관문입니다.
2. **Personal Access Token(PAT)**: HTTPS로 푸시할 때 비밀번호 대신 사용하는 더 안전한 토큰입니다.
3. **Dependabot**: `/.github/dependabot.yml` 파일을 설정하면 프로젝트의 의존성 라이브러리에 보안 업데이트가 있을 때 자동으로 PR을 생성해 알려줍니다.
4. **시크릿 관리**: API 키, 비밀번호 등은 절대 코드에 직접 적지 말고, 저장소 설정의 **Secrets**에 안전하게 저장하고 Actions에서 참조하세요.

## 정리: GitHub를 효과적으로 사용하는 핵심

GitHub는 단순한 코드 백업 도구가 아닙니다. 다음 원칙을 기억하세요.

- **작은 단위의 PR**: 이해하고 리뷰하기 쉬운 크기의 변경 사항을 제출하세요.
- **명확한 커뮤니케이션**: PR과 이슈에서 변경 배경과 목적을 명확히 기술하세요.
- **자동화 신뢰**: 테스트, 린트, 빌드를 Actions에 맡겨 반복 작업을 줄이고 품질을 확보하세요.
- **보안 우선**: 2FA, Dependabot, 시크릿 관리는 선택이 아닌 필수입니다.

GitHub는 강력한 도구이지만, 그 힘은 **일관된 팀 규칙과 책임감 있는 협업 문화**에서 비롯됩니다. 체계적인 PR 프로세스, 자동화된 검증 단계, 그리고 지속적인 보안 점검이 결합될 때 개발 생산성과 소프트웨어 품질은 함께 향상됩니다.
