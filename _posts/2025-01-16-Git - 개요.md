# GitHub란 무엇인가?

## 개요 재정리: Git vs GitHub와 **개발 수명주기 속 GitHub의 역할**

- **Git**: 로컬(또는 서버)에서 동작하는 **분산 버전 관리 시스템**. 커밋, 브랜치, 머지, 태그, 리베이스 등 **이력 관리**의 표준.
- **GitHub**: Git 저장소를 **원격으로 호스팅**하고, **협업 기능(issues, PR, 리뷰, Discussions)**, **자동화(Actions)**, **보안(코드 스캐닝/시크릿 스캐닝)**, **패키징/배포(Packages/Pages)**, **IDE(Codespaces)** 등을 제공하는 **개발 플랫폼**.

**현업 관점에서의 위치**
- 로컬 개발 → 원격 푸시 → **PR 리뷰** → **CI(빌드/테스트)** → **보안 점검** → **릴리스/배포** → **이슈 관리/프로젝트 보드** → 피드백 루프
- GitHub는 단순 저장소를 넘어서 위 **엔드투엔드(End-to-End) 개발 파이프라인**의 허브 역할을 수행

---

## 역사 요약 확장: 중요 사건과 기능 진화

- **2005**: 리누스 토르발스가 대규모 협업을 위해 **Git** 개발
- **2008**: GitHub 설립, Git 기반 협업 플랫폼 대중화
- **2018**: Microsoft가 GitHub 인수(약 75억 달러) → 엔터프라이즈급 통합·보안·생태계 급성장
- **이후**:
  - **Actions**(CI/CD) 도입으로 저장소 중심 자동화 확산
  - **Packages**(컨테이너/패키지 레지스트리)
  - **Codespaces**(클라우드 개발환경)
  - **Advanced Security**(코드/시크릿 스캐닝, Dependabot, 코드QL 분석)
  - **Copilot**(AI 코딩 보조) 등으로 **개발자 도구 플랫폼**으로 진화

## Git과 GitHub 차이 — 핵심 테이블 재현 + 실무 포인트

| 항목       | Git                                   | GitHub                                                        |
|------------|----------------------------------------|----------------------------------------------------------------|
| 정의       | 로컬/분산 버전관리 도구                | Git 저장소 원격 호스팅 + 협업/자동화 플랫폼                    |
| 주 기능    | 커밋/브랜치/머지/리베이스/태그          | PR/리뷰/이슈/프로젝트/Actions/보안/패키징/배포/코드스페이스     |
| 사용 위치  | 로컬 및 어떠한 원격 서버               | github.com 및 GitHub Enterprise Server                         |
| 장점       | 속도/분산/이력관리                     | 협업 표준화/정책·보안/자동화/생태계                           |
| 연결 방식  | `git remote add origin ...`            | 저장소/조직/팀/권한/브랜치 보호/필수 리뷰/상태체크 등 정책 제공 |

**실무 포인트**
- Git은 **이력 조작 능력**(리베이스/체리픽 등)이 강력하나, 팀 표준 없이 사용하면 충돌과 혼란이 커짐 → **GitHub 정책**(보호 브랜치, 리뷰 필수, 상태검사)로 **프로세스 제어**가 필수.

## 로컬 Git 초기 설정 — 확장

```bash
git config --global user.name  "홍길동"
git config --global user.email "hong@example.com"

# 권장: 기본 에디터/기본 브랜치명/색상/자격 증명 캐시

git config --global core.editor "code --wait"    # VS Code를 Git 편집기로
git config --global init.defaultBranch main       # 기본 브랜치명을 main으로
git config --global color.ui auto
git config --global credential.helper manager     # Win은 Manager, Mac은 osxkeychain 등
```

**GPG/SSH 서명 설정(권장)**
```bash
# SSH 키 생성 (ed25519 권장)

ssh-keygen -t ed25519 -C "hong@example.com"

# 공개키 출력 후 GitHub에 등록

cat ~/.ssh/id_ed25519.pub

# 커밋 서명(GPG) 활성화 예시

git config --global user.signingkey <GPG_KEY_ID>
git config --global commit.gpgsign true
```

## 첫 저장소 만들기와 기본 워크플로우

### 새 저장소 생성(두 가지 방식)

**A) GitHub UI**
1) New repository → 이름/공개 여부/라이선스/README 선택
2) 로컬에 클론 후 작업

```bash
git clone git@github.com:username/repo.git
cd repo
```

**B) 로컬 먼저 → 원격 연결**
```bash
mkdir myapp && cd myapp
git init
echo "# MyApp" > README.md
git add .
git commit -m "init: scaffold"
git branch -M main
git remote add origin git@github.com:username/myapp.git
git push -u origin main
```

### 브랜치 전략(초급 → 중급 → 실무)

- **초급**: `main` + 작업용 단기 브랜치(feature/hotfix) → PR 머지
- **중급**: Git Flow(`main`, `develop`, `feature/*`, `release/*`, `hotfix/*`)
- **실무 최소주의**: Trunk-based(`main` 보호) + **작은 PR** + **Feature Flag**로 점진 배포

```bash
# 작업 브랜치 생성/스위치

git checkout -b feature/login

# 작업/커밋

git add src/*
git commit -m "feat(login): add oauth login UI"

# 원격 푸시 후 PR 생성

git push -u origin feature/login
```

## Pull Request(PR), 코드 리뷰, 머지 전략

### PR 작성 베스트 프랙티스

- **제목 규칙**: `feat: ...`, `fix: ...`, `refactor: ...`, `docs: ...` 등
- **본문**: 변경 요약, 해결 이슈 `Closes #123`, 스크린샷/로그/측정치
- **체크리스트**: 테스트 통과 여부, 문서 업데이트, 마이그레이션 안내

**PR 템플릿 예시** `.github/pull_request_template.md`
```markdown
## 요약

- 변경사항을 요약합니다.

## 이슈

- Closes #ISSUE_ID

## 변경 내역

- [ ] 기능 추가
- [ ] 버그 수정
- [ ] 리팩터링
- [ ] 문서

## 테스트

- [ ] 단위 테스트 통과
- [ ] 로컬 시나리오 검증: <명시>
```

### 리뷰 문화와 정책

- **필수 리뷰어**/코드 오너: `CODEOWNERS`
- **상태 체크**: 빌드/테스트/보안 스캔 성공 필수
- **브랜치 보호**: `main`에 직접 푸시 금지, 리뷰/체크 필수

`CODEOWNERS` 예시:
```
# 팀/개인 자동 리뷰어 할당

/docs/ @docs-team
/frontend/ @fe-core @alice
```

### 머지 전략

- **Merge commit**: 이력 보존, 다만 잡음 증가
- **Squash merge**: PR을 한 커밋으로 압축(히스토리 깔끔)
- **Rebase merge**: 선형 이력, 충돌 해결 능력 요구

팀 표준: **Squash + Conventional Commits** 권장
버그 추적/릴리스 노트 자동화에 유리

## GitHub Actions — CI/CD 핵심 가이드와 실전 워크플로우

### 기본 구조

`.github/workflows/ci.yml`
```yaml
name: CI
on:
  push:
    branches: [ main ]
  pull_request:

jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '22'

      - name: Install
        run: npm ci

      - name: Lint
        run: npm run lint --if-present

      - name: Test
        run: npm test -- --ci --reporters=default --reporters=jest-junit
```

### 캐시/매트릭스/아티팩트

{% raw %}
```yaml
      - name: Cache deps
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: npm-${{ hashFiles('package-lock.json') }}
          restore-keys: npm-

      - name: Build
        run: npm run build

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
```
{% endraw %}

매트릭스 테스트:

{% raw %}
```yaml
strategy:
  matrix:
    node: [18, 20, 22]
steps:
  - uses: actions/setup-node@v4
    with: { node-version: ${{ matrix.node }} }
```
{% endraw %}

### 배포(예: GitHub Pages)

`.github/workflows/deploy-pages.yml`
```yaml
name: Deploy Pages
on:
  push:
    branches: [ main ]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build site
        run: |
          npm ci
          npm run build:docs
      - name: Upload pages artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./docs/dist
  deploy:
    needs: build-deploy
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Pages
        uses: actions/deploy-pages@v4
```

## 보안/컴플라이언스 — Advanced Security, Dependabot, 시크릿 스캐닝

### Dependabot

- 의존성 버전/보안 업데이트 자동 PR
- 구성 예 `.github/dependabot.yml`:
```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule: { interval: "weekly" }
    open-pull-requests-limit: 5
```

### 코드 스캐닝(CodeQL), 시크릿 스캐닝

- 취약점/시크릿 유출 자동 탐지
- PR에 **보안 검사 결과**를 상태 체크로 반영

### 브랜치 보호/필수 리뷰/필수 체크

- 강제 푸시 금지, 리뷰 1+명 요구, 상태 체크 통과 후 머지만 허용

## 저장소 구조/대규모 코드베이스 — 모노레포 vs 폴리레포, 서브모듈/서브트리

### 모노레포

- 장점: 원자적 변경, 일관 빌드, 단일 이슈/PR
- 과제: CI 시간/권한/도구 스케일링

### 폴리레포

- 장점: 독립 릴리스/권한 분리
- 과제: 변경 전파/버전 호환 관리

### 서브모듈

```bash
git submodule add https://github.com/org/libfoo external/libfoo
git submodule update --init --recursive
```
- 외부 고정 스냅샷 트래킹, 동기화 관리 필요

### 서브트리(대안)

- 외부 저장소를 **히스토리 포함 병합**해 단순화(별도 명령 필요)

## 배포 전략 — 환경/승인/릴리스 관리

- **환경**: `dev` → `staging` → `prod`
- **승인 게이트**: Staging→Prod 수동 승인
- **릴리스**: 태그/릴리스 노트/체인지로그 자동화
- **세만틱 버저닝**: `MAJOR.MINOR.PATCH`

## 실전 시나리오 모음

### 개인 프로젝트(웹앱) — 세팅부터 배포까지

1) `repo` 생성 → `main` 보호
2) `feature/*` 브랜치로 기능 단위 작업
3) PR 템플릿/Issue 템플릿 적용
4) Actions CI: Lint/Test/Build
5) GitHub Pages로 문서 자동 배포
6) 릴리스 태깅 + 노트 생성
7) Dependabot 주간 점검

**예시 워크플로우**: `.github/workflows/ci-release.yml`
```yaml
name: CI & Release
on:
  pull_request:
  push:
    branches: [ main ]
jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '22' }
      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build
  release:
    if: startsWith(github.ref, 'refs/tags/')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Create Release
        uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true
```

### 팀 프로젝트 — CODEOWNERS/보호/필수체크/환경승인

- `/frontend/`는 FE 팀, `/backend/`는 BE 팀이 리뷰
- `main`에 Squash 머지 + 테스트/보안 필수
- `staging` 배포 자동, `prod`는 승인 필요

### 데이터/모델 프로젝트 — LFS/아티팩트/런북

- 모델 파일 LFS/패키지 저장
- 학습 결과를 Actions 아티팩트로 보관
- 재현 가능한 실행 스크립트와 **런북(runbook)** 문서화

## 충돌/리베이스/체리픽 — 필수 실습

### 충돌 해결

```bash
git pull --rebase origin main
# 충돌 파일 열기 → 마커(<<<<, >>>>) 기준 수동 해결

git add <file>
git rebase --continue
```

### 커밋 수정/정리

```bash
git commit --amend
git rebase -i HEAD~5    # 커밋 스쿼시/재배열
```

### 체리픽

```bash
git checkout hotfix/login
git cherry-pick <commit-sha>
```

**주의**: 공유 브랜치의 이력 변경은 팀 규칙에 따라 제한

## 문제 해결 FAQ

- **푸시가 거부됨(브랜치 보호)**: PR로 변경 제출, 리뷰/체크 통과 필요
- **CI 느림**: 캐시/병렬/매트릭스 최적화, 경량화, 조건부 실행
- **시크릿 노출 사고**: 즉시 폐기/교체, 히스토리 정리, 시크릿 스캐닝 강화
- **서브모듈 꼬임**: `git submodule sync --recursive` 후 재초기화

## 마무리: 핵심 요약

- GitHub는 **코드 협업-자동화-보안-배포**를 **하나의 플랫폼**에서 제공
- 팀 효율을 좌우하는 것은 **정책/템플릿/자동화**의 일관성
- 작은 PR, 강한 브랜치 보호, 재현 가능한 CI, 명료한 문서화가 **품질과 속도**를 동시에 높임
