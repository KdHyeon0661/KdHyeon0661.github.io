# GitHub 저장소 생성 및 관리

## 핵심 요약

GitHub에 새 저장소를 만들고 관리하는 가장 안전하고 효율적인 방법을 설명합니다. 초보자도 쉽게 따라할 수 있도록 기본 시나리오와 고급 상황을 모두 다룹니다.

---

## 저장소 생성 기본 절차

1. **GitHub 로그인** 후 우측 상단 `+` → **New repository** 선택
2. 필수 정보 입력:
   - **Repository name**: 저장소 이름 (예: `my-project`)
   - **Visibility**: Public(공개) 또는 Private(비공개)
   - **Initialize this repository with**: 초기 파일 생성 옵션

### 초기화 옵션의 중요성
- **초보자/개인 프로젝트**: 모든 체크박스 해제 (빈 저장소 권장)
- **협업/문서화 프로젝트**: README 체크 가능 (단, 작업 절차가 달라짐)
- **특정 환경**: .gitignore와 License를 상황에 맞게 선택

> 조직 저장소라면 초기부터 브랜치 보호 규칙과 2FA(2단계 인증)를 설정하는 것이 좋습니다.

---

## 로컬에서 시작하는 표준 워크플로우

### 1. 빈 저장소로 시작 (가장 단순한 방법)

```bash
# 로컬 프로젝트 폴더 생성 및 초기화
mkdir my-project
cd my-project
git init

# 기본 파일 생성
echo "# My Project" > README.md

# 초기 커밋
git add .
git commit -m "Initial commit"

# GitHub에서 빈 저장소 생성 후 원격 연결
git remote add origin https://github.com/username/my-project.git
git branch -M main
git push -u origin main
```

**장점**: 충돌 가능성이 거의 없고 절차가 직관적입니다.

### 2. GitHub에서 README를 미리 생성한 경우

```bash
# 방법 1: 클론부터 시작 (가장 안전)
git clone https://github.com/username/my-project.git
cd my-project
# 작업 후 add/commit/push

# 방법 2: 기존 로컬 프로젝트가 있는 경우
git remote add origin https://github.com/username/my-project.git
git pull --rebase origin main  # 원격 커밋 동기화
git push -u origin main
```

**핵심**: 원격에 이미 커밋이 존재하면 로컬에서 바로 `push`할 수 없습니다. 반드시 먼저 동기화해야 합니다.

## 협업을 위한 기본 설정

### 브랜치 작업 흐름
```bash
# 기능 브랜치 생성
git checkout -b feature/new-feature

# 작업 완료 후
git add .
git commit -m "feat: add new feature"
git push -u origin feature/new-feature
# GitHub에서 Pull Request 생성
```

### 필수 운영 파일
1. **.gitignore**: 불필요한 파일 추적 방지
   ```
   node_modules/
   .env
   *.log
   .DS_Store
   ```

2. **README.md**: 프로젝트 문서화
   ```markdown
   # 프로젝트 이름
   ## 개요
   ## 설치 방법
   ## 사용법
   ```

3. **.gitattributes**: 크로스 플랫폼 호환성 보장
   ```
   * text=auto
   *.sh text eol=lf
   *.bat text eol=crlf
   ```

## 자주 발생하는 문제와 해결책

| 문제 상황 | 원인 | 해결 방법 |
|-----------|------|-----------|
| `remote origin already exists` | 원격 저장소가 이미 등록됨 | `git remote remove origin` 후 재등록 |
| `non-fast-forward` 오류 | 원격에 로컬에 없는 커밋이 존재 | `git pull --rebase origin main` 후 푸시 |
| `Permission denied` | 인증 실패 | HTTPS → PAT(개인 액세스 토큰) 사용 또는 SSH 키 설정 |
| `.gitignore`가 작동 안 함 | 이미 추적 중인 파일 | `git rm --cached <파일명>`으로 추적 해제 |
| `repository not found` | URL 오타 또는 권한 없음 | URL 확인, 비공개 저장소면 초대 수락 필요 |

## 고급 작업 시나리오

### 기존 프로젝트 GitHub에 업로드
```bash
# 기존 Git 프로젝트 디렉터리에서
git remote add origin https://github.com/username/project.git
git branch -M main
git push -u origin --all  # 모든 브랜치 푸시
git push -u origin --tags # 태그도 함께 푸시
```

### Fork한 저장소 동기화
```bash
git remote add upstream https://github.com/original/project.git
git fetch upstream
git checkout main
git merge upstream/main
```

### 릴리스 태그 관리
```bash
git tag -a v1.0.0 -m "Version 1.0.0 release"
git push origin v1.0.0
```

## 효율적인 협업을 위한 가이드라인

### 1. 커밋 메시지 규칙
Conventional Commits 형식을 따르면 변경 사항을 명확히 추적할 수 있습니다.
```
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 수정
style: 코드 포맷팅
refactor: 리팩토링
test: 테스트 코드
chore: 빌드 과정 또는 보조 기능
```

### 2. 브랜치 전략
- `main`: 안정적인 배포 버전
- `develop`: 다음 배포를 위한 개발 브랜치
- `feature/*`: 개별 기능 개발
- `hotfix/*`: 긴급 버그 수정

### 3. Pull Request 규칙
- PR 전에 로컬에서 테스트 실행
- 관련된 이슈 번호 참조
- 리뷰어 지정 필수
- CI/CD 체크 통과 확인

## 작업 환경 최적화

### SSH 키 설정 (선택 사항)
```bash
ssh-keygen -t ed25519 -C "your-email@example.com"
# 생성된 공개키(~/.ssh/id_ed25519.pub)를 GitHub에 등록
git remote set-url origin git@github.com:username/repo.git
```

### Git 설정 최적화
```bash
# 사용자 정보 설정
git config --global user.name "Your Name"
git config --global user.email "your-email@example.com"

# 유용한 별칭 설정
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.st status
```

## 결론

GitHub 저장소 관리는 단순한 기술적 절차를 넘어 프로젝트 협업의 기초를 형성합니다. 초기 단계에서 올바른 습관을 들이는 것이 장기적인 생산성에 결정적인 영향을 미칩니다.

**핵심 원칙**:
1. **시작은 단순하게**: 빈 저장소로 시작하면 초보자도 쉽게 시작할 수 있습니다.
2. **문서화는 초기부터**: README와 .gitignore는 프로젝트 시작과 동시에 작성하세요.
3. **협업 규칙은 미리 정하기**: 브랜치 전략, 커밋 규칙, PR 프로세스를 초기에 정의하세요.
4. **문제는 예방하기**: 자주 발생하는 오류 상황과 해결법을 미리 알아두면 시간을 절약할 수 있습니다.

GitHub은 단순한 코드 호스팅 서비스를 넘어 프로젝트 관리, 협업, 지식 공유의 플랫폼입니다. 이러한 도구들을 효과적으로 활용하면 개인 프로젝트부터 대규모 오픈소스 프로젝트까지 성공적으로 관리할 수 있습니다.

## 빠른 참조: 필수 명령어 모음

```bash
# 저장소 생성부터 푸시까지
git init
git add .
git commit -m "메시지"
git remote add origin https://github.com/username/repo.git
git branch -M main
git push -u origin main

# 원격 저장소 동기화
git pull origin main
git pull --rebase origin main  # 이력 정리하며 동기화

# 브랜치 관리
git checkout -b feature/name  # 새 브랜치 생성 및 이동
git branch -d branch-name     # 브랜치 삭제

# 상태 확인
git status
git log --oneline --graph     # 커밋 이력 그래프로 보기
```

Git과 GitHub을 효과적으로 사용하는 것은 계속된 실습과 경험을 통해 자연스럽게 익힐 수 있습니다. 작은 프로젝트부터 시작해 점차 복잡한 워크플로우로 확장해 나가는 것을 권장합니다.
