# C# 값 형식과 참조 형식: 메모리 관리와 객체지향의 핵심

C# 개발자 기술 면접에서 가장 단골로 등장하면서도, 실무 성능 최적화에 가장 직접적인 영향을 미치는 주제가 바로 **값 형식(Value Type)**과 **참조 형식(Reference Type)**의 차이입니다.

단순히 "구조체는 스택에, 클래스는 힙에 저장된다"라는 단편적인 암기식 지식으로는 현대 C#의 복잡한 메모리 관리 메커니즘을 온전히 이해할 수 없습니다. 이 글에서는 두 형식의 본질적인 차이부터 시작하여, 개발자들이 흔히 앓고 있는 메모리 할당 위치에 대한 오해, 그리고 최신 C#에 도입된 고성능 구조체 문법까지 매우 깊이 있게 파헤쳐 보겠습니다.

## 값 형식과 참조 형식의 본질

C#의 모든 데이터 타입은 변수가 데이터를 품고 있는 방식에 따라 두 가지로 완벽하게 나뉩니다. 핵심은 **"변수라는 상자 안에 진짜 데이터가 들어있는가, 아니면 데이터가 있는 곳의 주소표가 들어있는가"**입니다.

### 값 형식 (Value Type)

값 형식은 변수 자체가 데이터의 실체를 직접 가지고 있습니다. 기본 자료형(`int`, `double`, `bool` 등)과 구조체(`struct`), 열거형(`enum`)이 여기에 속합니다.

값 형식의 가장 큰 특징은 다른 변수에 값을 대입할 때 **데이터가 통째로 복사**된다는 점입니다. 복사된 이후에는 두 변수가 완전히 독립적인 상태가 되므로, 한쪽을 수정해도 다른 쪽에 전혀 영향을 주지 않습니다.

```csharp
public struct Point
{
    public int X;
    public int Y;
    
    public Point(int x, int y)
    {
        X = x;
        Y = y;
    }
}

Point p1 = new Point(10, 20);
Point p2 = p1; // p1의 데이터가 p2로 완벽하게 복사됩니다.

p2.X = 99;
Console.WriteLine(p1.X); // 출력: 10 (원본 p1은 전혀 영향을 받지 않습니다)

```

### 참조 형식 (Reference Type)

참조 형식은 변수 안에 실제 데이터가 아닌, 데이터가 저장된 메모리(Heap)의 **주소(참조값)**를 가지고 있습니다. 클래스(`class`), 인터페이스(`interface`), 대리자(`delegate`), 배열, 그리고 `string`이 여기에 속합니다.

다른 변수에 값을 대입하면 데이터 자체가 복사되는 것이 아니라 **주소표만 복사**됩니다. 결국 두 변수가 메모리 상의 동일한 객체를 가리키게 되므로, 한쪽에서 내부 데이터를 수정하면 다른 쪽에서도 그 변경된 값을 보게 됩니다.

```csharp
public class PointClass
{
    public int X;
    public int Y;
    
    public PointClass(int x, int y)
    {
        X = x;
        Y = y;
    }
}

PointClass pc1 = new PointClass(10, 20);
PointClass pc2 = pc1; // pc1이 들고 있는 주소값만 pc2로 복사됩니다.

pc2.X = 99;
Console.WriteLine(pc1.X); // 출력: 99 (두 변수가 같은 객체를 공유하고 있습니다)

```

## 최대의 오해: 메모리 할당 위치 (Stack vs Heap)

C# 입문자들이 가장 많이 갇히는 오해가 바로 "값 형식은 무조건 스택(Stack)에 저장되고, 참조 형식은 무조건 힙(Heap)에 저장된다"는 명제입니다. 이는 반만 맞고 반은 틀린 설명입니다.

정확한 규칙은 다음과 같습니다.

* **참조 형식의 실제 데이터(객체)는 항상 힙(Heap)에 저장됩니다.**
* **값 형식의 데이터는 변수가 선언된 위치(문맥)에 따라 저장 위치가 달라집니다.**

만약 값 형식이 메서드 내부의 지역 변수나 매개변수로 선언되었다면, 메서드가 실행되는 동안에만 존재하는 스택에 저장됩니다. 하지만 **값 형식이 클래스의 필드(멤버 변수)로 선언되었다면, 그 값 형식 데이터는 클래스 객체가 생성될 때 힙 메모리에 함께 묻혀서 저장**됩니다.

```text
[메모리 할당 구조의 진실]

public class Wrapper { public Point P; } // Point는 struct

Stack 메모리                       Heap 메모리
[ 지역변수 p1 (X:10, Y:20) ]       
[ 지역변수 wrapper (주소)  ]  ───>  [ Wrapper 객체 본체 ]
                                    [ └─ 필드 P (X:0, Y:0) ] 

```

위 텍스트 다이어그램에서 보듯, `Wrapper` 클래스 내부의 `Point P`는 구조체(값 형식)임에도 불구하고 부모 객체를 따라 힙에 생성되었습니다. 따라서 "struct는 수명이 짧은 스택에만 존재한다"는 고정관념을 버려야 올바른 메모리 설계가 가능합니다.

## 박싱(Boxing)과 언박싱(Unboxing)의 성능 함정

값 형식을 참조 형식의 최고 조상인 `object` 타입이나 인터페이스 타입으로 변환할 때 발생하는 현상을 **박싱(Boxing)**이라고 합니다. 반대로 박싱된 객체에서 다시 값 형식 데이터를 추출하는 것을 **언박싱(Unboxing)**이라고 합니다.

```csharp
int number = 123;
object obj = number;   // Boxing 발생: 힙에 새로운 객체가 할당되고 값이 복사됨
int unboxed = (int)obj; // Unboxing 발생: 힙에서 값을 꺼내와 스택에 복사함

```

박싱이 무서운 이유는 개발자도 모르는 사이에 **힙 메모리 할당(Allocation)과 메모리 복사 연산이 동시에 발생**하기 때문입니다. 힙에 쓰레기가 쌓이면 결국 가비지 컬렉터(GC)가 작동해야 하므로, 반복문 내부에서 박싱이 일어나면 애플리케이션에 심각한 성능 랙(Lag)이 발생합니다.

이 문제를 해결한 것이 바로 **제네릭(Generics)**입니다. 구시대의 유물인 `ArrayList` 대신 제네릭 컬렉션인 `List<T>`를 사용해야 하는 가장 큰 이유가 바로 박싱 방지에 있습니다.

```csharp
// [성능 악화 패턴]
ArrayList list = new ArrayList();
list.Add(10); // 매개변수가 object 타입이므로 int가 전달되는 순간 Boxing 발생

// [성능 최적화 패턴]
List<int> list2 = new List<int>();
list2.Add(10); // 내부적으로 int[] 배열을 사용하므로 Boxing이 전혀 발생하지 않음

```

## 특수한 참조 형식: 문자열 (string)

`string`은 클래스(참조 형식)이지만, 마치 값 형식처럼 행동하는 매우 특이하고 중요한 녀석입니다. 그 이유는 `string` 객체가 한 번 메모리에 만들어지면 내부의 글자를 절대 바꿀 수 없는 **불변(Immutable)** 특성을 가지고 설계되었기 때문입니다.

```csharp
string a = "Hello";
string b = a;          // 주소가 복사되어 같은 "Hello"를 가리킴
b = "World";           // 원본을 수정하는 것이 아니라, 힙에 새로운 "World" 객체를 만들고 주소를 바꿈

Console.WriteLine(a);  // 출력: "Hello"
Console.WriteLine(a == b); // 출력: false

```

문자열을 수정(`b = "World"`)하려 할 때 원본을 건드리지 않고 새로운 객체를 생성해 버리기 때문에, 참조 형식임에도 불구하고 값 형식과 동일한 예측 가능성을 제공합니다. 또한 C# 컴파일러는 `==` 연산자를 사용할 때 메모리 주소가 아닌 문자열의 내용물 자체를 비교하도록 특별하게 오버로딩해 두었습니다.

## 최신 C#의 고성능 struct 기능 (C# 7.0 ~ 10.0)

마이크로소프트는 성능의 극한을 끌어올리기 위해 최근 C# 버전에 구조체를 위한 강력한 최적화 기능들을 연달아 추가했습니다.

### readonly struct (불변 구조체)

구조체를 불변으로 강제하고 싶을 때 사용합니다. 내부의 모든 필드는 `readonly`로 선언되어야 합니다. 이 키워드를 붙이면 컴파일러는 해당 구조체가 절대 변경되지 않음을 확신하게 되어, `in` 매개변수로 전달할 때 발생하던 불필요한 **방어적 복사(Defensive Copy)** 과정을 과감하게 생략합니다.

```csharp
public readonly struct Point
{
    public int X { get; }
    public int Y { get; }
    
    public Point(int x, int y)
    {
        X = x;
        Y = y;
    }
}

```

### ref struct (스택 전용 구조체)

구조체가 힙(Heap) 메모리로 탈출하는 것을 원천적으로 봉쇄하는 강력한 키워드입니다. `ref struct`로 선언된 구조체는 오직 스택에만 존재할 수 있으며, 클래스의 필드로 사용되거나 박싱되는 것이 컴파일 단계에서 차단됩니다.

대표적인 예가 바로 메모리 최적화의 꽃인 `Span<T>`입니다. 스택 메모리나 비관리 메모리 포인터를 안전하게 다루기 위해 만들어진 엄격한 제약 규칙입니다.

```csharp
public ref struct MyRefStruct
{
    public int Value;
}

```

### 구조체의 매개변수 없는 생성자 (C# 10+)

과거 C#에서는 구조체에 매개변수가 없는 기본 생성자(`public Point()`)를 선언할 수 없었습니다. 무조건 모든 메모리를 

$$0$$

으로 초기화하는 기본 동작을 강제했기 때문입니다. 하지만 C# 10부터는 이 제약이 풀려, 구조체 생성 시 특정 기본값을 부여할 수 있게 되었습니다.

```csharp
public struct Point
{
    public int X;
    public int Y;
    
    public Point() // C# 10부터 허용됨
    {
        X = -1;
        Y = -1;
    }
}

```

## 실무 가이드: 언제 class를 쓰고 언제 struct를 써야 할까?

클래스와 구조체 중 무엇을 선택할지는 데이터의 성격과 성능 요구사항에 따라 결정됩니다. 다음의 판단 기준을 참고하시기 바랍니다.

1. **데이터의 크기가 작은가?**
구조체는 메서드 매개변수로 넘길 때마다 데이터가 통째로 복사됩니다. 데이터의 크기가 
$$16$$


 바이트를 넘어간다면, 복사 비용이 참조 포인터를 넘기는 비용보다 비싸져 오히려 성능이 하락합니다. 크기가 큰 데이터는 무조건 `class`를 사용하세요.
2. **논리적으로 불변(Immutable)인가?**
점(Point), 색상(Color), 좌표(Vector)처럼 한 번 값이 정해지면 그 자체가 하나의 독립적인 '값'을 의미하는 경우 `struct`가 적합합니다. 가급적 `readonly struct`로 설계하세요.
3. **상속이나 다형성이 필요한가?**
구조체는 상속을 받을 수도, 해줄 수도 없습니다. 객체지향적인 상속 트리가 필요하다면 `class`를 사용해야 합니다.
4. **수명이 짧고 엄청나게 많이 생성되는가?**
게임 엔진의 파티클 입자처럼 짧은 시간 동안 수만 개가 생성되었다 사라지는 데이터라면, 힙 메모리를 어지럽혀 GC 랙을 유발하는 `class` 대신 스택에 즉시 생성되고 사라지는 `struct`를 사용하는 것이 압도적으로 유리합니다.

## 마무리

값 형식과 참조 형식의 차이는 단순히 "어디에 저장되느냐"를 넘어서, 변수 간의 복사 동작, 가비지 컬렉터(GC)의 부하, 그리고 박싱/언박싱이라는 성능 이슈까지 애플리케이션의 뼈대를 관통하는 핵심 개념입니다.

크기가 작고 불변성을 가지는 데이터 구조체(`readonly struct`)와, 복사를 막아주는 참조 전달(`in`, `ref`)을 적절히 조합한다면, C++에 버금가는 고성능의 메모리 최적화 코드를 C#에서도 얼마든지 작성할 수 있습니다. 메모리의 동작 원리를 머릿속에 그리며 코딩하는 습관을 들여보시기 바랍니다.
