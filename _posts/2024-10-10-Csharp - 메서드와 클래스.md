# C# 메서드와 클래스: 객체 지향 프로그래밍의 정수

C#은 철저한 객체 지향 프로그래밍(OOP) 언어입니다. 프로그램의 모든 요소는 객체(Object)라는 단위로 구성되며, 이 객체를 생성하기 위한 정밀한 설계도가 바로 클래스(Class)입니다. 클래스는 데이터의 상태를 저장하는 필드와 프로퍼티, 그리고 그 데이터를 조작하고 행동을 정의하는 메서드(Method)로 이루어집니다.

이 글에서는 단순한 문법의 나열을 넘어, 객체를 어떻게 설계하고 보호하며, 최신 C#의 고급 기능들을 활용하여 어떻게 견고한 애플리케이션 아키텍처를 구축하는지 상세히 알아보겠습니다.

## 클래스: 데이터와 행동을 담는 설계도

클래스는 특정 종류의 객체가 가져야 할 데이터(속성)와 행동(기능)을 하나의 단위로 묶어내는 템플릿입니다. 클래스 자체는 메모리에 실체가 없는 개념적인 정의이며, `new` 키워드를 통해 메모리에 할당될 때 비로소 실제 동작하는 인스턴스(객체)가 됩니다.

```csharp
public class Car
{
    // 필드(Field): 객체의 원초적인 데이터를 저장합니다.
    public string Model;
    public int Year;
}

// 클래스를 이용해 메모리에 실제 객체(인스턴스)를 생성합니다.
Car myCar = new Car(); 
myCar.Model = "Tesla Model 3";
myCar.Year = 2023;

```

### 필드 대신 프로퍼티를 사용해야 하는 이유

위의 예제처럼 클래스의 필드를 `public`으로 직접 노출하는 것은 객체 지향의 핵심 원칙인 캡슐화(Encapsulation)에 위배됩니다. 외부에서 데이터를 마음대로 조작할 수 있게 되면 데이터의 무결성이 쉽게 깨집니다.

실무에서는 데이터를 안전하게 보호하면서도 유연하게 접근할 수 있도록 **프로퍼티(Property)**를 사용합니다. 프로퍼티는 겉보기에는 변수 같지만 내부적으로는 값을 읽고 쓰는 메서드(getter/setter)로 동작하여 유효성 검사 로직을 추가할 수 있습니다.

```csharp
public class Car
{
    // 자동 구현 프로퍼티: 컴파일러가 숨겨진 필드를 자동으로 생성해 줍니다.
    public string Model { get; set; } = string.Empty;
    
    // Init-only 프로퍼티: 오직 객체가 생성되는 그 순간에만 값을 할당할 수 있는 불변 속성입니다.
    public int Year { get; init; }
    
    // 계산된 프로퍼티: 저장된 값이 아니라 호출될 때마다 새로운 결과값을 조합하여 반환합니다.
    public string Description => $"{Model} ({Year})";
}

var myCar = new Car { Model = "Hyundai Ioniq 5", Year = 2024 };
// myCar.Year = 2025; // 컴파일 오류: init으로 선언되었으므로 생성 이후에는 값을 변경할 수 없습니다.

```

## 메서드: 객체의 생명력을 불어넣는 행동

메서드는 객체가 수행할 수 있는 구체적인 작업이나 로직을 정의하는 공간입니다. 데이터를 입력받아 가공하고, 상태를 변화시키거나 새로운 연산 결과를 반환합니다.

### 메서드의 구조와 다양한 표현 방식

메서드는 접근 제한자, 반환 타입, 메서드 이름, 그리고 입력값을 받는 매개변수로 구성됩니다.

```csharp
public class Calculator
{
    // 일반적인 인스턴스 메서드 (객체를 생성해야 호출 가능)
    public int Add(int a, int b)
    {
        return a + b;
    }
    
    // 정적(Static) 메서드: 객체 생성 없이 클래스 이름만으로 즉시 호출 가능
    public static int Multiply(int a, int b)
    {
        return a * b;
    }
    
    // 식 본문 메서드(Expression-bodied method): 로직이 단 한 줄일 때 화살표(=>)를 사용하여 극도로 간결하게 표현합니다.
    public double Square(double x) => x * x;
}

```

### 매개변수 전달의 고급 기법들

C#은 매개변수를 메모리 상에서 어떻게 취급할지 결정하는 세밀한 한정자들을 제공합니다.

```csharp
public class ParameterExamples
{
    // 선택적 매개변수 (Optional Parameter): 호출 시 값을 생략하면 기본값이 자동으로 들어갑니다.
    public void Greet(string name = "Guest")
    {
        Console.WriteLine($"Hello, {name}!");
    }
    
    // params 배열: 호출하는 쪽에서 개수에 제한 없이 인수를 쉼표로 나열하여 넘길 수 있습니다.
    public int Sum(params int[] numbers)
    {
        int total = 0;
        foreach (int num in numbers) total += num;
        return total;
    }
    
    // out 한정자: 메서드 내부에서 값을 만들어 외부로 밀어내는 출력 전용 파이프 역할을 합니다.
    public bool TryParseToInt(string input, out int result)
    {
        return int.TryParse(input, out result);
    }
    
    // in 한정자: 거대한 구조체를 넘길 때 데이터 복사 비용을 없애기 위해 메모리 주소만 읽기 전용으로 넘깁니다.
    public double CalculateDistance(in Point p1, in Point p2)
    {
        // 수학 공식 $$distance = \sqrt{dx^2 + dy^2}$$ 을 적용합니다.
        double dx = p2.X - p1.X;
        double dy = p2.Y - p1.Y;
        return Math.Sqrt(dx * dx + dy * dy);
    }
}

```

### 로컬 함수 (Local Function)

메서드 내부에서만 사용되는 작고 반복적인 로직이 있다면, 이를 클래스 밖으로 빼지 않고 메서드 내부에 선언하여 사용할 수 있습니다. 이를 로컬 함수라고 하며, 부모 메서드의 변수에 자유롭게 접근할 수 있어 클로저(Closure) 구현에 매우 유리합니다.

```csharp
public double CalculateHypotenuse(double a, double b)
{
    // 로컬 함수 정의
    double Square(double x) => x * x;
    
    return Math.Sqrt(Square(a) + Square(b));
}

```

## 생성자: 객체의 탄생과 초기화

생성자(Constructor)는 클래스와 동일한 이름을 가지며, 메모리에 객체가 생성되는 정확히 그 찰나에 자동으로 호출되는 특수한 메서드입니다. 객체가 쓰레기 값을 가지지 않고, 비즈니스 로직에 맞는 정상적인 상태로 출발할 수 있도록 초기 설정을 담당합니다.

```csharp
public class Person
{
    public string Name { get; }
    public int Age { get; }
    
    // 기본 생성자
    public Person()
    {
        Name = "Unknown";
        Age = 0;
    }
    
    // 매개변수가 있는 생성자 (객체 생성 시 필수 값을 강제합니다)
    public Person(string name, int age)
    {
        if (string.IsNullOrWhiteSpace(name))
            throw new ArgumentException("이름은 비어있을 수 없습니다.");
            
        Name = name;
        Age = age;
    }
    
    // 생성자 체이닝(Chaining): this 키워드를 통해 자신의 다른 생성자를 먼저 실행시킵니다. 코드 중복을 막아줍니다.
    public Person(string name, int age, string email) : this(name, age)
    {
        // Name과 Age는 위의 생성자가 처리하고, 여기서는 추가적인 작업만 수행합니다.
        Console.WriteLine($"이메일 설정됨: {email}");
    }
}

```

정적 필드만을 초기화해야 할 때는 클래스가 애플리케이션 내에서 최초로 사용되기 직전에 단 한 번만 호출되는 정적 생성자(`static Constructor()`)를 사용할 수도 있습니다.

## 접근 제한자: 정보 은닉과 캡슐화

객체 지향 프로그래밍에서 가장 중요한 철학 중 하나는 "외부에서 몰라도 되는 내부 사정은 철저히 숨긴다"는 것입니다. 이를 위해 접근 제한자를 사용합니다. 실무에서는 변수나 메서드를 무작정 `public`으로 열어두지 않고, 가장 폐쇄적인 `private`에서 시작하여 꼭 필요한 것만 점진적으로 개방하는 것이 원칙입니다.

* **public**: 제약 없이 어디서든 접근할 수 있는 완전한 개방 상태입니다. API의 출입구 역할을 합니다.
* **private**: 자신이 속한 클래스의 중괄호 내부에서만 접근할 수 있는 최고 수준의 보안 상태입니다.
* **protected**: 자신과, 자신을 상속받은 자식 클래스에게만 접근을 허락합니다.
* **internal**: 현재 속한 프로젝트(어셈블리) 내부에서는 자유롭지만, 외부 라이브러리에서는 접근할 수 없습니다.

## 상속과 다형성: 유연하고 확장 가능한 설계

상속은 기존에 만들어둔 클래스의 기능을 그대로 물려받아 새로운 클래스를 파생시키는 기능입니다. 중복 코드를 극적으로 줄이고, 여러 객체를 하나의 부모 타입으로 묶어 관리할 수 있게 해줍니다.

```text
[객체 지향의 상속과 인터페이스 구조]

   << interface >>
    IDrawable (도형을 그리는 규칙 정의)
         │
         ▼
  abstract class Shape (추상 클래스: 형태의 기본 틀 정의)
         │
    ┌────┴────┐
    ▼         ▼
  Circle    Square (다형성을 통해 각각 다른 방식으로 도형을 그림)

```

```csharp
// 추상 클래스: 자신은 불완전하여 인스턴스가 될 수 없고, 자식들에게 뼈대만 제공합니다.
public abstract class Shape
{
    // 추상 메서드: 자식 클래스에서 반드시 자신만의 방식으로 내용을 채워넣도록 강제합니다.
    public abstract double CalculateArea(); 
}

// 인터페이스: 클래스가 반드시 구현해야 하는 '행동의 계약서'입니다. 다중 상속이 가능합니다.
public interface IDrawable
{
    void Draw();
}

// 상속과 인터페이스 동시 적용
public class Square : Shape, IDrawable 
{
    public double Side { get; set; }
    
    // 부모의 추상 메서드를 오버라이딩(재정의)하여 구현합니다.
    public override double CalculateArea() => Side * Side;
    
    public void Draw()
    {
        Console.WriteLine($"한 변의 길이가 {Side}인 사각형을 그립니다.");
    }
}

```

상속을 사용하면 `Shape` 타입의 배열 하나에 원(`Circle`), 사각형(`Square`) 등 다양한 객체를 섞어서 담을 수 있으며, `CalculateArea()`를 호출할 때 각 객체가 자신만의 방식으로 계산을 수행합니다. 이를 다형성(Polymorphism)이라고 부릅니다.

## 제네릭 (Generics): 성능과 타입 안전성의 두 마리 토끼

제네릭은 클래스나 메서드를 설계할 때 데이터 타입을 미리 확정 짓지 않고, 외부에서 사용할 때 타입을 결정하도록 미뤄두는 강력한 기법입니다. 구시대의 `object` 타입을 사용할 때 발생하던 타입 캐스팅 오류와 심각한 성능 저하(박싱/언박싱)를 완벽하게 해결합니다.

```csharp
// 타입 매개변수 T를 사용하여 어떤 타입이든 담을 수 있는 범용 상자 생성
public class Box<T>
{
    public T Content { get; set; }
    public Box(T content) => Content = content;
}

// 메서드에도 제네릭을 적용할 수 있습니다.
public class Utility
{
    // where 키워드를 통해 T에 들어올 수 있는 타입의 조건을 강제합니다.
    public static T Max<T>(T a, T b) where T : IComparable<T>
    {
        return a.CompareTo(b) > 0 ? a : b;
    }
}

// 사용할 때 타입을 확정 짓습니다.
var intBox = new Box<int>(42);
string larger = Utility.Max("apple", "banana"); 

```

## 확장 메서드: 기존 클래스에 마법 부여하기

확장 메서드는 마이크로소프트가 미리 만들어 둔 내장 클래스나, 수정할 수 없는 서드파티 라이브러리의 클래스에 내가 원하는 새로운 메서드를 끼워 넣을 수 있는 문법적 설탕(Syntactic Sugar)입니다.

```csharp
public static class StringExtensions
{
    // 첫 번째 매개변수 앞에 this 키워드를 붙이면 확장 메서드가 됩니다.
    public static string Repeat(this string str, int count)
    {
        if (count <= 0) return string.Empty;
        return string.Concat(Enumerable.Repeat(str, count));
    }
}

// 원본 string 클래스에는 Repeat이라는 메서드가 없지만, 마치 원래 있던 것처럼 자연스럽게 호출됩니다.
string text = "Hello";
Console.WriteLine(text.Repeat(3)); // 출력: HelloHelloHello

```

## 비동기 처리: async / await

파일을 읽거나 네트워크 통신을 하는 I/O 작업은 CPU 입장에서 엄청나게 오랜 시간이 걸립니다. 이때 메인 스레드가 멈춰서 기다리는 것을 막기 위해 C#은 `async`와 `await` 패턴을 제공합니다. 이 패턴을 사용하면 비동기 코드를 마치 위에서 아래로 흐르는 동기 코드처럼 매우 직관적으로 작성할 수 있습니다.

```csharp
public class DataService
{
    // 비동기 메서드는 반드시 Task 또는 Task<T>를 반환해야 합니다.
    public async Task<string> FetchDataAsync(string url)
    {
        using HttpClient client = new HttpClient();
        
        // await를 만나는 순간, 이 메서드의 실행을 일시 정지하고 스레드를 다른 작업에 양보합니다.
        // 다운로드가 완료되면 그 시점부터 코드가 다시 재개됩니다.
        HttpResponseMessage response = await client.GetAsync(url);
        response.EnsureSuccessStatusCode();
        
        return await response.Content.ReadAsStringAsync();
    }
}

```

## 지연 실행: yield return과 이터레이터

`yield return`은 컬렉션의 데이터를 한꺼번에 모두 계산하여 메모리에 올려두는 대신, 호출하는 쪽에서 데이터를 하나씩 요청할 때마다 실시간으로 계산하여 던져주는 지연 실행(Lazy Evaluation) 구조를 만듭니다. 수백만 개의 데이터를 다룰 때 메모리 복잡도 

$$O(N)$$

의 부담을 없애주는 마법 같은 키워드입니다.

```csharp
public static IEnumerable<int> GetFibonacci(int count)
{
    int a = 0, b = 1;
    for (int i = 0; i < count; i++)
    {
        // 값을 밖으로 던져준 뒤 함수의 상태가 일시 정지되며, 다음 호출 시 이 지점부터 재개됩니다.
        yield return a; 
        
        int temp = a;
        a = b;
        b = temp + b;
    }
}

```

## 자원 관리: IDisposable과 using 구문

C#의 가비지 컬렉터(GC)는 메모리는 훌륭하게 청소해주지만, 파일 핸들이나 데이터베이스 연결 같은 운영체제 자원(Unmanaged Resource)이 열려있는 것은 알지 못합니다. 개발자가 직접 자원을 닫아주기 위해 `IDisposable` 인터페이스를 구현해야 합니다.

```csharp
public class FileLogger : IDisposable
{
    private StreamWriter _writer;
    
    public FileLogger(string filePath)
    {
        _writer = new StreamWriter(filePath, append: true);
    }
    
    public void Log(string message) => _writer.WriteLine(message);
    
    // 자원을 명시적으로 해제하는 메서드
    public void Dispose()
    {
        _writer?.Dispose();
    }
}

// using 구문을 사용하면 중괄호를 벗어날 때 컴파일러가 알아서 Dispose()를 호출해 줍니다.
using (var logger = new FileLogger("log.txt"))
{
    logger.Log("안전하게 기록됩니다.");
}

// C# 8.0 이상에서는 중괄호 없이 선언만으로 블록 끝에서 자동 해제되도록 간소화되었습니다.
using var logger2 = new FileLogger("log2.txt");
logger2.Log("코드가 훨씬 깔끔해졌습니다.");

```

## 실전 응용: 모던 작업 관리 아키텍처

지금까지 배운 모든 객체 지향의 개념(클래스, 속성, 인터페이스, 제네릭, 비동기, 확장 메서드)을 하나로 융합하여 실무적인 작업 관리 시스템의 뼈대를 설계해 보겠습니다. 이 예제는 데이터(Entity), 저장소(Repository), 비즈니스 로직(Service)을 분리하는 현대적인 아키텍처 패턴을 보여줍니다.

```csharp
using System;
using System.Collections.Generic;
using System.Threading.Tasks;

// 1. 엔티티(Entity) 클래스: 데이터의 구조와 상태 변경 로직을 캡슐화합니다.
public class TaskItem
{
    public int Id { get; }
    public string Title { get; private set; }
    public DateTime CreatedAt { get; }
    public bool IsCompleted { get; private set; }
    
    public TaskItem(int id, string title)
    {
        Id = id;
        Title = title;
        CreatedAt = DateTime.UtcNow;
        IsCompleted = false;
    }
    
    // 외부에서 속성을 직접 수정하지 못하게 하고 메서드를 통해 상태를 변경합니다.
    public void MarkComplete() => IsCompleted = true;
    
    public override string ToString() => $"[{(IsCompleted ? "V" : " ")}] {Title}";
}

// 2. 인터페이스: 데이터 저장소의 규격을 정의하는 계약서입니다.
public interface ITaskRepository
{
    Task<IEnumerable<TaskItem>> GetAllAsync();
    Task AddAsync(TaskItem task);
    Task UpdateAsync(TaskItem task);
}

// 3. 리포지토리 구현: 인터페이스의 계약을 메모리 환경에 맞게 구현합니다.
public class InMemoryTaskRepository : ITaskRepository
{
    private readonly Dictionary<int, TaskItem> _tasks = new();
    private int _nextId = 1;
    
    public Task<IEnumerable<TaskItem>> GetAllAsync()
    {
        return Task.FromResult<IEnumerable<TaskItem>>(_tasks.Values);
    }
    
    public Task AddAsync(TaskItem task)
    {
        var taskWithId = new TaskItem(_nextId++, task.Title);
        if (task.IsCompleted) taskWithId.MarkComplete();
        
        _tasks[taskWithId.Id] = taskWithId;
        return Task.CompletedTask;
    }
    
    public Task UpdateAsync(TaskItem task)
    {
        _tasks[task.Id] = task;
        return Task.CompletedTask;
    }
}

// 4. 확장 메서드: 모델 클래스를 더럽히지 않고 편리한 유틸리티 기능을 추가합니다.
public static class TaskExtensions
{
    public static bool IsOverdue(this TaskItem task, int daysThreshold = 7)
    {
        return !task.IsCompleted && (DateTime.UtcNow - task.CreatedAt).TotalDays > daysThreshold;
    }
}

// 5. 프로그램 실행
class Program
{
    static async Task Main()
    {
        // 다형성을 활용하여 인터페이스 타입의 변수에 실제 구현체를 주입합니다.
        ITaskRepository repository = new InMemoryTaskRepository();
        
        await repository.AddAsync(new TaskItem(0, "C# 객체 지향 마스터하기"));
        await repository.AddAsync(new TaskItem(0, "디자인 패턴 학습"));
        
        var tasks = await repository.GetAllAsync();
        
        foreach (var task in tasks)
        {
            Console.WriteLine(task.ToString());
            if (task.IsOverdue(1)) 
            {
                Console.WriteLine("  기한이 지난 작업입니다!");
            }
        }
    }
}

```

## 마무리

C#에서 클래스는 단순히 변수와 함수를 모아두는 가방이 아닙니다. 현실 세계의 복잡한 비즈니스 로직을 컴퓨터가 이해할 수 있는 도메인 개념으로 모델링하고, 데이터의 무결성을 보호하며, 타인이 내 코드를 가져다 쓸 때 발생할 수 있는 오류를 컴파일 단계에서 차단하는 강력한 설계 도구입니다.

`private`을 통한 캡슐화, `interface`를 통한 역할의 추상화, 그리고 `async`와 `yield`를 통한 자원 최적화 기법을 코드에 자연스럽게 녹여내는 연습을 지속하신다면, 견고하고 유지보수가 용이한 최상급의 C# 애플리케이션 아키텍처를 구성하실 수 있을 것입니다.
