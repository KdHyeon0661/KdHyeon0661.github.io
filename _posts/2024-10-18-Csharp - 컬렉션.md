# C# 컬렉션 클래스 완벽 가이드: 데이터 구조의 선택과 최적화

애플리케이션을 개발하다 보면 단일 변수만으로는 처리할 수 없는 방대한 양의 데이터를 다루게 됩니다. C#은 이러한 데이터를 효율적으로 저장, 검색, 조작할 수 있도록 `System.Collections.Generic` 네임스페이스를 통해 매우 강력하고 다채로운 컬렉션(Collection) 클래스들을 제공합니다.

어떤 컬렉션을 선택하느냐는 단순히 데이터를 담는 바구니를 고르는 문제가 아닙니다. 이는 애플리케이션의 응답 속도, 메모리 사용량, 그리고 미래의 확장성까지 결정짓는 가장 중요한 아키텍처 설계 중 하나입니다. 이 글에서는 C#이 제공하는 핵심 컬렉션들의 내부 동작 원리를 파헤치고, 실무에서 마주하는 다양한 상황에 맞춰 최적의 자료구조를 선택하는 기준을 제시해 보겠습니다.

## List<T>: 가장 범용적인 동적 배열

`List<T>`는 C# 개발자들이 가장 사랑하고 빈번하게 사용하는 컬렉션입니다. 크기가 고정된 일반 배열(`T[]`)과 달리, 데이터가 추가됨에 따라 스스로 크기를 늘려가는 '동적 배열(Dynamic Array)'의 특성을 가집니다.

```csharp
var inventory = new List<Product>();
inventory.Add(new Product("Laptop", 1200));
inventory.Add(new Product("Mouse", 25));

// 배열과 동일하게 인덱서를 통한 직접 접근이 가능합니다.
var firstItem = inventory[0];

```

### 내부 동작과 성능의 비밀 (Capacity vs Count)

`List<T>`는 내부적으로 일반 배열을 래핑하고 있습니다. 데이터를 추가(`Add`)하다가 내부 배열의 꽉 차게 되면, 리스트는 남몰래 **기존 배열보다 두 배 더 큰 새로운 배열을 할당받고 기존 데이터를 모두 복사**하는 이사 작업을 수행합니다.

* **인덱스 접근**: 
$$O(1)$$


 (내부가 배열이므로 메모리 주소 계산으로 즉시 접근합니다.)
* **끝에 추가**: 평균 
$$O(1)$$


(하지만 배열 재할당이 일어나는 순간에는 모든 데이터를 복사해야 하므로 일시적으로
$$O(n)$$


의 비용이 발생합니다.)
* **중간 삽입/삭제**: 
$$O(n)$$


 (데이터를 밀어내거나 당겨야 하므로 성능이 저하됩니다.)

**실무 최적화 팁:** 미래에 들어올 데이터의 크기를 대략적으로 알고 있다면, 생성 시점에 초기 용량(`Capacity`)을 지정해 불필요한 배열 재할당(메모리 복사)을 막아야 합니다.

```csharp
// 초기 용량을 10,000으로 명시하여 내부적인 배열 확장이 발생하지 않도록 최적화합니다.
var largeList = new List<int>(capacity: 10000);
for (int i = 0; i < 10000; i++)
{
    largeList.Add(i);
}

// 작업 완료 후, 너무 넉넉하게 잡아둔 빈 공간(Capacity)을 잘라내어 메모리를 회수합니다.
largeList.TrimExcess();

```

## Dictionary<TKey, TValue>: 키-값 매핑의 표준

학번으로 학생을 찾거나, 상품 코드로 상품 정보를 조회할 때 가장 먼저 떠올려야 하는 컬렉션입니다. 해시 테이블(Hash Table) 알고리즘을 기반으로 동작하여, 데이터가 수백만 개가 쌓여 있어도 압도적인 검색 속도를 자랑합니다.

```csharp
var employeeDirectory = new Dictionary<int, Employee>();
employeeDirectory[101] = new Employee("Alice", "Engineering");
employeeDirectory[102] = new Employee("Bob", "Marketing");

// 예외를 방지하는 가장 안전하고 표준적인 조회 패턴입니다.
if (employeeDirectory.TryGetValue(103, out var employee))
{
    Console.WriteLine(employee.Name);
}

```

### 해시 충돌과 성능 특성

`Dictionary`의 검색 속도는 평균적으로 완벽한 

$$O(1)$$

입니다. 하지만 이 마법 같은 속도를 유지하려면 키(Key)로 사용되는 객체가 고유한 식별 번호(`GetHashCode()`)를 잘 만들어내야 하고, 두 객체가 같은지 비교(`Equals()`)할 수 있어야 합니다. 만약 서로 다른 두 키가 같은 해시 코드를 반환하는 '해시 충돌'이 잦아지면, 검색 속도는 최악의 경우 

$$O(n)$$

까지 추락할 수 있습니다.

```csharp
// 문자열 키를 다룰 때 대소문자를 무시하면서 성능을 끌어올리는 커스텀 비교자 사용법
var caseInsensitiveDict = new Dictionary<string, int>(StringComparer.OrdinalIgnoreCase);

// 최신 C#에서 제공하는 유용한 API
// 값이 없으면 에러 대신 기본값(0 등)을 안전하게 반환합니다.
var value = employeeDirectory.GetValueOrDefault(999, defaultValue);

```

## HashSet<T>: 중복을 불허하는 고유 집합

어떤 데이터가 '존재하는지' 여부만 빠르게 확인하거나, 리스트에서 중복을 완전히 제거하고 싶을 때 사용하는 수학적 '집합(Set)' 자료구조입니다. 내부적으로 `Dictionary`와 똑같은 해시 알고리즘을 사용하지만, 값(Value) 없이 키(Key)만 보관한다고 이해하면 쉽습니다.

```csharp
var uniqueTags = new HashSet<string>();
uniqueTags.Add("C#");
uniqueTags.Add(".NET");
uniqueTags.Add("C#"); // 이미 존재하므로 아무 일도 일어나지 않고 무시됩니다.

Console.WriteLine(uniqueTags.Count); // 출력: 2

```

`HashSet<T>`는 요소의 추가, 삭제, 검색 모두 평균 

$$O(1)$$

의 속도를 보장합니다. 무엇보다 교집합, 합집합, 차집합 등 수학적인 집합 연산을 내장 API로 강력하게 지원합니다.

```csharp
var developers = new HashSet<string> { "Alice", "Bob", "Charlie" };
var designers = new HashSet<string> { "Bob", "Diana", "Eve" };

// 합집합 (양쪽 모두 포함)
var allTeamMembers = new HashSet<string>(developers);
allTeamMembers.UnionWith(designers);

// 교집합 (공통으로 속한 멤버 찾기)
var bothRoles = new HashSet<string>(developers);
bothRoles.IntersectWith(designers); // 결과: {"Bob"}

```

## Queue<T>와 Stack<T>: 순서가 통제되는 컨테이너

데이터가 들어오고 나가는 입구를 엄격하게 통제하여 특정한 흐름을 만들어야 할 때 사용합니다.

### Queue<T> (선입선출, FIFO)

먼저 들어온 데이터가 먼저 나갑니다(First-In-First-Out). 은행 창구의 대기열이나 서버의 메시지 비동기 처리, 게임의 매치메이킹 대기열을 구현할 때 필수적입니다. 데이터의 양끝(앞/뒤)에 접근하므로 삽입과 삭제의 성능은 모두 

$$O(1)$$

입니다.

```csharp
var printQueue = new Queue<PrintJob>();
printQueue.Enqueue(new PrintJob("Report.pdf"));  // 줄 서기
printQueue.Enqueue(new PrintJob("Invoice.docx"));

while (printQueue.Count > 0)
{
    var job = printQueue.Dequeue(); // 가장 앞에 있는 작업 빼내기
    ProcessPrintJob(job);
}

```

### Stack<T> (후입선출, LIFO)

나중에 들어온 데이터가 가장 먼저 나갑니다(Last-In-First-Out). 웹 브라우저의 '뒤로 가기', 문서 편집기의 '실행 취소(Undo)', 괄호 짝 맞추기 알고리즘 등에 사용됩니다.

```csharp
var navigationHistory = new Stack<string>();
navigationHistory.Push("Home");            // 아래에 쌓기
navigationHistory.Push("Product Details"); // 그 위에 쌓기

// 가장 위에 있는(가장 최근의) 페이지 꺼내기
var currentPage = navigationHistory.Pop(); 

```

## 항상 정렬을 유지하는 컬렉션

데이터를 넣을 때마다 알아서 순서를 맞춰주는 지능적인 컬렉션입니다. 내부적으로 이진 탐색 트리(Red-Black Tree 등)를 사용하여 구현되므로 삽입/삭제/검색 모두 

$$O(\log n)$$

의 균일한 성능을 냅니다.

### SortedDictionary<TKey, TValue>와 SortedSet<T>

```csharp
var scoreboard = new SortedDictionary<int, string>();
scoreboard[1500] = "Alice";
scoreboard[1700] = "Bob";
// 넣은 순서와 상관없이 키(점수)를 기준으로 자동 정렬되어 보관됩니다.

var sortedNumbers = new SortedSet<int> { 5, 1, 9, 3, 7 };
// 범위 기반의 빠른 추출이 가능합니다.
var middleRange = sortedNumbers.GetViewBetween(3, 7); // {3, 5, 7}

```

## 실무의 딜레마: 컬렉션 타입과 인터페이스 반환

실무에서 메서드를 작성할 때, 만든 컬렉션을 도대체 어떤 타입으로 반환해야 할지 고민에 빠지게 됩니다. `List<T>`로 던져줄지, 아니면 추상화된 `IEnumerable<T>`나 `IQueryable<T>`로 던져줄지는 애플리케이션의 동작 방식을 완전히 뒤바꿉니다.

### List<T> vs IEnumerable<T>

* **List<T>**: 이미 메모리에 데이터가 100% 로드되어 있고, 결과를 확정 지은 상태(즉시 실행)입니다. 호출자가 인덱스로 데이터에 접근하거나 내용을 직접 수정해야 할 때 반환합니다.
* **IEnumerable<T>**: 데이터의 '흐름'만 반환합니다. 호출자가 `foreach`를 돌거나 `.ToList()`를 명시적으로 호출할 때까지 실제 연산이 미뤄지는 **지연 실행(Deferred Execution)**의 특성을 가집니다. 클래스 내부의 원본 리스트를 외부에서 함부로 조작하지 못하게 숨기고 싶을 때 매우 유용합니다.

```csharp
// 내부 리스트의 요소를 보호하면서 지연 실행 쿼리만 제공합니다.
public IEnumerable<Product> GetActiveProducts()
{
    return _productList.Where(p => p.IsActive); 
}

```

### IQueryable<T> (데이터베이스와의 대화)

Entity Framework 같은 ORM을 사용할 때 등장합니다. `IEnumerable<T>`가 메모리 위에서 동작하는 LINQ라면, `IQueryable<T>`는 작성된 쿼리를 C# 코드가 아닌 **SQL 문장으로 번역하기 위해 대기하는 상태**를 의미합니다.

```csharp
using (var context = new AppDbContext())
{
    // 아직 DB에 아무런 요청도 보내지 않았습니다. 쿼리문만 조립 중입니다.
    IQueryable<Customer> query = context.Customers
                                        .Where(c => c.City == "Seoul")
                                        .OrderBy(c => c.Name);
    
    // ToList()가 호출되는 이 순간, 조립된 내용이 실제 SQL로 변환되어 DB 서버를 때립니다.
    List<Customer> result = query.ToList(); 
}

```

## 컬렉션 선택을 위한 최종 가이드라인

위에서 살펴본 내용들을 토대로 어떤 상황에서 무엇을 골라야 할지 명확한 기준을 정리해 드립니다.

1. **가장 흔하게 쓴다면?**: `List<T>` (고민될 때는 십중팔구 이것이 정답입니다.)
2. **특정 고유 키로 데이터를 번개처럼 찾아야 한다면?**: `Dictionary<TKey, TValue>`
3. **중복된 값을 철저하게 걸러내야 한다면?**: `HashSet<T>`
4. **줄을 서서 차례대로 처리해야 한다면?**: `Queue<T>` (선입선출)
5. **되돌리기 기능이 필요하다면?**: `Stack<T>` (후입선출)
6. **항상 순서대로 정렬된 상태를 유지해야 한다면?**: `SortedSet<T>` 또는 `SortedDictionary<TKey, TValue>`
7. **멀티스레드에서 동시에 접근한다면?**: `ConcurrentDictionary`, `ConcurrentQueue` 등 동시성 전용 컬렉션
8. **데이터베이스에 쿼리를 날려야 한다면?**: `IQueryable<T>`

## 실전 시나리오: 웹 서버 요청 분석기

다양한 컬렉션을 혼합하여 성능을 극대화한 실전 아키텍처 예제입니다.

```csharp
public class RequestAnalyzer
{
    // 1. Queue: 최근 들어온 요청 1000개를 순서대로 기억하며, 오래된 것은 밀어냅니다.
    private readonly Queue<RequestLog> _recentRequests = new();
    
    // 2. Dictionary: API 엔드포인트별로 고유한 통계 객체를 1:1 맵핑하여 즉시 찾습니다.
    private readonly Dictionary<string, EndpointStats> _endpointStats = new();
    
    // 3. HashSet: 중복된 세션 ID를 걸러내어, 순수한 활성 접속자 수를 파악합니다.
    private readonly HashSet<string> _activeSessions = new();
    
    // 4. SortedSet: 처리 속도를 기준으로 가장 느린 악성 요청 상위 10개를 자동 정렬하여 유지합니다.
    private readonly SortedSet<RequestDuration> _slowestRequests = new();
    
    public void LogRequest(HttpRequest request, TimeSpan duration)
    {
        // 큐 관리 (1000개 제한 FIFO)
        _recentRequests.Enqueue(new RequestLog(request));
        if (_recentRequests.Count > 1000) _recentRequests.Dequeue();
        
        // 딕셔너리 조회 및 갱신 (O(1) 속도)
        if (!_endpointStats.TryGetValue(request.Path, out var stats))
        {
            stats = new EndpointStats();
            _endpointStats[request.Path] = stats;
        }
        stats.AddDuration(duration);
        
        // 해시셋을 통한 중복 세션 무시
        _activeSessions.Add(request.SessionId);
        
        // 정렬된 집합으로 상위 10개 지연 시간 유지 (O(log N) 속도)
        _slowestRequests.Add(new RequestDuration(request.Path, duration));
        if (_slowestRequests.Count > 10) _slowestRequests.Remove(_slowestRequests.Min);
    }
}

```

## 마무리

C#이 제공하는 컬렉션들은 각각 극명한 장단점과 최적의 사용처를 가지고 있습니다. 단순히 문법을 암기하는 수준을 넘어, 컬렉션이 데이터를 메모리(스택과 힙, 연속된 배열과 분산된 노드)에 어떻게 배치하고 찾는지 그 내부 원리를 이해해야 합니다.

데이터의 삽입이 빈번한지, 검색이 빈번한지, 중복을 허용할 것인지, 스레드 안전성이 필요한지를 명확히 분석하고 알맞은 컬렉션을 선택하는 습관을 들이십시오. 올바른 컬렉션의 선택은 수백 줄의 알고리즘 최적화보다 단번에 애플리케이션의 성능을 몇 배 이상 끌어올릴 수 있는 가장 훌륭한 무기입니다.
