# C# 데이터 모델링의 삼총사: 열거형, 구조체, 튜플 완벽 가이드

C#에서 데이터를 표현할 때 무조건 무거운 클래스(Class)를 생성하는 것은 좋은 설계가 아닙니다. 프로그램의 성능을 최적화하고 코드의 의도를 명확히 드러내기 위해, C#은 다양한 경량 데이터 타입들을 제공합니다.

상태나 종류를 명확한 이름으로 정의하는 **열거형(enum)**, 힙 메모리 할당 없이 스택에서 빠르게 동작하는 값 기반의 **구조체(struct)**, 그리고 메서드 간에 가볍게 여러 데이터를 묶어 전달하는 **튜플(tuple)**이 바로 그 주인공들입니다. 이 글에서는 각 타입의 내부 동작 원리와 최신 C# 문법, 그리고 실무에서 언제 어떤 타입을 선택해야 하는지에 대한 명확한 기준을 살펴보겠습니다.

## 열거형(enum): 의미 있는 상수의 집합

프로그래밍을 하다 보면 '1은 대기 중, 2는 진행 중, 3은 완료'와 같이 숫자에 특별한 의미를 부여하여 사용하는 경우가 많습니다. 이러한 '매직 넘버(Magic Number)'는 코드의 가독성을 심각하게 해치고 버그를 유발합니다. 열거형(`enum`)은 이러한 숫자들에 이름을 붙여 강한 타입 검사를 제공하는 우아한 해결책입니다.

### 기본 개념과 메모리 구조

기본적으로 열거형의 내부 데이터는 정수형(`int`)으로 처리되지만, 필요에 따라 메모리를 아끼기 위해 `byte`나 `short` 등으로 기저형(Underlying Type)을 변경할 수 있습니다.

```csharp
// 일반적인 열거형 선언
enum DayOfWeek
{
    Sunday, Monday, Tuesday, Wednesday, Thursday, Friday, Saturday
}

// 메모리 최적화를 위해 내부 타입을 ushort로 명시적 지정
enum StatusCode : ushort
{
    OK = 200,
    NotFound = 404,
    ServerError = 500
}

```

### 플래그 열거형과 비트 연산

실무에서 하나의 변수가 여러 개의 상태를 동시에 가져야 할 때가 있습니다. (예: '읽기' 권한과 '쓰기' 권한을 동시에 가짐). 이때 `[Flags]` 특성(Attribute)과 비트 연산을 활용하면 복합 상태를 매우 효율적으로 관리할 수 있습니다.

```text
[비트 마스킹을 활용한 Flags 열거형의 원리]

None    = 0000 (0)
Read    = 0001 (1)
Write   = 0010 (2)
Execute = 0100 (4)

Read | Write = 0011 (3) -> "읽기와 쓰기 권한이 모두 있음"

```

```csharp
[Flags]
enum Permissions
{
    None    = 0,
    Read    = 1 << 0,  // 1
    Write   = 1 << 1,  // 2
    Execute = 1 << 2,  // 4
    All     = Read | Write | Execute
}

var userPermissions = Permissions.Read | Permissions.Write;

// 특정 권한이 포함되어 있는지 확인 (비트 AND 연산 또는 HasFlag 메서드 사용)
if (userPermissions.HasFlag(Permissions.Read))
{
    Console.WriteLine("읽기 권한이 확인되었습니다.");
}

// 권한 추가 (비트 OR 연산)
userPermissions |= Permissions.Execute;

// 특정 권한만 제거 (비트 NOT 연산과 AND 연산의 조합)
userPermissions &= ~Permissions.Write;

```

### 열거형과 패턴 매칭

C#의 최신 `switch` 식(Expression)을 사용하면 열거형 값에 따른 분기 처리를 극도로 간결하게 작성할 수 있습니다.

```csharp
string GetDayType(DayOfWeek day) => day switch
{
    DayOfWeek.Saturday or DayOfWeek.Sunday => "주말",
    DayOfWeek.Monday => "월요일 블루스",
    _ => "평일"
};

```

## 구조체(struct): 값 의미론의 강력한 구현

구조체는 클래스와 비슷하게 데이터와 메서드를 묶을 수 있지만, 본질적으로 **값 타입(Value Type)**이라는 결정적인 차이가 있습니다.

클래스가 힙(Heap) 메모리에 할당되고 가비지 컬렉터(GC)의 관리를 받는 반면, 구조체는 기본적으로 스택(Stack) 메모리에 할당됩니다. 이는 구조체를 사용하면 메모리 할당 및 해제 비용이 거의 

$$0$$

에 수렴한다는 것을 의미합니다.

### 가변 구조체의 위험성과 readonly struct

구조체는 변수에 대입하거나 메서드의 매개변수로 전달할 때, 주소값이 아닌 **데이터 전체가 통째로 복사**됩니다. 만약 내부 상태를 수정할 수 있는(Mutable) 구조체를 만들면, 원본이 아닌 복사본의 데이터만 변경되는 심각한 논리적 오류에 빠지기 쉽습니다.

```csharp
struct MutablePoint
{
    public int X, Y;
    public void Move(int dx, int dy) { X += dx; Y += dy; }
}

var point = new MutablePoint { X = 5, Y = 10 };
var pointCopy = point;  // 데이터가 복사됨

pointCopy.Move(2, 3);   // 복사본의 좌표만 이동됨

Console.WriteLine($"원본: ({point.X}, {point.Y})");       // 출력: (5, 10) - 원본은 변하지 않음!
Console.WriteLine($"복사본: ({pointCopy.X}, {pointCopy.Y})"); // 출력: (7, 13)

```

이러한 예기치 않은 부작용을 원천 차단하기 위해, C#에서는 구조체를 설계할 때 가급적 불변(Immutable) 상태로 만드는 것을 표준 모범 사례로 삼습니다. `readonly struct`를 사용하면 컴파일러가 모든 필드의 수정을 엄격하게 금지합니다.

```csharp
public readonly struct Vector2D
{
    public double X { get; }
    public double Y { get; }

    public Vector2D(double x, double y) => (X, Y) = (x, y);

    // 연산자 오버로딩: 기존 값을 수정하지 않고, 새로운 구조체 객체를 만들어 반환합니다.
    public static Vector2D operator +(Vector2D a, Vector2D b)
        => new(a.X + b.X, a.Y + b.Y);
}

```

### C# 10의 혁신: record struct

데이터를 담는 불변 구조체를 만들 때 매번 생성자와 비교 연산자(`Equals`, `==`)를 타이핑하는 것은 매우 번거롭습니다. C# 10부터 도입된 `record struct`를 사용하면 이 모든 보일러플레이트 코드가 단 한 줄로 해결됩니다.

```csharp
public readonly record struct Position(int X, int Y);

var pos1 = new Position(3, 4);
// with 식을 사용하여 특정 필드만 변경된 새로운 불변 복사본을 쉽게 생성합니다.
var pos2 = pos1 with { X = 5 }; 

// record struct는 내부 데이터 값 자체를 비교하므로 직관적입니다.
Console.WriteLine(pos1 == new Position(3, 4)); // 출력: True

```

### 한계 돌파: ref struct

최고의 성능을 쥐어짜 내야 하는 시스템 프로그래밍 영역에서 사용되는 `ref struct`는 **절대 힙 영역으로 도망치지 못하도록 스택에만 존재할 것을 강제**받는 특수한 구조체입니다. 대표적인 예가 메모리 블록을 힙 할당 없이 안전하게 가리키는 `Span<T>`입니다.

`ref struct`는 박싱(Boxing)이 불가능하고, 일반 클래스의 필드가 될 수 없으며, 비동기(`async/await`) 메서드 내에서 사용할 수 없다는 강력한 제약이 따르지만, GC 랙(Lag)을 완전히 없애야 하는 시나리오에서는 필수적인 도구입니다.

## 튜플(tuple): 경량 다중 값 컨테이너

메서드에서 한 번에 두 개 이상의 값을 반환해야 할 때, 과거에는 `out` 매개변수를 사용하거나 별도의 클래스/구조체를 일일이 만들어야 했습니다. C# 7.0에서 도입된 **값 튜플(ValueTuple)**은 이러한 불편함을 완벽하게 해소했습니다.

*주의:* 과거의 `System.Tuple`은 참조 타입(클래스)이라 힙 메모리 할당이 발생했지만, 괄호 `()` 구문으로 사용하는 모던 C#의 `ValueTuple`은 내부적으로 구조체(값 타입)로 구현되어 있어 할당 비용이 전혀 없습니다.

### 명명된 튜플과 분해(Deconstruction)

튜플의 요소에 직관적인 이름을 부여하고, 반환된 결과를 다시 여러 변수로 분해하여 받을 수 있습니다.

```csharp
// 여러 값을 직관적인 이름과 함께 반환합니다.
(bool IsSuccess, int Result, string? ErrorMessage) TryParsePositive(string input)
{
    if (!int.TryParse(input, out var number))
        return (false, 0, "숫자가 아닙니다.");
    
    if (number <= 0)
        return (false, 0, "양수가 아닙니다.");
    
    return (true, number, null);
}

// 튜플 분해(Deconstruction)를 통해 반환값을 각각의 독립된 지역 변수로 받습니다.
var (success, result, error) = TryParsePositive("42");

if (success) Console.WriteLine($"결과: {result}");
else         Console.WriteLine($"오류: {error}");

```

### 튜플 사용 시 주의할 점

튜플은 내부 로직을 처리하거나 `private` 메서드에서 임시로 데이터를 묶어 반환할 때는 최고의 효율을 자랑합니다.

[지양] 하지만 외부로 노출되는 `public` API나 클래스 라이브러리의 반환 타입으로 크고 복잡한 튜플을 사용하는 것은 설계 결함입니다. 데이터의 의미와 계약이 모호해지기 때문입니다. 외부 인터페이스라면 명시적으로 `record`나 `struct`를 정의하여 사용하는 것이 올바른 객체 지향 설계입니다.

## 통합 활용 사례: 도메인 모델링

지금까지 배운 열거형, 구조체, 튜플을 모두 조합하여 쇼핑몰의 장바구니 도메인을 메모리 효율적으로 모델링해 보겠습니다.

```csharp
// 1. 열거형: 상품의 복합 속성을 비트 플래그로 표현
[Flags]
enum ProductFeatures
{
    None = 0,
    EcoFriendly = 1,
    Recyclable = 2,
    Biodegradable = 4
}

// 2. 구조체 (record struct): 도메인 식별자와 돈(Money)이라는 불변 값 모델링
public readonly record struct ProductId(int Value);

public readonly record struct Price(decimal Amount, string CurrencyCode)
{
    public static Price operator +(Price a, Price b)
    {
        if (a.CurrencyCode != b.CurrencyCode)
            throw new InvalidOperationException("통화 코드가 일치하지 않습니다.");
        
        return new Price(a.Amount + b.Amount, a.CurrencyCode);
    }
}

// 장바구니에 담긴 개별 항목
public readonly record struct CartItem(ProductId Id, int Quantity, Price UnitPrice)
{
    public Price TotalPrice => new(UnitPrice.Amount * Quantity, UnitPrice.CurrencyCode);
}

class ShoppingCart
{
    private readonly List<CartItem> _items = new();
    
    public void AddItem(CartItem item) => _items.Add(item);
    
    // 3. 튜플: 일회성 통계 데이터를 가볍게 묶어서 반환
    public (int TotalQuantity, Price TotalValue, int UniqueProductCount) GetSummary()
    {
        var totalQty = _items.Sum(item => item.Quantity);
        var totalValue = _items.Aggregate(new Price(0, "KRW"), (sum, item) => sum + item.TotalPrice);
        var uniqueCount = _items.Select(item => item.Id).Distinct().Count();
        
        return (totalQty, totalValue, uniqueCount);
    }
}

```

위 코드에서 `Price`, `ProductId`, `CartItem`은 모두 구조체이므로, 수천 개의 상품을 처리하더라도 힙 영역에 단 하나의 객체 쓰레기도 남기지 않습니다. 가비지 컬렉터(GC)가 개입할 일이 없으므로 애플리케이션은 극도의 성능 안정성을 유지할 수 있습니다.

## 결론: 적재적소의 데이터 타입 선택

C#의 데이터 타입 삼총사는 애플리케이션의 성능과 코드 품질을 결정짓는 핵심 설계 도구입니다.

1. **enum (열거형)**: 코드에 의미를 부여하고 상태나 플래그 옵션을 관리할 때 사용합니다. 정수형 기반이라 매우 가볍고 빠릅니다.
2. **struct (구조체)**: 좌표, 화폐, 시간처럼 크기가 작고(
$$16$$


 바이트 이하 권장) 불변의 의미를 지니는 값 기반 데이터를 모델링할 때 무거운 클래스를 대신하여 사용합니다. `readonly record struct`를 적극 활용하세요.
3. **tuple (튜플)**: 메서드에서 의미론적인 별도 타입(클래스/구조체)을 선언하기에는 너무 사소하고 일회성인 여러 데이터를 묶어 반환할 때 사용합니다.

이 세 가지 경량 타입들의 차이를 완벽히 이해하고 시스템 요소요소에 알맞게 배치한다면, 메모리 누수가 없고 응답 속도가 빠른 최상급의 C# 코드를 작성하실 수 있을 것입니다.
