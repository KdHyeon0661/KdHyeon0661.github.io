# C# 접근 제어자와 클래스 관계: 캡슐화와 설계의 기술

C#에서 클래스를 설계하고 접근 제어자(Access Modifier)를 지정하는 작업은 단순히 문법적인 제약을 거는 행위가 아닙니다. 이는 다른 개발자(혹은 미래의 나)에게 "이 데이터는 마음대로 써도 좋습니다" 혹은 "이 로직은 내부의 핵심 엔진이므로 절대 건드리지 마십시오"라고 전달하는 **아키텍처의 의사소통 방식**입니다.

이 글에서는 코드를 안전하게 보호하는 캡슐화의 원리부터, 최신 C#에 도입된 모던 접근 제어자(`file`, `init`), 그리고 객체 지향의 영원한 난제인 '상속(Inheritance) 대 컴포지션(Composition)'의 실무적인 선택 기준까지 깊이 있게 파헤쳐 보겠습니다.

## 접근 제어자: 가시성의 정밀한 통제

접근 제어자는 클래스의 멤버(필드, 속성, 메서드)가 외부로 노출되는 범위를 결정합니다. 모든 멤버를 `public`으로 열어두면 당장은 코딩하기 편하지만, 프로젝트가 커질수록 객체 간의 결합도가 치솟아 유지보수가 불가능한 스파게티 코드가 됩니다.

### 다양한 접근 제어자의 실전 적용

각 접근 제어자는 특정한 설계 의도를 담고 있습니다. 은행 계좌 클래스를 통해 가시성을 어떻게 미세 조정하는지 살펴보겠습니다.

```csharp
public class BankAccount
{
    // public: 컴포넌트의 공식 인터페이스입니다. 누구나 읽을 수 있습니다.
    public decimal Balance => _balance;
    
    // private: 클래스 내부의 핵심 상태입니다. 외부에서 절대 볼 수 없습니다.
    private decimal _balance;
    
    // internal: 같은 프로젝트(어셈블리) 내부의 모듈끼리만 협력할 때 사용합니다.
    internal string InternalId { get; }
    
    // protected: 이 클래스를 상속받은 자식(파생) 클래스에게만 접근을 허락합니다.
    protected DateTime CreatedAt { get; }
    
    public BankAccount(string internalId)
    {
        InternalId = internalId;
        CreatedAt = DateTime.UtcNow;
        _balance = 0;
    }
    
    // 공개 메서드: 외부에서 안전하게 상태를 변경할 수 있는 유일한 통로입니다.
    public void Deposit(decimal amount)
    {
        ValidateAmount(amount);
        _balance += amount;
        RecordTransaction("입금", amount);
    }
    
    // 비공개 메서드: 구현 세부 사항입니다. 외부에서는 이런 검증이 일어나는지 알 필요가 없습니다.
    private void ValidateAmount(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("금액은 0보다 커야 합니다.");
    }
    
    // 보호된 가상 메서드: 자식 클래스에서 이 동작을 입맛에 맞게 재정의(Override)할 수 있도록 열어둡니다.
    protected virtual void RecordTransaction(string type, decimal amount)
    {
        Console.WriteLine($"{DateTime.Now}: {type} {amount:C}");
    }
}

```

### C# 11의 file 접근 제어자

C# 11에서는 `internal`보다 더욱 강력하게 범위를 좁히는 `file` 접근 제어자가 탄생했습니다. 이 키워드가 붙은 타입은 **오직 자신이 선언된 소스 코드 파일(`.cs`) 내부에서만** 존재하며, 파일 밖을 벗어나면 마치 존재하지 않는 것처럼 취급됩니다.

이는 주로 소스 생성기(Source Generator)가 코드를 자동 생성할 때, 다른 파일의 클래스 이름과 충돌하는 것을 완벽하게 방지하기 위해 사용됩니다.

```csharp
// StringHelpers.cs 파일
file class Formatter
{
    public static string TrimAndLower(string input) => input.Trim().ToLower();
}

public class PublicService
{
    public void Process()
    {
        // 같은 파일 안이므로 Formatter 사용이 가능합니다.
        var result = Formatter.TrimAndLower("  Hello  ");
    }
}

```

### 혼동하기 쉬운: protected internal vs private protected

이 두 제어자는 이름이 길고 헷갈리기 쉬워 면접에서도 자주 묻는 주제입니다. 다음과 같은 차이가 있습니다.

```text
[가시성 범위 비교 요약]

Assembly A (클래스가 정의된 프로젝트)               Assembly B (외부 프로젝트)
[BaseClass]                                    [DerivedClass : BaseClass]
     │
     ├── protected internal    <─────────────>  접근 가능 (상속받았으므로 합격)
     │   (내부이거나 상속받았거나 둘 중 하나)
     │
     └── private protected     <──────X──────>  접근 불가 (상속은 받았지만 외부 어셈블리이므로 탈락)
         (내부이면서 동시에 상속받아야 함)

```

즉, `protected internal`은 합집합(OR)의 개념으로 접근 권한을 넓혀주고, `private protected`는 교집합(AND)의 개념으로 접근 권한을 철저히 좁히는 역할을 합니다.

## 캡슐화: 상태 보호와 불변성의 유지

캡슐화(Encapsulation)는 단순히 필드를 `private`으로 숨기고 `public get/set`을 열어두는 것이 아닙니다. 객체가 가질 수 없는 '불가능한 상태(예: 음수 나이, 절대영도 이하의 온도)'가 메모리에 존재하는 것 자체를 원천 차단하는 방어막입니다.

### 불변 조건을 보장하는 캡슐화 패턴

온도를 표현하는 클래스에서 섭씨와 화씨를 동시에 지원하면서 데이터의 무결성을 지키는 설계를 확인해 보겠습니다.

```csharp
public class Temperature
{
    // 내부는 절대온도(켈빈) 단 하나로만 상태를 관리하여 데이터 동기화 버그를 막습니다.
    private double _kelvin;
    
    public double Celsius
    {
        get => _kelvin - 273.15;
        set
        {
            // 과학적으로 불가능한 상태(절대영도 미만)가 입력되는 것을 방어합니다.
            if (value < -273.15)
                throw new ArgumentOutOfRangeException(nameof(value), "섭씨 -273.15도 이하는 불가능합니다.");
            _kelvin = value + 273.15;
        }
    }
    
    public double Fahrenheit
    {
        get => (_kelvin - 273.15) * 9/5 + 32;
        set => Celsius = (value - 32) * 5/9; // 화씨 세팅 시에도 섭씨의 검증 로직을 타도록 재사용합니다.
    }
    
    // 생성자를 비공개로 막아 무분별한 생성을 차단합니다.
    private Temperature() { }

    // 팩토리 메서드를 통해서만 명확한 의도를 가지고 객체를 생성하도록 유도합니다.
    public static Temperature FromCelsius(double celsius)
    {
        var temp = new Temperature();
        temp.Celsius = celsius; // 프로퍼티의 검증 로직 통과 확인
        return temp;
    }
}

```

### init 접근자와 record 타입 (불변 객체 설계)

상태가 한 번 정해지면 절대 변하지 않는 불변 객체(Immutable Object)는 멀티스레드 환경에서 동기화 버그를 원천 차단하는 훌륭한 설계입니다. C# 9.0에 도입된 `init`과 `record`는 이를 아주 쉽게 달성하게 해 줍니다.

```csharp
// 레코드(record)를 사용하면 단 한 줄만으로 완벽한 불변 객체가 탄생합니다.
// 내부적으로 init 전용 프로퍼티, 값 기반 비교(Equals), 디스트럭처링 메서드가 자동 생성됩니다.
public record Person(string FirstName, string LastName, DateTime BirthDate);

var person1 = new Person("John", "Doe", new DateTime(1990, 1, 1));

// with 표현식을 사용하면 원본 객체는 그대로 둔 채, 특정 값만 변경된 새로운 객체를 복제해 냅니다.
var person2 = person1 with { LastName = "Smith" }; 

// record는 참조(메모리 주소)가 달라도 안의 내용물이 같으면 같은 객체로 취급(true)합니다.
Console.WriteLine(person1 == new Person("John", "Doe", new DateTime(1990, 1, 1))); 

```

## 상속 vs 컴포지션: 재사용의 두 가지 길

코드의 중복을 피하고 재사용성을 높이기 위해 우리는 객체 간의 관계를 맺습니다. 이때 "A는 B이다(is-a)"의 상속(Inheritance)을 쓸 것인지, "A는 B를 가지고 있다(has-a)"의 컴포지션(Composition)을 쓸 것인지는 아키텍트의 가장 큰 고민거리입니다.

### 상속의 함정: 생성자 내 가상 메서드 호출

상속은 다형성을 제공하는 강력한 무기지만, 부모 클래스가 자식 클래스의 구현에 강하게 결합되는 취약한 기반 클래스 문제를 야기합니다. 특히 C#에서 상속을 다룰 때 가장 치명적인 버그를 유발하는 안티 패턴이 바로 **생성자 내부에서 가상(virtual) 메서드를 호출하는 것**입니다.

```text
[객체 생성 시 메모리 초기화 순서의 진실]
1. 자식 클래스의 필드들이 메모리에 할당됨 (숫자는 0, 참조형은 null 상태)
2. 부모 클래스의 생성자가 호출됨
   -> 이때 부모 생성자 안에서 가상 메서드를 호출하면?
   -> 자식 클래스에서 오버라이드한 메서드가 실행됨!
3. 자식 클래스의 생성자 본문이 비로소 실행됨

```

```csharp
public class BaseClass
{
    public BaseClass()
    {
        // 위험합니다! 자식 객체가 아직 완전히 생성되지도 않았는데 자식의 코드를 호출하게 됩니다.
        VirtualMethod(); 
    }
    
    protected virtual void VirtualMethod() => Console.WriteLine("부모의 기본 동작");
}

public class DerivedClass : BaseClass
{
    private readonly string _name;
    
    public DerivedClass(string name)
    {
        _name = name; // 3. 마지막으로 여기에 값이 들어옵니다.
    }
    
    protected override void VirtualMethod()
    {
        // 2. 부모 생성자에 의해 이곳이 강제로 먼저 실행됩니다.
        // 이때 _name은 아직 할당되지 않은 null 상태이므로 심각한 에러가 발생할 수 있습니다!
        Console.WriteLine($"자식의 동작: {_name.Length}"); 
    }
}

```

이러한 문제를 피하기 위해 생성자에서는 절대 가상 메서드를 호출하지 않는 것이 C#의 굳건한 모범 사례입니다. 상속의 구조가 닫혀있어야 한다면 클래스에 `sealed` 키워드를 붙여 더 이상의 상속을 원천 차단하는 것도 좋은 설계입니다.

### 컴포지션의 승리: 전략 패턴(Strategy Pattern) 활용

현대 소프트웨어 공학에서는 부작용이 많은 상속보다는 유연한 **컴포지션(Composition)**을 우선시합니다. 어떤 로직을 직접 물려받는 대신, 로직을 수행할 수 있는 인터페이스(부품)를 내부에 장착하는 방식입니다.

```csharp
// 정렬 알고리즘이라는 '전략'을 인터페이스로 추상화합니다.
public interface ISortStrategy<T>
{
    void Sort(List<T> items);
}

// 퀵 정렬, 병합 정렬 등 다양한 부품을 구현합니다.
public class QuickSortStrategy<T> : ISortStrategy<T> where T : IComparable<T> { /* 구현 생략 */ }
public class MergeSortStrategy<T> : ISortStrategy<T> where T : IComparable<T> { /* 구현 생략 */ }

// 컨텍스트 클래스: 상속을 받지 않고, 정렬 부품을 '조립(Composition)'하여 사용합니다.
public class DataSorter<T>
{
    private ISortStrategy<T> _strategy;
    
    // 의존성 주입(DI): 밖에서 원하는 부품을 끼워 넣습니다.
    public DataSorter(ISortStrategy<T> strategy)
    {
        _strategy = strategy ?? throw new ArgumentNullException(nameof(strategy));
    }
    
    // 실행 도중에도 언제든지 다른 정렬 부품으로 갈아 끼울 수 있는 엄청난 유연성을 가집니다.
    public void ChangeStrategy(ISortStrategy<T> newStrategy)
    {
        _strategy = newStrategy;
    }
    
    public void SortData(List<T> items)
    {
        // 실제 작업은 조립된 부품에게 위임(Delegation)합니다.
        _strategy.Sort(items);
    }
}

```

만약 상속으로 이 문제를 풀었다면 `DataSorter`를 상속받은 `QuickDataSorter`와 `MergeDataSorter`를 각각 만들어야 했을 것이며, 런타임에 정렬 방식을 교체하는 것은 불가능했을 것입니다. 컴포지션은 시스템의 결합도를 낮추고 단위 테스트를 극도로 쉽게 만들어줍니다.

## 어셈블리 경계와 내부 가시성 (InternalsVisibleTo)

규모가 큰 엔터프라이즈 솔루션에서는 모듈을 여러 개의 프로젝트(어셈블리)로 쪼갭니다. 이때 핵심 비즈니스 로직을 보호하기 위해 클래스를 `internal`로 선언하면, 정작 단위 테스트(Unit Test) 프로젝트에서 해당 클래스에 접근할 수 없어 테스트 코드를 작성할 수 없는 딜레마에 빠집니다.

이를 우아하게 해결하는 방법이 바로 `[InternalsVisibleTo]` 속성입니다.

```csharp
// 비즈니스 로직이 담긴 원본 프로젝트 (MySystem.Core.csproj)
// 프로젝트 설정 파일이나 AssemblyInfo.cs에 아래 속성을 부여합니다.
[assembly: System.Runtime.CompilerServices.InternalsVisibleTo("MySystem.Tests")]

namespace MySystem.Core
{
    // 외부의 다른 프로그램들은 사용할 수 없는 내부 전용 클래스
    internal class SecretTaxCalculator
    {
        internal decimal Calculate(decimal amount) => amount * 0.1m;
    }
}

```

이렇게 설정하면 캡슐화 원칙을 완벽하게 지키면서도, `MySystem.Tests`라는 이름을 가진 단위 테스트 프로젝트에서만큼은 마치 `public`인 것처럼 내부 클래스의 메서드를 호출하고 검증할 수 있게 됩니다.

## 결론

C#의 접근 제어자와 클래스 관계 설계는 단순히 코드가 돌아가게 만드는 것을 넘어, 시스템이 어떻게 변해야 하고 또 어떻게 변하지 말아야 하는지를 규정하는 아키텍트의 도면입니다.

무분별한 `public`을 지양하고 꼭 필요한 만큼만 인터페이스를 열어주는 **캡슐화**, 상태 변이로 인한 치명적 버그를 막아주는 **`record` 기반의 불변성**, 그리고 딱딱한 상속의 굴레를 벗어나 레고 블록처럼 유연하게 조립하는 **컴포지션(전략 패턴)**을 자유자재로 다루신다면, 어떤 비즈니스 요구사항 앞에서도 흔들리지 않는 견고한 소프트웨어를 만들어 내실 수 있을 것입니다.
